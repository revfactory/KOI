# 1회차 강의자료: AI 시대의 오픈소스: 왜 지금 다시 중요한가

## 메타데이터

- 일시: 2026-05-06 수 19:00-21:00
- 주제: AI 시대의 오픈소스와 6주 과정 오리엔테이션
- 핵심 메시지: 오픈소스는 코드 공개만이 아니라 AI 시대의 학습, 협업, 커리어, 제품 확산 방식이다.
- 목표 산출물: 개인 GitHub 프로필 README 초안, 후보 프로젝트 3개, 팀 구성, 팀 운영 약속 초안, AI 후보 분석 로그
- 트랙: 전원 공통으로 시작하되 후보 정리 시 외부 오픈소스 기여와 자체 공개 저장소 운영 가능성을 모두 남긴다.

## 학습 목표

수강생은 이 회차 후 다음을 수행할 수 있어야 한다.

1. AI 시대에 오픈소스가 학습, 협업, 커리어, 제품 확산 방식으로 중요해진 이유를 설명한다.
2. GitHub 저장소를 README, Issue, Pull Request, Release, LICENSE 관점에서 살펴본다.
3. 6주 동안 따라갈 후보 프로젝트 3개를 정리한다.
4. 팀 구성과 팀 운영 약속 초안을 만든다.
5. AI가 제안한 후보 분석 결과를 사람이 검증하는 방식으로 첫 AI 사용 로그를 남긴다.

## 수업 전 준비

수강생은 GitHub 계정을 준비하고, AI 도구 하나에 로그인해 둔다. 최근 사용한 오픈소스 패키지 이름 하나를 메모해 온다. 관심 분야도 하나 이상 정한다. 예시는 웹, AI, 데이터, 개발자 도구, 교육, 캠퍼스 문제 해결이다.

가능하면 Git이 설치되어 있는지 확인한다.

```bash
git --version
```

Git 사용 경험이 없어도 수업에는 참여할 수 있다. 이 회차에서는 저장소를 읽고 후보를 고르는 것이 우선이다.

## 수업 흐름

1. 최근 사용한 패키지를 떠올리며 오픈소스를 이미 쓰고 있다는 사실을 확인한다.
2. 6주 과정의 최종 약속과 평가 기준을 공유한다.
3. AI 시대 오픈소스 생태계의 변화를 간단히 다룬다.
4. 관심 저장소를 5개 관점으로 분석한다.
5. AI로 후보 저장소를 요약하게 하고 사람이 검증한다.
6. 팀을 구성하고 팀 운영 약속 초안을 만든다.
7. 개인 회고와 다음 회차 준비 항목을 정리한다.

## 핵심 내용

이 과정의 중심 약속은 6주 후 모든 팀이 공개 저장소 하나를 운영하거나 외부 오픈소스 프로젝트에 실제 기여 흔적을 남기는 것이다. 발표만 잘하는 과정이 아니라 GitHub에 남는 공개 증거를 만드는 과정이다.

오픈소스 프로젝트를 볼 때 별 수만 보지 않는다. README가 친절한지, Issue가 관리되는지, Pull Request가 리뷰되는지, Release가 이어지는지, LICENSE가 명확한지 함께 본다. 이 다섯 가지가 이후 기여 가능성을 판단하는 출발점이다.

AI 도구는 후보 분석을 빠르게 도울 수 있다. 그러나 AI가 추천한 프로젝트나 기여 아이디어를 그대로 확정하지 않는다. 최근 활동, 라이선스, 기여 가이드, 이슈 난이도는 사람이 직접 확인해야 한다.

## 활동

### 오픈소스 사용 자각

각자 최근에 설치하거나 사용한 패키지 이름을 적는다. 예시는 `react`, `requests`, `pandas`, `pytorch`, `axios` 같은 라이브러리다. 그 패키지를 누가 만들었는지, 왜 만들었는지, 어떤 라이선스로 공개했는지 생각해 본다.

이 활동의 목적은 정답을 맞히는 것이 아니다. 이미 오픈소스를 쓰고 있었지만 그 출처와 규칙을 잘 모르고 있었다는 점을 확인하는 것이다.

### 저장소 5개 관점 분석

관심 분야의 GitHub 저장소를 고르고 다음 질문에 답한다.

- README만 보고 프로젝트 목적과 사용법을 이해할 수 있는가?
- 열린 Issue가 관리되고 있는가?
- 최근 Pull Request가 리뷰되고 병합되는가?
- Release 또는 버전 관리 흔적이 있는가?
- LICENSE가 명시되어 있는가?
- 6주 안에 작은 기여를 시도할 수 있는가?

팀은 후보 3개를 고른다. 각 후보에는 저장소 URL, 선택 이유, 예상 기여 방식, 주요 리스크, 다음 조사 항목을 적는다.

### AI 후보 분석

후보 저장소 하나를 골라 AI에게 요약을 요청한다.

```text
이 GitHub 저장소를 대학생 오픈소스 기여 수업 관점에서 분석해줘.
출력은 한 줄 요약, 첫 기여 후보 3개, 6주 안에 가능한 이유, 사람이 직접 확인해야 할 리스크로 나눠줘.
확실하지 않은 내용은 추정이라고 표시해줘.
```

AI 결과를 받은 뒤 사람이 다음을 확인한다.

- 실제 최근 Commit 날짜
- LICENSE 존재와 종류
- Issue와 Pull Request의 최근 관리 상태
- AI가 존재하지 않는 기능, 파일, 정책을 말하지 않았는지 여부

### 팀 운영 약속 초안

팀은 다음 내용을 한 페이지로 정리한다.

- 팀명
- 한 줄 비전
- 잠정 역할: Maintainer, Contributor, Docs, AI-Automation
- 정기 회의 방식
- 의사결정 방식
- 연락 두절이나 역할 미수행이 생겼을 때의 처리 방식
- 후보 프로젝트 3개

이 문서는 2회차 이후 팀 저장소의 `docs/team-constitution.md`로 옮길 수 있다.

## 멘토링

멘토링에서는 후보 3개와 팀 운영 약속을 점검한다. 팀이 지나치게 큰 프로젝트를 고른 경우 작은 문서 기여, 재현, 테스트 추가, 예제 보강 같은 방향으로 범위를 줄인다.

확인할 질문은 다음과 같다.

- 후보 3개가 실제 URL과 선택 이유를 갖추었는가?
- 최소 1개 후보는 6주 안에 작은 기여가 가능한가?
- 팀 운영 약속이 잠정적으로라도 정리되었는가?
- 팀 안에 GitHub 경험이 부족하다면 누가 어떤 방식으로 보완할 것인가?
- AI 후보 분석에서 사람이 검증한 항목이 남아 있는가?

## AI 활용

AI는 README 요약, 저장소 구조 설명, 첫 기여 후보 추천, 후보 비교 초안 작성에 사용할 수 있다. 사람이 확인해야 할 항목은 최근 활동, 라이선스, 기여 가이드, 이슈 난이도, 팀 역량과 일정이다.

AI 사용 로그는 다음 항목을 문장으로 남긴다.

- 어떤 후보를 분석했는가
- AI에게 무엇을 요청했는가
- AI가 무엇을 제안했는가
- 사람이 무엇을 확인했는가
- 최종 후보 선정에 어떻게 반영했는가

## 과제

팀 과제는 후보 프로젝트 3개 정리와 팀 운영 약속 초안 작성이다. 개인 과제는 GitHub 프로필 README 초안 한 단락과 AI 후보 분석 로그 한 건이다.

제출 증거는 후보 저장소 URL, 후보 비교 문서, 팀 운영 약속 문서, 개인 프로필 초안, AI 사용 로그다.
