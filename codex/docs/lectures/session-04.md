## 핵심 내용

AI는 저장소 구조를 빠르게 요약하고, 관련 파일 후보를 제안하고, Pull Request 본문 초안을 만드는 데 도움이 된다. 그러나 AI 결과는 초안이다. 실제 실행, 테스트, diff 확인, 보안, 개인정보, 라이선스 검토는 사람이 책임진다.

AI 사용은 다음 5단계로 기록한다.

1. 이슈 분석: AI는 문제 요약과 원인 후보를 제안할 수 있다. 사람은 이슈 원문과 프로젝트 규칙을 확인한다.
2. 재현: AI는 재현 절차 초안을 만들 수 있다. 사람은 실제로 실행하고 실패 로그를 확인한다.
3. 수정 범위 산정: AI는 변경 파일 후보를 제안할 수 있다. 사람은 범위가 너무 크지 않은지 판단한다.
4. Pull Request 본문 작성: AI는 본문 초안을 만들 수 있다. 사람은 실제 diff와 일치하는지 확인한다.
5. 리뷰 대응: AI는 답변 초안을 만들 수 있다. 사람은 메인테이너 의도와 실제 수정 여부를 확인한다.

RAG는 외부 검색이나 문서를 LLM 답변에 결합하는 접근이다. README와 Issue를 AI에게 함께 제공하는 활동은 가장 단순한 형태의 RAG로 볼 수 있다.

MCP는 LLM 애플리케이션이 외부 데이터와 도구에 표준 방식으로 연결되도록 하는 프로토콜이다. GitHub, 파일, 데이터베이스, API 같은 도구 연결을 이해하는 배경이 된다.

A2A는 서로 다른 AI 에이전트가 통신하고 협업하도록 하는 프로토콜이다. 5회차에서 Planner, Coder, Reviewer, Tester, Documenter 역할을 나누는 흐름과 연결된다.

## 활동

### 낯선 저장소 분석 비교

먼저 AI 없이 README, Issues, Pull Requests, LICENSE를 보고 저장소 구조와 기여 가능성을 판단한다. 이후 같은 저장소 정보를 AI에게 제공하고 요약을 받는다.

비교할 내용은 다음과 같다.

- AI가 빠르게 찾아준 정보는 무엇인가
- 사람이 더 정확하게 확인한 정보는 무엇인가
- AI가 추정하거나 틀릴 수 있는 부분은 무엇인가
- 기여 후보로 삼아도 되는 작업은 무엇인가

### 외부 오픈소스 기여 Pull Request 초안

외부 기여 팀은 작은 Issue를 하나 고른다. `good first issue`, `docs`, `help wanted` 같은 라벨이 붙은 작업이 좋다.

진행 순서는 다음과 같다.

1. Issue 원문을 읽는다.
2. CONTRIBUTING과 LICENSE를 확인한다.
3. AI에게 이슈 요약과 수정 후보를 요청한다.
4. 사람이 가장 작은 Pull Request 범위를 고른다.
5. Fork, Branch, Commit, Pull Request 초안을 만든다.
6. Pull Request 본문에 AI 사용 로그를 작성한다.

### 자체 프로젝트 기능 또는 문서 Pull Request

자체 프로젝트 팀은 팀 저장소의 Issue 하나를 고른다.

진행 순서는 다음과 같다.

1. Issue의 완료 조건을 확인한다.
2. AI로 파일 구조와 영향 범위를 분석한다.
3. 변경 범위를 작은 기능 하나 또는 문서 개선 하나로 줄인다.
4. 테스트 또는 문서 체크리스트를 작성한다.
5. Pull Request 초안을 만든다.
6. AI 사용 로그와 사람 검증 기록을 작성한다.

### README 또는 changelog 보강

Pull Request 변경 내용을 바탕으로 README 또는 changelog를 보강한다.

AI에게 요청할 수 있는 예시는 다음과 같다.

```text
다음 Pull Request 변경 내용과 현재 README를 보고 README 보강안을 작성해줘.
실제 변경에 없는 기능은 쓰지 말고, 설치, 사용, 검증 방법 중 빠진 항목만 제안해줘.
마지막에 사람이 검증해야 할 항목을 붙여줘.
```

AI가 만든 문서는 실제 변경과 맞는 부분만 반영한다.

## 멘토링

멘토링에서는 Pull Request 초안, AI 사용 로그, 검증 체크리스트를 확인한다.

확인할 질문은 다음과 같다.

- Pull Request 초안이 실제 링크로 존재하는가?
- 변경 범위가 작고 리뷰 가능한가?
- AI 사용 로그 5단계가 비어 있지 않은가?
- 사람이 검증한 항목이 구체적인가?
- 테스트 또는 문서 증거가 있는가?
- RAG, MCP, A2A 한 줄 정의가 남아 있는가?
- 5회차에 멀티 에이전트로 처리할 Issue가 정해졌는가?

## AI 활용

AI는 코드 설명, 원인 후보, 테스트 초안, Pull Request 본문, README 또는 changelog 초안을 만드는 데 사용할 수 있다. 사람이 확인해야 할 것은 실행 결과, 테스트 통과 여부, 실제 diff, 기존 스타일, 보안, 개인정보, 라이선스다.

AI 사용 로그에는 다음을 문장으로 남긴다.

- 이슈 분석에서 AI가 제안한 것
- 재현 과정에서 사람이 실행한 것
- 수정 범위를 줄인 기준
- Pull Request 본문에서 사람이 고친 부분
- 리뷰 대응이나 문서 보강에서 검증한 내용
