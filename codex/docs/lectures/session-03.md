## 학습 목표

수강생은 이 회차 후 다음을 수행할 수 있어야 한다.

1. 주요 오픈소스 라이선스의 차이를 사용 시나리오 중심으로 설명한다.
2. LICENSE, README, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, Issue Template, Pull Request Template이 저장소 신뢰에 기여하는 방식을 설명한다.
3. AI 프로젝트에서 코드, 데이터, 모델, weights, 문서의 공개 범위가 다를 수 있음을 설명한다.
4. 팀 저장소에 운영 문서를 Pull Request로 추가한다.
5. AI가 제안한 라이선스 리스크를 공식 출처와 팀 운영 정책으로 검증한다.

## 수업 전 준비

수강생은 2회차 팀 저장소와 첫 Pull Request, Issue 목록, 트랙 선언문을 준비한다. 팀 저장소에 LICENSE가 있는지 확인하고, 없다면 어떤 라이선스를 선택할지 생각해 온다.

외부 기여를 시도하는 팀은 대상 저장소의 LICENSE와 CONTRIBUTING을 읽어 온다. 자체 프로젝트를 운영하는 팀은 프로젝트가 라이브러리, CLI, 웹 서비스, 문서, 데이터 중 어떤 형태에 가까운지 정리한다.

## 수업 흐름

1. 지난 회차의 저장소가 외부인이 신뢰할 수 있는 상태인지 확인한다.
2. "오픈"이라는 표현과 오픈소스 라이선스의 차이를 다룬다.
3. 주요 라이선스의 차이를 사용 시나리오로 정리한다.
4. 팀 저장소에 운영 신뢰 구조 문서를 추가한다.
5. AI 시대의 라이선스 쟁점을 다룬다.
6. AI에게 팀 프로젝트의 라이선스 리스크를 묻고 사람이 검증한다.
7. 다음 회차에서 AI를 활용해 Pull Request를 만들 준비를 한다.

## 핵심 내용

코드가 공개되어 있다고 해서 곧바로 오픈소스가 되는 것은 아니다. 사용, 수정, 배포, 기여를 허용하는 조건이 명확해야 한다. 그 조건의 첫 페이지가 LICENSE다.

하지만 LICENSE만으로는 저장소가 운영되지 않는다. 누가 기여할 수 있는지, 어떤 방식으로 Issue와 Pull Request를 만들지, 보안 문제는 어디로 신고할지, 커뮤니티에서 어떤 행동을 기대하는지 함께 정리되어야 한다.

이 회차의 핵심은 라이선스 법률 지식을 외우는 것이 아니라, 저장소를 외부 기여자가 들어올 수 있는 신뢰 구조로 만드는 것이다.

주요 라이선스는 다음 관점으로 이해한다.

- MIT는 짧고 관대한 허용형 라이선스로, 누구나 쉽게 쓰고 fork하기를 원하는 작은 프로젝트에 자주 쓰인다.
- Apache-2.0은 특허 조항을 포함한 허용형 라이선스로, 기업이나 외부 사용 가능성을 조금 더 의식할 때 고려한다.
- GPL 계열은 파생물도 같은 라이선스로 공개하도록 요구하는 카피레프트 성격이 강하다.
- AGPL은 네트워크 서비스 형태까지 고려하는 강한 카피레프트 라이선스다.
- LGPL과 MPL은 라이브러리 사용이나 파일 단위 공개 조건처럼 중간적 성격을 가진다.

AI 프로젝트에서는 공개 범위가 더 복잡해진다. 코드가 공개되어도 데이터셋, 학습 코드, 모델 weights, 평가 코드, 문서는 각각 다른 조건일 수 있다. 따라서 "오픈"이라는 마케팅 표현과 OSI 기준의 오픈소스 여부를 구분해야 한다.

## 활동

### 오픈과 오픈소스 구분

여러 프로젝트나 모델 이름을 보고 "오픈소스인지", "사용은 가능하지만 오픈소스라고 단정하기 어려운지", "판단 보류인지"를 구분한다. 이 활동의 목적은 정답 암기가 아니라 이름과 라이선스를 분리해서 보는 습관을 만드는 것이다.

### 운영 신뢰 구조 만들기

팀 저장소에 다음 문서를 추가하거나 보강한다.

- LICENSE
- CONTRIBUTING.md
- CODE_OF_CONDUCT.md
- SECURITY.md
- Issue Template
- Pull Request Template

CONTRIBUTING에는 최소한 다음 내용을 포함한다.

- 개발 환경 또는 문서 수정 방법
- Issue 작성 기준
- Branch와 Commit 규칙
- Pull Request 작성 기준
- Review와 Merge 기준
- 라이선스 정책
- AI 사용 로그 제출 방식

Pull Request Template에는 다음 내용을 포함한다.

- 무엇을 바꿨는가
- 왜 바꿨는가
- 어떻게 확인했는가
- AI를 사용했다면 무엇을 맡겼는가
- 사람이 검증한 것은 무엇인가
- 외부 코드나 의존성 라이선스를 확인했는가

외부 기여를 시도하는 팀은 외부 저장소의 문서를 복사하지 않는다. 해당 저장소의 규칙을 읽고, 팀 분석 저장소에 기여 준비 메모를 작성한다.

### AI 라이선스 리스크 검토

팀이 선택한 라이선스와 프로젝트 설명을 AI에게 제공하고 리스크를 묻는다.

```text
우리는 [라이선스 이름]을 선택했고, 프로젝트는 [한 줄 설명]이다.
6개월 뒤 발생할 수 있는 라이선스, 의존성, AI 생성 코드 리스크 3개를 제안해줘.
각 리스크마다 사람이 확인해야 할 공식 출처, 대응 정책, CONTRIBUTING에 넣을 문장을 제안해줘.
법률 조언처럼 단정하지 말고 교육용 점검 항목으로 작성해줘.
```

AI 답변을 받은 뒤 다음을 확인한다.

- 라이선스 원문 또는 OSI 라이선스 목록과 맞는가
- 팀의 실제 배포 방식과 관련 있는가
- 의존성 라이선스와 충돌 가능성이 있는가
- 문장이 법률 자문처럼 단정적으로 보이지 않는가

검증한 내용은 CONTRIBUTING의 라이선스 정책 섹션에 반영한다.

## 멘토링

멘토링에서는 팀 저장소의 운영 문서와 라이선스 선택 근거를 확인한다.

확인할 질문은 다음과 같다.

- LICENSE가 존재하고 라이선스 이름이 명확한가?
- CONTRIBUTING에 기여 절차와 라이선스 정책이 있는가?
- SECURITY에 보안 문제 신고 경로가 있는가?
- Pull Request Template에 AI 사용 로그와 라이선스 출처 확인 항목이 있는가?
- AI 라이선스 리스크 답변을 사람이 검증한 흔적이 있는가?
- 4회차에 AI로 분석할 Issue 또는 Pull Request 후보가 정해졌는가?

## AI 활용

AI는 라이선스 차이 요약, CONTRIBUTING 초안, 템플릿 초안, 리스크 시나리오 제안에 사용할 수 있다. 사람이 확인해야 할 것은 라이선스 원문, OSI 승인 여부, 팀의 배포 방식, 의존성 정책, 실제 저장소 상태다.

AI 사용 로그에는 다음을 문장으로 남긴다.

- 어떤 라이선스 리스크를 물었는가
- AI가 어떤 리스크를 제안했는가
- 어떤 공식 출처나 원문으로 확인했는가
- CONTRIBUTING에 어떤 문장을 반영했는가

## 과제

팀 과제는 운영 신뢰 구조 문서를 Pull Request로 제출하는 것이다. 최소한 LICENSE, CONTRIBUTING, Pull Request Template은 포함해야 한다. 가능하면 CODE_OF_CONDUCT, SECURITY, Issue Template까지 포함한다.

개인 과제는 팀 프로젝트에서 사람이 반드시 검증해야 할 라이선스 또는 운영 리스크 두 가지를 작성하는 것이다.

제출 증거는 운영 문서 Pull Request URL, 라이선스 선택 메모, AI 리스크 검토 로그, CONTRIBUTING의 라이선스 정책 섹션이다.
