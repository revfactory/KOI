- 일시: 2026-05-28 목 19:00-21:00
- 주제: Planner, Coder, Reviewer, Tester, Documenter 역할 분리와 Pull Request 완성
- 핵심 메시지: AI를 잘 쓰는 개발자는 하나의 챗봇에게 질문하는 사람이 아니라 역할이 나뉜 AI 작업 흐름을 설계하는 사람이다.
- 목표 산출물: 멀티 에이전트 흐름으로 만든 Pull Request, 테스트 또는 데모, 문서 업데이트, 역할별 AI 사용 기록, 최종 발표 초안 5장, 5분 측정 결과
- 트랙: 외부 오픈소스 기여와 자체 프로젝트 운영으로 나뉘어 활동한다.

5. 6회차 발표 초안을 5장으로 만들고 5분 안에 말해 본다.

## 수업 전 준비

수강생은 4회차 Pull Request 초안과 AI 사용 로그를 다시 읽는다. 한 명의 AI에게 너무 많은 일을 맡겨 결과가 흐려졌던 지점을 하나 찾는다.

팀은 이번 회차에 끝까지 처리할 작은 Issue 하나를 고른다. 또한 6회차 발표에 사용할 공개 증거 링크를 모아 둔다.

## 수업 흐름

1. 4회차에서 AI를 한 명의 도구처럼 썼을 때의 한계를 돌아본다.
2. 하나의 작업을 다섯 역할로 나누는 이유를 이해한다.
3. MCP와 A2A가 역할 분리 흐름과 어떻게 연결되는지 정리한다.
4. 트랙별로 작은 Issue 하나를 멀티 에이전트 흐름으로 처리한다.
5. 안전 장치 네 가지를 Pull Request에 적용한다.
6. 발표 초안 5장을 만들고 5분 안에 말해 본다.
7. 다음 회차 발표와 포트폴리오 정리 준비 항목을 확정한다.

## 핵심 내용

4회차에서는 AI를 기여 보조 시스템으로 사용했다. 그러나 하나의 AI에게 이슈 분석, 구현, 테스트, 리뷰, 문서화를 모두 맡기면 결과가 흐려질 수 있다. 5회차에서는 작업을 역할별로 나누고, 사람은 전체 흐름의 책임자로 검증한다.

Planner는 Issue와 README를 보고 작업을 작은 카드로 나눈다. Coder는 작업 카드 하나를 코드나 문서 변경으로 구현한다. Tester는 변경이 실제로 동작하는지 확인한다. Reviewer는 hallucination, 보안, 라이선스, 의존성, 스타일을 점검한다. Documenter는 README, changelog, Pull Request 본문이 실제 변경과 맞도록 정리한다.

MCP는 AI가 파일, GitHub, 데이터베이스, API 같은 외부 도구에 연결되는 방식과 관련된다. A2A는 여러 에이전트가 서로 역할을 나누고 결과를 주고받는 방식과 관련된다. 이 수업에서 중요한 것은 프로토콜 구현이 아니라 역할 분리와 검증 책임이다.

안전 장치는 네 가지다. AI가 존재하지 않는 API나 파일을 말하면 머지하지 않는다. 비밀키, 개인정보, 과도한 권한이 보이면 머지하지 않는다. 출처가 불명확한 코드나 호환성 불명 의존성이 있으면 머지하지 않는다. 불필요하거나 관리되지 않는 패키지를 추가하지 않는다.

## 활동

### 역할 분리 설계

팀은 선택한 Issue 하나를 다섯 역할로 나눈다.

Planner가 할 일은 문제를 작은 작업 카드로 나누는 것이다. 작업 카드는 완료 조건, 예상 변경 파일, 테스트 방법, 리스크를 포함한다.

Coder가 할 일은 작업 카드 하나만 구현하는 것이다. 범위를 넘기지 않는 것이 중요하다.

Tester가 할 일은 실제 실행 또는 검증 결과를 남기는 것이다. 테스트가 없다면 최소한 재현 절차나 수동 검증 결과를 남긴다.

Reviewer가 할 일은 변경 사항을 검토하는 것이다. 특히 hallucination, 보안, 라이선스, 의존성을 확인한다.

Documenter가 할 일은 README, changelog, Pull Request 본문을 실제 변경과 맞게 갱신하는 것이다.

### 외부 오픈소스 기여 흐름

외부 오픈소스 기여 팀은 대상 저장소의 Issue를 작은 작업 카드로 쪼갠다. fork 저장소에서 하나의 카드만 구현하고, 대상 프로젝트의 CONTRIBUTING 기준에 맞춰 테스트와 문서를 정리한다.

외부 프로젝트의 응답이 늦다면 그것도 오픈소스 현실의 일부로 기록한다. 이 경우 팀 분석 저장소나 자체 보조 저장소에서 같은 멀티 에이전트 흐름을 시연하고, 6회차 발표에서 외부 응답 지연과 대체 전략을 설명한다.

### 자체 프로젝트 운영 흐름

자체 프로젝트 팀은 팀 저장소의 Issue 하나를 골라 작업 카드로 나눈다. 코드 변경이 어려우면 문서 개선이나 테스트 도입도 가능하다.

Reviewer 역할은 팀원 한 명과 AI를 함께 사용할 수 있다. 단, 최종 책임은 팀에게 있다. Documenter는 변경 내용을 README, changelog, Pull Request 본문에 반영한다.

### 역할별 AI 사용 기록

역할별로 다음 내용을 문장으로 남긴다.

- 역할 이름
- AI에게 맡긴 일
- AI가 만든 결과
- 사람이 확인하거나 수정한 내용
- 최종 Pull Request에 반영된 내용

예시는 다음과 같다.

```markdown
## 역할별 AI 사용 기록

### Planner
- 맡긴 일:
- 결과:
- 사람이 확인한 것:

### Coder
- 맡긴 일:
- 결과:
- 사람이 확인한 것:

### Tester
- 맡긴 일:
- 결과:
- 사람이 확인한 것:

### Reviewer
- 맡긴 일:
- 결과:
- 사람이 확인한 것:

### Documenter
- 맡긴 일:
- 결과:
- 사람이 확인한 것:
```

### 발표 초안 5장 만들기

6회차 발표는 5분 발표와 5분 질의응답으로 진행된다. 5회차에서는 발표 초안을 5장으로 압축하고 한 번 시간을 측정한다.

1장은 선택한 문제와 트랙을 설명한다. 2장은 6주 동안 만든 Issue, Pull Request, Commit을 보여준다. 3장은 AI 활용과 사람이 검증한 부분을 보여준다. 4장은 라이선스, 운영 정책, 로드맵을 정리한다. 5장은 배운 점과 다음 기여 약속을 담는다.

발표를 해 본 뒤 실제 발표 시간, 5분 초과 원인, 잘라낼 내용, 추가해야 할 공개 링크를 기록한다.

## 멘토링

멘토링에서는 Pull Request, 역할별 AI 사용 기록, 테스트 또는 데모, 발표 초안을 확인한다.

확인할 질문은 다음과 같다.

- 다섯 역할이 모두 사용되었거나 최소한 시뮬레이션되었는가?
- Reviewer가 hallucination, 보안, 라이선스, 의존성을 확인했는가?
- Tester가 실제 실행 또는 검증 결과를 남겼는가?
- Documenter가 README, changelog, Pull Request 본문 중 하나를 갱신했는가?
- 발표 초안 5장이 있는가?
- 발표가 5분 안에 끝나는가?
- 6회차 발표 8개 항목이 빠지지 않았는가?

## AI 활용

AI는 작업 카드 작성, 코드 또는 문서 초안, 테스트 케이스, 리뷰 코멘트, 발표 구조를 만드는 데 사용할 수 있다. 사람이 확인해야 할 것은 역할 간 모순, 실제 실행, 테스트 의미, 문서와 기능의 일치, 기여 범위 과장 여부다.

역할별 AI 사용 기록에는 프롬프트 전문을 모두 붙일 필요는 없다. 대신 어떤 역할에 무엇을 맡겼고, 사람이 무엇을 검증했는지 명확히 남긴다.

## 과제

팀 과제는 멀티 에이전트 흐름으로 만든 Pull Request 하나를 제출하는 것이다. 코드 기능이 아니어도 문서 또는 테스트 Pull Request가 가능하다. 테스트 또는 데모 증거와 역할별 AI 사용 기록도 함께 제출한다.

추가로 발표 슬라이드 5장 초안과 5분 측정 결과를 제출한다. 개인 과제는 자신이 맡은 역할과 사람이 검증한 내용을 회고하는 것이다.

제출 증거는 Pull Request URL, 실행 또는 테스트 결과, 문서 변경, 역할별 AI 사용 기록, 발표 초안이다.
