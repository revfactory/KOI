# 교육 기획서: AI 시대의 오픈소스 실전

## 과정 개요

- 과정명: AI 시대의 오픈소스 실전: 기여자에서 메인테이너까지
- 한 줄 소개: 대학생·대학원생이 AI 시대의 오픈소스 생태계를 이해하고 GitHub 협업, 라이선스, 기여 문화, AI 개발 도구를 실습해 6주 안에 실제 기여 또는 공개 프로젝트 운영 경험을 완성하는 실전형 과정
- 대상: 대학생·대학원생
- 기간: 2026-05-06부터 2026-06-04까지, 총 6회
- 시간: 매회 19:00-21:00, 한국 시간 기준
- 운영 형식: 강의, 데모, 짧은 실습, 팀별 멘토링
- 최종 약속: 6주 후 모든 팀이 하나의 공개 저장소를 운영하거나 외부 오픈소스 프로젝트에 실제 기여 흔적을 남긴다.

## 기획 배경

오픈소스 교육은 단순한 GitHub 사용법이나 저장소 공개 경험에 머물러서는 충분하지 않다. `source.txt`는 AI, 에이전트, 타입 언어가 개발 생태계의 변화를 만들고 있으며 GitHub 활동도 증가하고 있다는 문제의식을 제시한다. 따라서 이 과정은 학생들이 오픈소스를 사용하는 수준을 넘어 Issue, Branch, Commit, Pull Request, Review, LICENSE, CONTRIBUTING, README, AI 사용 로그까지 공개적으로 확인 가능한 결과물을 남기도록 설계한다.

이 과정에서 AI는 별도 주제가 아니라 매주 반복되는 작업 방식이다. 1회차에는 프로젝트 분석, 2회차에는 PR 요약과 이슈 정리, 3회차에는 라이선스 검토 질문, 4회차에는 기여·문서화·테스트 자동화, 5회차에는 멀티 에이전트 개발 흐름, 6회차에는 포트폴리오 정리에 활용한다. 모든 AI 활용은 사람이 검증한 근거와 함께 기록한다.

### 출처와 인용 공백

`source.txt`에는 GitHub Blog, GitHub Docs, OSI, arXiv, MCP, A2A 관련 인용 표기가 있으나 `[1]`부터 `[11]`까지의 링크 정의가 포함되어 있지 않다. 본 문서는 원문에 제시된 주장과 인용 표기를 교육 설계의 근거로만 사용하며, 실제 배포 전에는 원문 링크를 확인해 참고문헌을 보완해야 한다.

## 학습 목표

수료 시 학습자는 다음을 수행할 수 있어야 한다.

1. 오픈소스 생태계와 기여 문화를 설명하고, 프로젝트의 README, 이슈, PR, 릴리즈, 라이선스 구조를 읽어낼 수 있다.
2. GitHub 기반 협업 워크플로우를 사용해 Issue 생성, Branch 분리, Commit 작성, Pull Request 제출, Review 대응을 수행할 수 있다.
3. LICENSE, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, Issue Template, PR Template이 저장소 신뢰 구조에서 맡는 역할을 설명할 수 있다.
4. AI를 활용해 낯선 저장소 분석, 이슈 재현, PR 요약, 문서화, 테스트 작성, 리뷰 대응을 보조하되 사람이 검증해야 할 지점을 구분할 수 있다.
5. 팀 단위로 외부 오픈소스 기여를 시도하거나 자체 AI·오픈소스 프로젝트를 공개 저장소로 운영할 수 있다.
6. 포트폴리오와 커리어에 연결 가능한 공개 증거를 정리할 수 있다.

## 운영 구조

### 기본 운영 원칙

- 강의는 핵심 개념, 사례, 데모 중심으로 운영하며 매회 주제별 개념을 실제 저장소 화면과 연결한다.
- 실습은 20-30분의 짧은 작업 블록으로 운영하며 이슈 작성, PR 초안, 템플릿 추가, AI 검증 로그 작성처럼 제출 가능한 산출물을 만든다.
- 멘토링은 팀별 프로젝트와 기여 방향을 점검하는 시간으로 운영하며 매회 말 팀별 의사결정과 다음 액션을 확정한다.
- 공개 증거는 저장소와 외부 기여 기록을 중심으로 남긴다. 핵심 증거는 Issue, PR, Commit, Review, README, LICENSE, AI 사용 로그다.

`source.txt`에는 "총 2시간"과 "90분 강의 + 60분 멘토링"이 함께 제시되어 있어 시간 합계가 일치하지 않는다. 본 패키지는 확정 일정인 19:00-21:00을 기준으로 각 강의자료를 2시간 압축안으로 작성하되, 교육 운영안에는 원문의 권장 구조인 강의 90분과 멘토링 60분 모델을 함께 보존한다. 멘토링 60분을 엄격히 운영해야 할 경우 별도 온라인 멘토링 또는 수업 후 연장 블록을 둔다.

### 팀 역할

- Maintainer: 저장소 방향, 이슈 우선순위, PR 병합 기준, 발표 흐름을 관리한다.
- Contributor: 기능 구현, 버그 수정, 테스트 추가, 외부 프로젝트 기여를 실행한다.
- Docs: README, CONTRIBUTING, 튜토리얼, 발표 자료, 포트폴리오를 문서화한다.
- AI/Automation: AI 사용 로그, 자동화 설정, 테스트·리뷰 보조, 인간 검증 체크리스트를 관리한다.

## 팀 프로젝트 트랙

### A트랙: 외부 오픈소스 기여

기존 오픈소스 프로젝트에 실제 기여를 시도하는 트랙이다. 기존 코드 읽기와 커뮤니케이션을 경험하고 싶은 팀에 적합하다. 최소 증거는 외부 이슈, 재현 기록, PR 또는 문서·테스트 기여 흔적이다. README 개선, 번역, 예제 추가에서 기능 구현·CI 개선까지 난이도를 확장할 수 있다. PR이 병합되지 않아도 제출, 리뷰, 대응 기록을 인정한다.

### B트랙: 자체 AI·오픈소스 운영

작은 오픈소스 프로젝트를 직접 만들고 공개 저장소를 운영하는 트랙이다. 문제 정의, 저장소 설계, 운영 문서화를 직접 경험하고 싶은 팀에 적합하다. 최소 증거는 공개 저장소, README, LICENSE, Issue/PR/Commit 기록이다. 교육 자료, 개발자 도구, RAG 데모, MCP 예제, 캠퍼스 자동화 도구처럼 범위를 줄일 수 있다. 기능 규모보다 운영 구조와 공개 증거의 완성도를 우선한다.

## 최종 제출물

- 공개 저장소 또는 외부 기여: 저장소 URL, 외부 Issue/PR URL, Commit 기록, Review 대응 기록
- 운영 문서: README, LICENSE, CONTRIBUTING, CODE_OF_CONDUCT 또는 SECURITY 중 필요한 항목
- 프로젝트 관리 기록: Issue 목록, 라벨, Milestone 또는 프로젝트 보드, PR Template
- AI 활용 기록: 사용 목적, 입력한 요청 요약, AI 결과, 사람이 검증한 항목, 수정 내역
- 발표 자료: 문제 정의, 선택한 트랙, 6주간의 Issue/PR/Commit, 라이선스와 운영 정책, 향후 로드맵
- 개인 회고: 본인의 기여, 배운 점, 다음 오픈소스 액션 플랜

## 평가 기준

- 오픈소스 이해도, 20%: 라이선스 선택 이유, 기여 문화 설명, 운영 파일의 역할 이해, 프로젝트 구조 분석을 인정 증거로 본다.
- GitHub 협업, 20%: Issue, Branch, Commit, Pull Request, Review 사용 기록과 커뮤니케이션 품질을 인정 증거로 본다.
- 실제 기여·운영, 25%: 외부 PR·이슈 기여 또는 자체 공개 저장소 운영 결과, 기능·문서·테스트 산출물을 인정 증거로 본다.
- AI 활용 능력, 20%: AI를 기여·리뷰·문서화·테스트에 적절히 활용하고 사람이 검증한 로그를 인정 증거로 본다.
- 발표·문서화, 15%: README, 발표 자료, 회고, 포트폴리오 연결성, 향후 로드맵을 인정 증거로 본다.

## 운영 리스크와 대응

- 학습자 간 Git/GitHub 숙련도 차이: 1-2회차에 작은 실습을 배치하고, 팀 내 역할을 분리해 입문자는 문서·이슈·재현부터 시작한다.
- 외부 PR이 6주 안에 병합되지 않음: 병합 여부보다 좋은 이슈 작성, 재현, PR 제출, 리뷰 대응 기록을 평가 증거로 인정한다.
- 라이선스·법률 내용을 과도하게 단정함: 교육 목적의 선택 기준과 확인 절차로 설명하고, 실제 법률 판단은 공식 문서와 전문가 확인이 필요하다고 명시한다.
- AI 생성 결과의 오류: 모든 회차에 인간 검증 항목을 포함하고, AI 사용 로그에 검증 내용과 수정 내역을 남긴다.
- 공개 산출물이 약함: 매회 산출물을 최종 발표 항목과 연결하고, Issue/PR/Commit/README 등 확인 가능한 증거를 우선한다.
- 자체 프로젝트 범위가 과도함: 1회차 후보 검토와 2회차 역할 분담에서 최소 기능, 문서, 운영 파일 중심으로 범위를 줄인다.

## 성공 기준

과정 종료 시 팀별로 다음 중 하나 이상이 공개적으로 확인되어야 한다.

- 외부 오픈소스 기여 흔적: 외부 프로젝트 Issue, PR, 리뷰 대화, Commit, 문서·테스트 기여 URL
- 자체 공개 저장소 운영: 공개 GitHub 저장소, README, LICENSE, CONTRIBUTING, Issue/PR 기록
- AI 활용과 검증 기록: AI 사용 로그, 검증 체크리스트, 사람이 수정한 근거
- 발표와 포트폴리오 연결: 최종 발표 자료, GitHub 프로필 또는 pinned repo 정리, 다음 기여 계획
