# Lecture Writing Style

회차 강의자료(`_workspace/04_lecture_{N}_*.md`)를 작성할 때 따르는 표준.

## 파일 헤더 표준

```markdown
# {N}회차 — {제목}

> **핵심 메시지** {한 문장}
> **선수 지식** {1~3줄}
> **이 회차가 끝나면 학생은**
> - {LO 1}
> - {LO 2}
> - {LO 3}

> **회차 일시** 2026-{MM}-{DD}({요일}) 19:00~21:00 (강의 90분 + 멘토링 60분)
> **트랙 분기** A트랙(외부 OSS 기여) / B트랙(자체 OSS 운영)
```

## 분 단위 헤딩 규칙

H2(##)에 분 단위 시간 박스를 붙인다. 학생/강사가 페이스를 즉시 파악할 수 있게 한다.

```markdown
## 0:00~0:10 — 도입: {도입 훅 제목}
## 0:10~0:35 — 개념 1: {핵심 개념}
## 0:35~1:00 — 실습 1 (EX-{회차}A): {실습 제목}
## 1:00~1:10 — 휴식 / 미니 회고
## 1:05~1:30 — 개념 2 + 라이브 데모
## 1:30~1:55 — 실습 2 (EX-{회차}B) / 토론
## 1:55~2:00 — 다음 주 예고 + KPT
## (멘토링) 2:00~3:00 — 팀별 산출물 점검
```

## 강사 톤

- **친근한 대학 강의 톤**: "오늘 우리가 살펴볼 건…", "이 부분이 헷갈릴 수 있는데"
- 반말은 강사 톤 가이드에서만 사용, 학생 핸드아웃 본문은 **존댓말**
- 외래어 첫 등장 시 한글/영문 병기: PR(풀 리퀘스트), Issue(이슈), Branch(브랜치). 두 번째부터는 한글 통일
- 이모지 사용 금지(사용자가 명시 요청한 경우 제외)

## 강사용 vs 학생용 분리

각 슬라이드 단위(##) 내에서 **강사용 메모**는 인용 블록으로 분리:

```markdown
## 0:35~1:00 — 실습 1 (EX-1A): GitHub Trending 분석

(학생 핸드아웃 본문)

> **강사 메모**
> - 이 실습은 학생당 노트북 1대 필수
> - 30명 기준 시간 압박 있음 — 조별로 1개 프로젝트만 분석
> - 트러블슈팅: GitHub Rate Limit 걸리면 토큰 발급 안내
```

## 슬라이드 변환 규칙

마크다운을 그대로 슬라이드(예: Marp, reveal.js)로 변환 가능하도록:

- H2 = 슬라이드 한 장
- H3 = 슬라이드 내 섹션 (선택)
- 한 H2 안의 불릿은 **5개 이내**, 불릿 한 줄은 **40자 이내**
- 코드 블록은 화면에 들어가는 길이로 제한 (15줄 이내, 초과 시 부록으로 분리)
- 표는 4열 이내

## 실습 블록 표준 포맷

```markdown
### 실습 EX-{회차}{알파벳}: {제목}

**소요시간** {분}분
**난이도** 입문/초급/중급/고급
**산출물** {학생 GitHub에 무엇이 남는가}
**도구** {GitHub, Claude Code, 등}

**목표**
{한 문장}

**진행**
1. {1단계}
   ```bash
   {명령어}
   ```
2. {2단계}
3. ...

**체크리스트**
- [ ] {확인할 항목 1}
- [ ] {확인할 항목 2}

**막혔을 때 (트러블슈팅)**
| 증상 | 원인 | 대응 |
|-----|-----|-----|
| {증상1} | {원인1} | {대응1} |

**확장 (시간 남으면)**
- {추가 도전 과제}
```

## AI 활용 데모 포맷

매 회차 1개 이상의 AI 활용 데모를 포함한다.

```markdown
### 라이브 데모: AI로 {작업}하기

**도구** Claude Code / Cursor / Copilot 중 강사 선택
**입력 프롬프트 예시**
\`\`\`
{실제 프롬프트 텍스트}
\`\`\`

**기대 응답 패턴**
- {AI가 무엇을 출력하는가}
- {강사가 학생과 함께 검증할 포인트}

**사람이 검증할 부분**
1. {AI 출력의 어느 부분을 사람이 확인해야 하는가}
2. {hallucination 위험 지점}
```

## 인용 양식

reference-curator의 `_workspace/03_reference_pool.md`에 등재된 출처만 인용한다.

```markdown
GitHub Issues는 아이디어, 피드백, 작업, 버그를 추적하는 도구로 프로젝트 관리와 연결됩니다.[^github-issues]

[^github-issues]: GitHub Docs, "About issues", https://docs.github.com/en/issues, 검색일 2026-05-04
```

또는 인라인 링크 + 부록:

```markdown
... 자세한 내용은 [GitHub 공식 문서](https://docs.github.com/en/issues) 참고.
```

회차 끝의 "부록: 인용 출처" 섹션에 모든 인용을 한 번 더 정리한다.

## 회차 끝 표준 섹션

모든 회차는 다음 섹션으로 마무리:

1. **다음 주 예고** — 다음 회차의 핵심 메시지·선수 과제 (3~5줄)
2. **사후 과제** — 학생이 다음 회차 전까지 해야 할 일 (체크리스트)
3. **멘토링 60분 가이드** — 멘토가 팀별로 무엇을 점검하는지
4. **강사 트러블슈팅** — 자주 막히는 지점과 대응
5. **부록: 슬라이드 변환 가이드** — 발표 도구별 변환 팁
6. **부록: 인용 출처** — 이 회차의 모든 인용

## 분량 기준

- 회차 한 파일은 **마크다운 1500~2500줄** 목표
- 1회차/6회차는 더 길어도 됨 (각각 오리엔테이션/발표로 콘텐츠 많음)
- 코드/실습이 많은 회차(2, 4, 5)는 줄 수 상한 완화
- 분량 초과 시 우선순위가 낮은 사례·부연 설명 압축, 필수 실습·체크리스트는 보존
