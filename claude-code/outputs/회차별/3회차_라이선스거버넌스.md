> **이 회차의 한 줄 요약**: 라이선스를 "법률 지식"이 아니라 **오픈소스 운영의 신뢰 구조**로 이해하고, LICENSE 한 파일을 넘어 LICENSE/CONTRIBUTING/COC/SECURITY/Templates 5종 세트를 팀 저장소에 적용한다.
> **외래어 표기 원칙**: 본 강의자료의 외래어 표기는 첫 등장 시 영문 + 한글(약어)을 병기하고, 두 번째부터 약어/한글 표기로 통일한다.

### 0.1 핵심 메시지

> **오픈소스는 "공개"가 아니라 사용·수정·배포·기여를 허용하는 신뢰 구조다.**
> **라이선스는 그 신뢰의 첫 페이지다.**

### 0.2 학습 목표

- **LO 1**: 학생은 MIT, Apache-2.0, GPL-2.0, GPL-3.0, BSD-3-Clause, MPL-2.0의 핵심 차이를 사용 시나리오(개인/기업/SaaS/연구) 단위로 매핑해 설명할 수 있다.
- **LO 2**: 학생은 자기 팀 저장소에 LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT.md, SECURITY.md, Issue/PR Template을 추가하고, 그 이유를 설명할 수 있다.
- **LO 3**: 학생은 AI 시대의 라이선스 쟁점(모델 weights, 학습 데이터, OSI Open Source AI Definition)을 한 사례 이상으로 설명할 수 있다.

### 0.3 선수 지식

- 2회차 완료: 팀 GitHub 저장소 보유 + 첫 머지 PR 경험 + 트랙 확정 (A/B)

## 5. 0:13~0:30 — 개념 1블록: 라이선스 6종 비교 매트릭스 (17분)

라이선스 6종을 암기할 필요는 없다. 대신 "내 프로젝트가 라이브러리인지, SaaS인지, 연구 코드인지"를 기준으로 선택지를 좁힐 수 있어야 한다.

### 5.1 슬라이드 7 — 라이선스 6종 한 줄 정의

- **MIT**
  - 핵심: 거의 자유롭게 사용·수정·배포할 수 있고, 출처와 저작권 표시만 유지하면 된다.
  - 카피레프트 강도: 없음.
  - 대표 감각: 개인 프로젝트, 교육용 코드, 널리 퍼지길 원하는 라이브러리에 잘 맞는다.

- **Apache-2.0**
  - 핵심: MIT와 비슷하게 자유롭지만, 특허 권리 부여와 NOTICE 처리 규칙이 명시된다.
  - 카피레프트 강도: 없음.
  - 대표 감각: 기업이 가져다 쓰기 쉬운 라이브러리·도구에 잘 맞는다.

- **GPL-2.0**
  - 핵심: 파생물을 배포할 때 같은 GPL 계열 의무를 유지해야 하는 강한 카피레프트 라이선스다.
  - 카피레프트 강도: 강함.
  - 대표 감각: Linux처럼 자유 소프트웨어 생태계를 강하게 보장하려는 프로젝트에 쓰인다.

- **GPL-3.0**
  - 핵심: GPL-2.0의 강한 카피레프트에 특허·티보이제이션 관련 보완이 더해진다.
  - 카피레프트 강도: 강함.
  - 대표 감각: GPL-2.0보다 현대적 호환성과 특허 이슈를 더 의식할 때 선택한다.

- **BSD-3-Clause**
  - 핵심: MIT처럼 자유롭지만, 프로젝트 이름이나 기여자 이름을 홍보에 함부로 쓰지 못하게 하는 조항이 있다.
  - 카피레프트 강도: 없음.
  - 대표 감각: 학술·인프라 프로젝트에서 "자유로운 사용 + 이름 오용 방지"가 필요할 때 맞다.

- **MPL-2.0**
  - 핵심: 파일 단위 카피레프트다. MPL 파일을 수정하면 그 파일은 공개해야 하지만, 전체 프로젝트를 모두 같은 라이선스로 묶지는 않는다.
  - 카피레프트 강도: 약함.
  - 대표 감각: 라이브러리나 제품 일부를 공개하면서 전체 결합은 유연하게 남기고 싶을 때 맞다.

참고로 원문 자료에 등장한 **AGPL-3.0**은 GPL에 네트워크 사용 공개 의무를 더한 가장 강한 카피레프트이고, **LGPL**은 라이브러리 동적 링크에는 더 유연한 약한 카피레프트다.

### 5.2 슬라이드 8 — 사용 시나리오 매핑

- **개인 프로젝트**
  - 추천: MIT, Apache-2.0, BSD-3-Clause.
  - 가능: GPL-2.0, GPL-3.0, MPL-2.0.
  - 주의: AGPL-3.0은 네트워크 공개 의무까지 고려해야 한다.

- **기업 라이브러리**
  - 강력 추천: Apache-2.0. 특허 권리 명시가 있어 기업 사용자가 이해하기 쉽다.
  - 추천: MIT, BSD-3-Clause.
  - 신중: GPL-2.0, GPL-3.0은 파생물 공개 의무가 강하다.
  - 가능: MPL-2.0은 파일 단위 공개로 타협점을 만들 수 있다.
  - 참고: LGPL은 라이브러리 동적 링크 상황에서 자주 검토된다.

- **SaaS 서비스**
  - 추천: MIT, Apache-2.0, BSD-3-Clause.
  - 가능: GPL-3.0. 단, SaaS로만 제공하면 일반적으로 배포가 아니므로 공개 의무가 발동하지 않을 수 있다.
  - 매우 신중: AGPL-3.0은 네트워크 사용에도 공개 의무를 걸기 위해 만들어졌다.
  - 가능: MPL-2.0은 파일 단위 공개를 전제로 검토할 수 있다.

- **연구 코드**
  - 추천: MIT, Apache-2.0, BSD-3-Clause. 재사용과 재현을 쉽게 만든다.
  - 가능: GPL-2.0, GPL-3.0, MPL-2.0.
  - 신중: 연구 결과를 산업체와 함께 쓰거나 SaaS로 전환할 가능성이 있으면 강한 카피레프트는 미리 합의해야 한다.

핵심 패턴은 단순하다. 기업·라이브러리·범용 공개는 MIT/Apache/BSD가 편하고, 자유 보장을 강하게 밀고 싶으면 GPL 계열을 검토한다. 전체가 아니라 파일 단위로 공개 범위를 나누고 싶으면 MPL-2.0을 본다.

### 5.3 슬라이드 9 — 대표 프로젝트 매핑

- **MIT**: React, Vue, Next.js, Express, Ruff, uv, toss/slash.
- **Apache-2.0**: Kubernetes, TensorFlow, Airflow, Mistral Large 3, OLMo 2, kakao/khaiii.
- **GPL-2.0**: Linux, Bash, GIMP, WordPress.
- **GPL-3.0 / AGPL-3.0 계열 참고**: Nextcloud, Mastodon, OpenInterpreter, MongoDB(과거).
- **LGPL 참고**: glibc, FFmpeg(부분), Qt(부분).
- **MPL-2.0**: Mozilla 제품군(Firefox 일부, pdf.js — 일부 Apache 전환).

> 자료 풀 §5.1 라이선스 케이스북. 강의자료 §3.3 회차 3 사례 저장소 참조.

### 5.4 슬라이드 10 — 라이선스 호환성 1페이지 메모

```text
MIT / BSD / Apache-2.0  -> 모든 곳에 흡수 가능 (가장 자유로운 흐름)
GPL                      -> 다른 GPL 프로젝트와 결합할 때 의무가 강하게 따라옴
                            (GPL-2.0과 Apache-2.0은 충돌, GPL-3.0은 호환 가능)
AGPL                     -> 가장 제약 강함, SaaS 의무까지 고려
LGPL                     -> 라이브러리 동적 링크면 비공개 가능
MPL                      -> 파일 단위 카피레프트

핵심 원칙:
강한 카피레프트(GPL/AGPL) 코드를 약한 라이선스(MIT/Apache/BSD)
프로젝트에 복사·붙여넣기 하면 안 된다.
복사하는 순간 너의 프로젝트 전체가 GPL/AGPL 의무에 묶일 수 있다.
```

> 자료 풀 §5.4 라이선스 호환성 핵심.

### 5.6 핸드아웃 라이선스 결정 트리

```text
START — 너희 팀 프로젝트는 무엇인가?

  ├ 라이브러리 또는 도구 (남이 가져다 쓸 것) ?
  │   ├ 가장 자유롭게 퍼지길 원함 -> MIT
  │   ├ 기업이 안전하게 쓰길 원함 (특허 보호) -> Apache-2.0
  │   ├ 이름 오용 방지 조항까지 원함 -> BSD-3-Clause
  │   └ 파일 단위 공개 균형을 원함 -> MPL-2.0
  │
  ├ 애플리케이션 / 도구 / 서비스 ?
  │   ├ 자유 보장이 핵심 가치 -> GPL-3.0
  │   ├ 오래된 GPL 생태계와 맞춰야 함 -> GPL-2.0
  │   ├ SaaS여도 코드 공개 강제 -> AGPL-3.0
  │   └ 기업 친화 + 자유 균형 -> Apache-2.0 또는 MIT
  │
  └ 잘 모르겠음 / 학생 학습 프로젝트
      -> 일단 MIT 또는 Apache-2.0

결정 못 하면 https://choosealicense.com/ 에서 5분 안에 결정한다.
```


### 7.2 슬라이드 13 — OSI Open Source AI Definition 1.0

OSI가 정의한 **Open Source AI System**의 4 freedoms는 다음과 같다.

- **Use**: 목적 제한 없이 사용한다. 별도 허가를 요구하지 않는다.
- **Study**: 시스템이 어떻게 동작하는지 검사할 수 있다.
- **Modify**: 목적에 맞게 수정할 수 있다. 출력 변경도 포함한다.
- **Share**: 수정 여부와 무관하게 재배포할 수 있다.

이 자유를 행사하려면 다음 3가지 접근이 필요하다.

- **데이터 정보**: 학습 데이터에 대한 충분한 설명. 데이터 자체 공개가 아니라, 합법성까지 고려한 "정보"를 요구한다.
- **코드**: 학습과 실행에 사용된 완전한 소스 코드.
- **파라미터**: 모델 weights와 configuration.

> 출처: OSI, *The Open Source AI Definition – 1.0* (2024-10-28). https://opensource.org/ai/open-source-ai-definition

### 7.3 슬라이드 14 — Llama vs Apache-2.0

**Llama 4 Community License (Meta)**

- 사용: 조건부로 가능.
- 수정: 조건부로 가능.
- 공유: "Built with Llama" 의무가 붙는다.
- 특허 권리: Apache-2.0처럼 명시적으로 정리되어 있지 않다.
- 사용자 한계: 700M MAU 이상이면 별도 허가가 필요하다.
- 학습 데이터: 비공개.
- OSI 인정: OSI 기준 오픈소스로 보기 어렵다.

**Apache-2.0 (OLMo 2 사례)**

- 사용: 가능.
- 수정: 가능.
- 공유: 가능.
- 특허 권리: 명시되어 있다.
- 사용자 한계: 없음.
- 학습 데이터: 공개.
- OSI 인정: OLMo 2는 OSI 정의 충족 사례로 볼 수 있다.

> 자료 풀 §5.2.

## 8. 1:10~1:22 — AI에게 라이선스 리스크 질문 (12분, AI 미니, 보강 1+2)

라이선스를 외우는 것보다 중요한 것은 자기 프로젝트에 적용해보는 것이다. 이 구간에서는 AI에게 6개월 뒤 리스크를 묻고, 그 답을 사람이 검증해서 CONTRIBUTING.md에 남긴다.

### 8.1 산출물 개요

- **분량**: 12분.
- **트랙**: 공통(A/B 모두).
- **AI 도구**: Claude / Copilot / Cursor 중 1개. 학생 본인 계정을 사용한다.
- **산출물**: AI 출력, 사람 검증 메모, CONTRIBUTING.md "라이선스 정책" 섹션 추가 PR.

### 8.2 슬라이드 16 — 진행 절차

```text
12분 안에 너희가 할 일

  STEP 1 (3분)  너희 라이선스 + 프로젝트 한 줄 요약을 AI에 입력
  STEP 2 (3분)  AI에 다음 질문:
                "이 라이선스를 우리 프로젝트에 쓸 때
                 6개월 뒤 발생할 수 있는 리스크 3개"
  STEP 3 (3분)  AI 답변 받음 + 1개 이상 사람이 검증
                (자료 풀 §5 / OSI / choosealicense.com 으로 확인)
  STEP 4 (3분)  검증 결과를 CONTRIBUTING.md "라이선스 정책" 섹션에 녹임
                + PR 머지

AI 답변을 그대로 안 쓴다.
사람이 1줄이라도 수정/확인한 흔적이 남아야 한다.
```

### 8.3 슬라이드 17 — 실제 프롬프트 예시

#### 프롬프트

```text
나는 [팀명]이다. 우리는 [프로젝트 한 줄 설명]을 만들고 있다.
저장소 URL은 [https://github.com/...]이다.
우리는 [라이선스 SPDX] 라이선스를 선택했다.

이 라이선스를 우리 프로젝트에 쓸 때 6개월 뒤 발생할 수 있는
리스크 3개를 구체적 시나리오와 함께 알려줘.
각 리스크마다:
  1. 시나리오 1줄
  2. 발동 조건 1줄
  3. 우리가 미리 할 수 있는 대응 1줄

마지막에 OSI 또는 choosealicense.com URL 1개 이상을
검증 자료로 함께 제공해줘.
```

#### 예상 응답 패턴

```text
리스크 1: 의존성 라이선스 충돌
  시나리오: 너희 MIT 프로젝트가 GPL 라이브러리를 의존성으로 추가하면,
           너희 프로젝트 전체가 GPL 의무에 묶일 가능성.
  발동 조건: GPL 라이브러리를 직접 import 하고, 너희 코드와 같은
           바이너리/배포물에 포함될 때.
  대응: 의존성 추가 전 SPDX 라이선스 확인 + ScanCode/FOSSA 자동 검사
       워크플로우 추가.
  검증: https://choosealicense.com/licenses/mit/

리스크 2: 특허권 분쟁
  시나리오: MIT는 특허 권리를 명시하지 않으므로, 기여자가 자기
           특허를 주장하면 사용자가 분쟁에 휘말릴 수 있음.
  발동 조건: 외부 contributor가 특허 보유한 알고리즘을 기여하고
           추후 권리 주장.
  대응: Apache-2.0으로 변경 검토 (특허 권리 명시) 또는 CLA 도입.
  검증: https://opensource.org/licenses/MIT vs https://opensource.org/licenses/Apache-2.0

리스크 3: 외부 contributor 부재
  시나리오: MIT가 너무 자유로워서 외부 contributor가 "왜 굳이 기여?"
           라고 느낄 수 있음 — 카피레프트로 자유 보장 호소력 부재.
  발동 조건: 6개월 뒤에도 외부 PR이 0건일 때.
  대응: 이슈 라벨 정비 + good-first-issue 풀 + 응답 SLA 약속.
  검증: https://opensource.guide/best-practices/
```

### 8.4 사람 검증 단계

AI 답변 중 **최소 1개 리스크**에 대해 다음 중 하나를 한다.

```text
[ ] 검증 방법 A — 답변에 나온 URL 직접 방문해서 1줄 인용 추가
[ ] 검증 방법 B — 자료 풀 §5 라이선스 케이스북에서 같은 사례 찾기
[ ] 검증 방법 C — choosealicense.com 또는 opensource.org에서 반대 의견 찾기
[ ] 검증 방법 D — AI 답변이 틀렸다고 판단하면 그 이유를 1줄로 적기
```

검증 결과는 CONTRIBUTING.md에 다음 형식으로 녹인다.

```markdown
## 라이선스 정책
우리는 [라이선스 이름]을 선택했다. 이유: ...

### 알려진 리스크 (AI 분석 + 사람 검증)
- 리스크 1: [한 줄 시나리오]
  - 대응: [한 줄]
  - 검증 출처: [URL]
- 리스크 2: ...
- 리스크 3: ...

### AI 사용 로그
- 단계: 라이선스 리스크 분석
- 도구: Claude / Copilot / Cursor 중 사용한 도구
- 프롬프트 요약: "6개월 뒤 리스크 3개..."
- 사람이 검증한 것: 리스크 1 URL 직접 확인 / 리스크 2는 OSI 페이지 추가
```

### 8.5 예상 상황과 대응

- **AI 답변이 너무 일반적인 경우**: "리스크: 라이선스 위반"처럼 추상적이면 구체 사례로 다시 묻는다. 예: "GPL 라이브러리를 MIT 프로젝트에 import 하면 어떻게 되나?"
- **AI 답변에 환각이 있는 경우**: 없는 URL이나 가짜 사례를 발견했다면 그 사실 자체를 검증 메모에 남긴다. 검증 없는 AI 사용의 위험을 보여주는 사례다.
- **AI 도구 접근 권한이 없는 경우**: 옆 사람과 페어로 진행한다. 본인이 프롬프트를 작성하고, 페어가 입력·출력을 도와준다.
- **12분 안에 CONTRIBUTING 머지를 못 하는 경우**: AI 출력 + 검증 메모를 GitHub Issue 1개로 우선 발행하고, 멘토링 60분에 PR로 정리한다.

### 8.6 1:22 핵심 정리

> **AI 활용 능력 = 검증 능력**이다. 평가에서는 "AI를 썼는가"보다 "AI를 사용하고 검증했는가"를 본다. 오늘 CONTRIBUTING.md에 한 줄이라도 사람이 검증한 흔적이 남으면, 그것이 4회차 본격화의 첫 기반이다.

## 12. 사후 과제 (수업 후 다음 주까지)

본 회차 산출물 6개는 1주일 뒤 4회차에서 다시 등장한다. AI가 짠 코드를 머지할 때 "라이선스 출처 확인" 룰을 발동시키려면, 이번 주에 5종 세트를 실제 저장소에 남겨야 한다.

### 12.1 필수 과제 (1시간 이내)

1. **본 회차에 머지 못 한 5종 세트 항목 마무리**
   - 산출물: Community Standards 100%.
   - 마감: 2026-05-18 (일) 23:59.
2. **"라이선스 정책" 섹션을 CONTRIBUTING.md에 녹임**
   - 산출물: PR 1개 머지.
   - 마감: 2026-05-18 (일) 23:59.
3. **4회차에 분석할 외부 OSS 1개 또는 자체 저장소 Issue 1개 사전 결정**
   - 산출물: 팀 단톡방 또는 Discussions에 URL 공유.
   - 마감: 2026-05-20 (화) 23:59.

### 12.2 강력 권장 과제 (1시간 이내)

1. **GitHub Settings -> Code security -> Secret scanning ON / Dependabot ON**
   - 산출물: 스크린샷 1장.
2. **`.gitignore` 표준 템플릿 적용**
   - 참고: https://github.com/github/gitignore
   - 산출물: PR 머지.
3. **우리 팀이 고른 라이선스의 OSI 인증 페이지 1줄 인용을 CONTRIBUTING에 추가**
   - 산출물: PR 머지.

### 12.3 선택 과제 (시간 여유 있을 때)

1. **ScanCode Toolkit을 자기 저장소에 1회 돌려보고 결과 캡처**
   - 참고: https://github.com/aboutcode-org/scancode-toolkit
2. **다른 팀의 5종 세트를 봐주고 PR 코멘트 1개 남기기**
   - 목적: 상호 리뷰.
3. **OLMo 2의 LICENSE/CONTRIBUTING/COC를 분석하고 우리와 비교 메모 1장 작성**
   - 참고: https://github.com/allenai/OLMo

### 12.4 4회차 사전 학습 (다음 주 수업 전 30분)

> **상세는 4회차 강의자료 §1에서 안내**. 미리 보면 좋은 것:
>
> - Claude Code (CLI) 또는 Cursor 설치 (학생 무료 옵션)
> - 자료 풀 §4 AI 도구 카탈로그 — 6종 비교 자료 1회 훑기
> - "낯선 저장소"로 분석할 후보 1개 결정. 4회차의 첫 분석 활동에서 5분 안에 사용할 후보가 된다.
