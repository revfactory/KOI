## 5. 0:08~0:18 — GitHub 프로필 정돈 1차 (10분)

### 5.1 why-now

> 직전 5분에 "내 프로필이 채용 담당자에게 어떻게 보이는가"를 옆 사람을 통해 들었다. 그 충격이 식기 전에, 10분 안에 1차 정돈을 끝낸다. 완벽이 아니라 "대분류" — README 1단락 / pinned 6개 / 사회 링크. 미세 조정은 사후 과제.

### 5.2 정돈 순서

1. **프로필 README 1단락**: 한 줄 자기소개, 한 줄 관심 분야, 한 줄 동작 호출을 적는다. 영어든 한국어든 괜찮고, 둘 다면 더 좋다.
2. **pinned repo 6개**: 팀 저장소 1개, 본 강의 누적 산출물 저장소 1개, 본인이 자랑하고 싶은 4개를 고른다. 빈 자리는 약점으로 보일 수 있다.
3. **사회 링크와 이메일**: LinkedIn, 블로그, 이메일 등 외부에서 연락 가능한 경로를 최소 1개 연결한다.
4. **프로필 사진과 Bio**: 기본 회색 실루엣을 피하고, 160자 Bio에 관심 분야를 압축한다.
5. **1분 갤러리 워크**: 옆 사람과 바뀐 점을 30초씩 확인한다.

```text
   - Settings → Public profile → Activity 모두 공개
```

### 5.4 프로필 README 1단락 표준 골격 (학생이 그대로 복붙)

```markdown

- 🌱 **지금 배우는 것**: [한 줄]
```

### 5.5 pinned repo 6개 선택 가이드

- **1번 자리 — 본 강의 팀 저장소**: 최근 6주 동안 실제로 협업한 경험을 보여준다.

- **3번 자리 — 가장 코드 비중이 큰 본인 프로젝트**: 코드 능력을 증명한다.
- **4번 자리 — 특정 도메인 프로젝트**: AI, Web, Data 등 관심 분야 시그널을 만든다.
- **5번 자리 — 잘 정리된 학습용 저장소**: 학습 능력과 문서화 능력을 보여준다.
- **6번 자리 — 실험용 또는 사이드 프로젝트**: 본인의 호기심과 성장 폭을 보여준다.

## 6. 0:18~1:08 — 팀별 5분 발표 + 5분 Q&A (50분)

### 6.1 why-now

> 6주 동안 GitHub에 흔적을 쌓았다. 그러나 **공개적으로 본인 입으로 말한 적**은 한 번도 없다. 발표 5분은 그 흔적을 본인 입으로 정의하는 의식이다. 5팀 기준으로 50분을 사용한다.

### 6.2 5팀 기준 발표 진행 순서

1. **1팀**: 발표 0:20:00~0:25:00, Q&A 0:25:00~0:30:00
2. **2팀**: 발표 0:30:30~0:35:30, Q&A 0:35:30~0:40:30
3. **3팀**: 발표 0:41:00~0:46:00, Q&A 0:46:00~0:51:00
4. **4팀**: 발표 0:51:30~0:56:30, Q&A 0:56:30~1:01:30
5. **5팀**: 발표 1:02:00~1:07:00, Q&A 1:07:00~1:08:00

6~8팀일 경우 2그룹 병렬 운영이 될 수 있다. 이때도 각 팀 발표의 핵심은 5분 안에 8개 항목을 모두 보여주는 것이다.

### 6.2.1 발표 5분 권장 분배

```text
[발표 0:00~0:30]  발표자 인사 + 슬라이드 1 — 우리 문제
[발표 0:30~1:00]  슬라이드 2 — 6주 흔적
[발표 1:00~3:00]  슬라이드 3 — AI 활용 + 검증
[발표 3:00~4:00]  슬라이드 4 — 라이선스·로드맵
[발표 4:00~4:30]  슬라이드 5 — 배운 점 + 다음 기여
[발표 4:30~5:00]  핵심 URL, 다음 약속, 마무리 한 줄
[Q&A 5:00~10:00]  동료 팀 질문 먼저, 답변은 30초 안에
```

### 6.2.2 Q&A 진행 룰

- 동료 팀 질문 1~2개를 우선 받는다.
- 멘토 질문 또는 코멘트는 1개로 압축한다.
- 답변이 길어지면 "더 자세한 내용은 README 또는 이슈로 답변하겠다"고 말하고 끊는다.
- 마지막 1분은 박수와 다음 팀 전환에 쓴다.

### 6.3 학생 행동 매뉴얼 — 발표 팀

```text
[T-30분]  발표 슬라이드가 발표용 화면 또는 공유 기기에 준비됐는지 확인
[T-10분]  발표자 3명 모두 대기
[T-2분]   슬라이드 첫 페이지 띄워두기 + 마이크 테스트
[T-0]     발표 시작 — 5분 정확히
[T+5]     Q&A — 답변은 30초 안에. 길어지면 "더 자세한 건 README 보세요"
[T+10]    박수 + 다음 팀에게 자리
```

### 6.3.1 학생 핸드아웃 — 발표 직전 자가 채점 8항목 (5분)

```text
발표 직전 학생 자가 채점 — 5분 점검표 (팀당 1장)

☐ 핵심 메시지 1문장이 첫 슬라이드에 보이는가
☐ 6주 산출물(후보→PR→신뢰5종→AI 로그→멀티PR→발표) 5개 이상이 시각으로 보이는가
☐ 5분 안에 끝낼 수 있는 분량인가(슬라이드 ≤ 8장)
☐ AI 사용 로그가 1번 이상 인용/스크린샷으로 등장하는가
☐ 다음 30일 안에 할 다음 기여 1건이 명시되어 있는가
☐ 데모 영상이 1분 이내인가(또는 캡처 3장으로 대체)
☐ 마지막 슬라이드에 GitHub URL과 연락처가 있는가
☐ Q&A 5분 동안 답할 수 없으면 "확인 후 이슈로 답변" 멘트 준비됐는가
```

### 6.4 학생 행동 매뉴얼 — 청중 팀

```text
□ 발표 듣는 중 — 핸드아웃 §6.5 발표 청취 노트에 1팀당 3줄 메모
□ 발표 끝 — 30초 안에 Q&A 질문 1개 만들기
□ 좋은 질문 = "왜 그 라이선스를 골랐나" / "AI가 틀렸을 때 어떻게 알았나" / "30일 후 그 약속이 진짜 가능한가"
□ Q&A 안 만들면 학습 안 한 셈 — 모든 청중은 발표마다 1개 질문 후보를 메모
```

## 7. 1:08~1:13 — 본 회차 유일한 강의 블록 (5분)

### 7.1 why-now

> 90분 중 강의는 단 5분이다. 이 5분에 "오픈소스 커리어 + 좋은 기여자의 태도 + 메인테이너로 성장하는 길"을 다 담는다. 압축의 압축.

### 7.2 슬라이드 (3장)

#### 슬라이드 6-4 (1:08~1:10) — 좋은 기여자의 태도 4가지

**제목**: 좋은 기여자는 누구인가

- **1. 작은 기여**: 1줄 typo, 번역, docs PR로 시작한다. 시작점은 `up-for-grabs`, First Timers Only, `good-first-issue` 같은 라벨이다.
- **2. 꾸준함**: 1주 1 PR보다 1년 50 PR이 강하다. contribution graph가 꾸준함을 증명한다.
- **3. 리뷰 대응**: 리뷰어의 코멘트에 24시간 안에 응답한다. 받아들이지 않더라도 답변, 근거, 재PR로 이어간다.
- **4. 문서화**: 코드만 짜는 사람은 좋은 기여자로 충분하지 않다. README, CHANGELOG, comment까지 남겨야 한다.

#### 슬라이드 6-5 (1:10~1:11) — 오픈소스 커리어 5경로

**제목**: GitHub 흔적이 만드는 다음 자리

```text
1. 인턴십 — 채용 담당자 30초가 본인 GitHub. 기여 그래프가 살아있는가.
2. 연구 — 연구실 코드를 GitHub에 공개. 동료 평가가 인용으로 이어진다.
3. 취업 — 1년 200 PR이 학력보다 강한 신호일 수 있다.
4. 창업 — 본인이 만든 OSS의 user 100명이 첫 customer 100명이다.
5. 커뮤니티 — 한국 OSS 커뮤니티 (KOSSLAB · 한국 카카오·네이버·라인 OSS) → 멘토 → 메인테이너.
```

#### 슬라이드 6-6 (1:11~1:13) — 메인테이너로 성장하는 길

**제목**: 기여자 → 메인테이너 6단계

```text
1. 첫 PR 머지              ← 1~5회차에서 했음
2. 같은 저장소 5번째 PR     ← 30일 액션 플랜
3. Issue triage 도와주기    ← 90일 액션
4. 다른 기여자 PR 리뷰      ← 6개월 안
5. 작은 sub-module 책임     ← 1년 안
6. core maintainer 후보     ← 2년 이상
```

## 10. 1:30~2:00 — 6주 통합 회고 (KPT 확장, 30분)

### 10.1 why-now

> 매 회차 끝 5분 KPT는 단편이었다. 6주 통째로 보면 "Keep이 무엇" / "Problem이 무엇" / "Try가 무엇"의 패턴이 나온다. 이 30분이 본 강의의 진짜 종강 의식이다. AI 활용 회고도 같은 양식에 묶는다 (보강 요청 2).

### 10.1.1 5분 미니 KPT와 30분 통합 KPT 비교

#### 5분 미니 KPT (Keep/Problem/Try, 1:25~1:30)

5분 미니 KPT의 목적은 오늘 발표 직후 Keep, Problem, Try를 1줄씩 빠르게 남기는 것이다. 개인 3분 작성과 2분 단톡방 또는 옆자리 공유로 진행한다. 종이에 즉시 쓰고, 오늘 발표에서 남은 감정과 배운 점이 식기 전에 기록한다.

학생 산출물은 Keep 1줄, Problem 1줄, Try 1줄이다. 이 3줄은 30분 통합 KPT의 입력으로 그대로 가져간다.

#### 30분 통합 KPT (1:30~2:00)

30분 통합 KPT의 목적은 6주 전체 패턴, AI 활용 회고, 30일 약속을 하나로 묶는 것이다. 안내 2분, 개인 작성 18분, 팀 공유 5분, 전체 공유 3분, 마무리 2분으로 진행한다.

진행 방식은 §10.3 양식에 따라 6주 산출물, KPT, AI 회고, 30일 약속을 순서대로 작성하는 것이다. 학생 산출물은 통합 회고 마크다운 1개와 프로필 README에 넣을 졸업 1줄이다. 파일명은 `docs/retrospective/2026-final-<학생ID>.md`를 권장한다. D+7까지 팀 저장소에 push하고, D+30 재모임에서 다시 연다.

### 10.2 30분 통합 회고 진행 절차

1. **1:30~1:32, 안내 2분**: 양식 1장을 받고 6주 회고 룰을 확인한다.
2. **1:32~1:50, 개인 작성 18분**: 학생 1인당 1장씩 조용히 작성한다.
3. **1:50~1:55, 팀 공유 5분**: 팀 안에서 1인 1분씩 가장 중요한 1줄을 소리 내어 읽는다.
4. **1:55~1:58, 전체 공유 3분**: 몇 명이 1줄씩 공유하고, 다른 학생은 본인 회고에 옮길 표현을 메모한다.
5. **1:58~2:00, 마무리 2분**: 30일 후 재모임 일자와 follow-up 채널을 최종 확인한다.

## 13. A/B 트랙 분기 — 6회차는 통합 (분기 없음)

> 6회차 발표는 두 트랙이 같은 8개 항목으로 발표한다. 이로써 평가 형평성이 유지된다.

### 13.1 본 회차 분기 없음의 이유

- **평가 형평성**: A/B 트랙이 발표 형식·평가 항목이 다르면 채점이 어그러진다. 8개 항목은 트랙과 무관하게 통일한다.
- **청중 학습**: A트랙과 B트랙 결과를 동일 무대에서 보면 서로 배운다. 분기 운영은 학습 손실이 크다.
- **30일 액션 플랜**: A/B 트랙과 무관하게 같은 양식을 쓴다. 저장소, 기여 종류, AI 도구, 검증 방법을 모두 채운다.

### 13.2 단, 발표 슬라이드에서 트랙 구분 1줄

발표 슬라이드 1번 상단에 **"트랙: A (외부 OSS 기여)"** 또는 **"트랙: B (자체 공개 저장소 운영)"** 1줄을 명시한다. 청중도 트랙 차이를 알고 질문해야 한다.

### 13.3 트랙별 발표 8개 항목 차이 (참고)

- **1. 우리 문제**
  - A트랙: 외부 OSS의 어떤 사용자/개발자 문제인지 설명한다.
  - B트랙: 자체 프로젝트의 어떤 문제인지 설명한다.
- **2. 선택한 OSS 또는 자체 프로젝트**
  - A트랙: 외부 OSS 이름과 URL을 제시한다.
  - B트랙: 자체 프로젝트 이름과 URL을 제시한다.
- **3. 6주 Issue/PR/Commit**
  - A트랙: 외부 PR(머지·미머지)과 분석 저장소를 보여준다.
  - B트랙: 자체 저장소와 외부 fork·issue 받기 노력을 보여준다.
- **4. AI 활용**
  - A트랙: 낯선 저장소 분석과 PR 본문 작성에 AI를 어떻게 썼는지 말한다.
  - B트랙: 기능 구현, 테스트, 문서에 AI를 어떻게 썼는지 말한다.
- **5. 사람 검증**
  - A트랙: 라이선스 출처와 코드 머지 가능성을 검증한 흔적을 말한다.
  - B트랙: 보안, 동작, 외부 사용자 피드백을 검증한 흔적을 말한다.
- **6. 라이선스·운영**
  - A트랙: 외부 OSS 라이선스 분석과 자체 분석 저장소 적용을 말한다.
  - B트랙: 직접 추가한 LICENSE, CONTRIBUTING, COC, SECURITY 5종을 말한다.
- **7. 로드맵**
  - A트랙: 외부 OSS의 다음 PR 후보와 자체 도구화 계획을 말한다.
  - B트랙: 외부 fork·issue 받기와 기능 확장 계획을 말한다.
- **8. 배운 점**
  - A트랙: 외부 OSS 응답·리뷰 경험에서 배운 점을 말한다.
  - B트랙: 운영자 경험, 라이선스, 코드 리뷰, 문서화에서 배운 점을 말한다.
