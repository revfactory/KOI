# 1회차 — AI 시대의 오픈소스: 왜 지금 다시 중요한가

> **핵심 메시지** 오픈소스는 코드 공개가 아니라 AI 시대의 학습·협업·커리어·제품 확산 방식이다.
>
> **선수 지식**
> - GitHub 계정 보유
> - 한 가지 프로그래밍 언어를 하나 이상 만져본 경험
> - AI 챗봇(ChatGPT, Claude, Gemini 중 하나)을 한 번이라도 써본 경험
>
> **이 회차가 끝나면 학생은**
> - 2025년 오픈소스 생태계의 변화 3가지(AI/에이전트, 신규 개발자 유입, 타입 언어 확산)를 자기 언어로 설명할 수 있다. (LO 1)
> - GitHub Trending에서 인기 프로젝트를 골라 README · Issue · PR · Release · License 5개 축으로 분석할 수 있다. (LO 2)
> - 자신의 관심 분야에서 6주 동안 따라갈 후보 OSS(Open Source Software, 오픈소스 소프트웨어) 프로젝트 3개를 후보군으로 정리할 수 있다. (LO 3)

> **회차 일시** 2026-05-06(수) 19:00~21:00 (강의 90분 + 멘토링 60분)
> **트랙 분기** 본 회차는 트랙 분기 없음(전원 공통). 단 후보 프로젝트 3개를 정리할 때 두 트랙(A: 외부 OSS 기여 / B: 자체 공개 저장소 운영)의 양쪽 후보를 모두 두는 것을 권장.
> **누적 산출물 (오늘 GitHub에 남길 것)**
> 1. 개인 GitHub 프로필 README 초안 (1단락 + 학습 흔적 영역)
> 2. 후보 OSS 프로젝트 3개 정리 자료 (저장소 / 활동성 / 라이선스 / 기여 난이도 후보)
> 3. 팀 결성 + 팀 헌법 1페이지 초안 (팀명 / 역할 / 회의 주기 / 의사결정 / 이탈 처리)
> 4. AI(인공지능)로 후보 1개를 분석한 노트 (AI 사용 로그 양식 첫 노출)
>
> **표기 각주** 본 강의자료의 외래어 표기: 첫 등장 시 영문 + 한글(약어) 병기, 두 번째부터 약어/한글로 통일.

```
  ② 신규 개발자 유입 → 진입장벽 변화
  ③ 타입 언어 확산 → 코드 품질 기대치 변화
```

2025년 Octoverse가 보여주는 변화는 세 가지입니다.

- **AI / 에이전트**: AI가 보조 도구를 넘어 협업 과정에 들어오기 시작했습니다. PR을 설명하고, 변경을 제안하고, 리뷰 초안을 만들 수 있습니다.
- **신규 개발자 유입**: 오픈소스는 더 이상 "시간 많은 고수만의 영역"으로 남아 있지 않습니다. 새로 들어오는 개발자가 빠르게 늘고 있습니다.
- **타입 언어 확산**: TypeScript의 성장은 AI 시대에 명시적인 구조와 검증 가능한 코드가 더 중요해졌다는 신호로 읽을 수 있습니다.

### 과거의 오픈소스와 AI 시대의 오픈소스

- **목적**
  - 과거: 비용 절감 / 자유
  - AI 시대: 학습 + 협업 + 커리어 + 제품 확산
- **참여자**
  - 과거: 시간 많은 고수, 학자, 회사 엔지니어
  - AI 시대: 매일 새로 들어오는 신규 개발자
- **도구**
  - 과거: git + IDE
  - AI 시대: git + IDE + AI 에이전트
- **단위**
  - 과거: 코드
  - AI 시대: 코드 + 데이터 + 모델 weights + 사용 로그
- **검증**
  - 과거: 커뮤니티 리뷰
  - AI 시대: 커뮤니티 리뷰 + AI 자동 리뷰 + 사람 검증

이 변화를 한 줄로 묶으면 오늘의 핵심 메시지가 됩니다. **오픈소스는 코드 공개가 아니라 AI 시대의 학습·협업·커리어·제품 확산 방식이다.**

### 자주 받는 질문

- **"TypeScript가 1위라는 건 JavaScript가 죽었다는 건가요?"**
  아니요. JS와 TS는 거의 한 가족이고, 합산하면 여전히 압도적입니다. 다만 "AI 시대에 타입 명시가 더 중요해졌다"는 신호로 읽어봅니다.
- **"Copilot이 PR 1M을 만졌다는 게 사람보다 많이 했다는 뜻인가요?"**
  아닙니다. 전체 PR의 일부입니다. 중요한 점은 AI가 PR의 주체로 등장했다는 사실입니다.
- **"그래서 우리가 왜 오픈소스를 해야 하나요?"**
  GitHub URL이 채용·연구·인턴 시장에서 공개적으로 검증 가능한 실력의 흔적이 될 수 있기 때문입니다.
- **"Octoverse는 GitHub 자기 PR 아닌가요? 데이터 신뢰할 수 있나요?"**
  GitHub의 1차 자료라는 점은 감안해야 합니다. 동시에 대규모 저장소 데이터를 기반으로 한 흐름 자료라 큰 방향을 읽는 데 유용합니다.
- **"한국 OSS는 약하지 않나요?"**
  NAVER·Kakao·Toss·당근 등 한국 조직도 활발한 GitHub 활동을 합니다. 후보 프로젝트를 고를 때 한국어 문서가 있는 저장소도 함께 봅니다.

## 0:25~0:55 — Trending 5축 분석

### 실습 개요

**소요시간** 30분 (작업 25분 + 갤러리 워크 5분)  
**난이도** ★★ (입문~초급)  
**산출물** 페어당 1슬라이드 또는 1포스트잇 (5축 분석 결과)  
**도구** GitHub Trending, 학생 노트북, 핸드아웃의 5축 분석 양식  
**팀 단위** 2명 1조 페어 (3명일 경우 한 명은 옵저버 역할 후 갤러리 워크에서 보완)

**목표**  
GitHub Trending 또는 추천 풀에서 인기 OSS 프로젝트 1개를 골라 README · Issue · PR · Release · License 5개 축으로 분석하고 한 장에 정리합니다. "별 수가 많다고 좋은 프로젝트는 아니다"는 통찰을 직접 확인합니다.

### 진행

1. **페어 짝짓기 (1분)**
   - 2명 1조로 앉습니다.
   - 한 명은 노트북을 보고, 한 명은 핸드아웃을 작성합니다. 5분 후 역할을 바꿉니다.
2. **Trending 페이지 열기 (1분)**
   - https://github.com/trending
   - 기본은 "All languages, Today" 탭입니다.
   - 필요하면 TypeScript / Python / Rust 또는 This week 탭도 봅니다.
3. **분석 대상 1개 선택 (3분)**
   - 페어 둘 다 분야에 관심이 있는 프로젝트를 우선합니다.
   - README가 영어 또는 한국어로 읽히는 프로젝트를 고릅니다.
   - 별 수 100개 이상이면 분석하기 좋습니다.
   - 선택이 어려우면 아래 추천 풀에서 1개를 골라도 됩니다.
4. **5축 분석 (15분)**
   - README, Issues, Pull Requests, Releases, License를 각각 봅니다.
   - 빈칸이 있으면 "없음" 또는 "못 찾음"이라고 적습니다.
5. **한 슬라이드 / 포스트잇 정리 (5분)**
   - 5축 핵심 발견을 한 장에 압축합니다.
   - 다음 활동에서 AI에게 물어볼 입력으로도 사용할 수 있습니다.
6. **갤러리 워크 (5분)**
   - 4분 동안 근처 페어의 자료를 둘러보고, 내가 분석한 프로젝트와 가장 다른 점 1개를 메모합니다.
   - 한 팀이 30초 동안 인상 깊은 점 한 줄을 공유합니다.
   - 남은 30초에는 다음 활동을 위해 프로젝트 URL과 README 위치를 정리합니다.

### 5축 분석 양식


### AI 도구가 OSS 협업을 바꾸는 방식

AI가 등장하면서 오픈소스 기여의 진입장벽은 낮아졌습니다. 새 저장소를 읽고, README를 요약하고, 이슈를 분류하고, PR 본문 초안을 만드는 일을 더 빨리 시작할 수 있습니다. 다만 AI가 제안한 내용은 그대로 믿지 않고 사람이 검증해야 합니다.

  - OSS 여부: 클로즈드
  - 학생 시작점: 무료 티어

- [ ] 가능하면 PR 경험자가 팀 안에 1명 이상 있음
- [ ] 팀명 정해짐
- [ ] 팀 헌법 4항목 모두 채워짐 (역할 / 회의 / 의사결정 / 이탈)
- [ ] 모든 팀원이 서명/이니셜 적음
- [ ] 한 줄 비전 적힘

### 막혔을 때

- **한 영역에 사람이 너무 몰릴 때**: "AI 응용 / AI 인프라 / AI 평가"처럼 더 작은 주제로 나눕니다.
- **한 팀에 PR 경험자가 없을 때**: 다음 주 초반에 GitHub Flow를 더 천천히 따라갈 준비를 합니다.
- **팀 인원이 1~2명뿐일 때**: 가까운 도메인의 다른 팀과 합병을 논의합니다.
- **팀 헌법을 5분 안에 못 채울 때**: 오늘은 잠정안으로 적고, 다음 주 멘토링에서 확정합니다.

### 시간 남으면

- 팀 헌법을 GitHub Issue 1개로 발행합니다.
- 팀명 로고를 손그림으로 1개 그려 둡니다. 6회차 발표 자료에 활용할 수 있습니다.

## 사후 과제 (강의 종료 후 ~ 5/13 24:00)

1회차 90분은 짧습니다. 다음 주 사이 30~60분을 들여 오늘 만든 흔적을 GitHub에 남깁니다.

### 필수 과제

- [ ] **개인 GitHub 프로필 README 초안 커밋**
  - 본인 GitHub `username/username` 저장소(특수 저장소)에 `README.md`를 만들고 1단락 + 학습 흔적 영역을 추가합니다.
  - 참고: GitHub Docs, "Managing your profile README" — https://docs.github.com/en/account-and-profile/setting-up-and-managing-your-github-profile/customizing-your-profile/managing-your-profile-readme

```markdown
# 안녕하세요, [이름]입니다

**6주 동안 'AI 시대의 오픈소스 실전' 강의에서 흔적을 쌓고 있습니다.**

## 지금 관심 있는 것
- [도메인 1]
- [도메인 2]

- [x] 1주차 — 후보 3개 발견 + 팀 결성
```

- [ ] **후보 OSS 프로젝트 3개 정리 자료를 팀 노션 또는 GitHub Issue로 옮기기**
  - 팀의 임시 워크스페이스에 5축 분석 결과 3개를 한 곳에 모읍니다.
  - 다음 주 트랙 결정 회의의 1차 입력으로 사용합니다.

- [ ] **팀 헌법 1페이지를 디지털화**
  - 팀 노션 또는 임시 GitHub Gist에 팀 헌법을 옮깁니다.
  - 다음 주 팀 저장소가 만들어지면 `docs/team-constitution.md`로 이전합니다.

- [ ] **AI 사용 로그 1행 추가 (총 2행 누적)**
  - 강의 시간에 채운 1행 외, 후보 3개 중 또 다른 1개를 AI에게 분석시켜 1행을 추가합니다.

### 강력 권장 과제

- [ ] **사전 진단 설문 1분 답안 제출**
- [ ] **후보 3개 중 1개의 good-first-issue 라벨 이슈를 1개 이상 읽기** (PR 안 보내도 됨, 읽기만)
- [ ] **2회차 사전 읽기**: GitHub Docs "GitHub flow" — https://docs.github.com/en/get-started/using-github/github-flow (5분)

### 평가 매핑

- **프로필 README 초안**: 발표·문서화 15%의 진입점입니다.
- **후보 3개 정리 자료**: OSS 이해도 20%와 실제 기여 25%의 1차 측정점입니다.
- **팀 헌법 디지털화**: GitHub 협업 20%의 진입점입니다.
- **AI 사용 로그 1행 추가**: AI 활용 능력 20%의 1차 측정점입니다.

핵심 채점 룰은 네 가지입니다.

1. **AI 활용 20%는 "썼는가"가 아니라 "사용하고 검증했는가"로 채점**합니다.
2. **실제 기여 25%는 6주 누적**으로 봅니다. 1회차에서는 후보 3개의 품질만 1차로 봅니다.
3. **발표·문서화 15%는 6회차 발표가 핵심**이지만, 1회차의 프로필 README와 KPT도 누적됩니다.
4. 1회차 단독 활동에서 만점/0점이 갈리지는 않습니다. 다만 6주 결과의 출발선이 됩니다.
