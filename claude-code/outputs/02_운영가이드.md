# 02. 교수법 운영 책 (Pedagogy Runbook) — AI 시대의 오픈소스 실전

이 문서는 강사·멘토·운영팀이 6주 과정을 분 단위로 운영하기 위한 매뉴얼이다. 강사가 강의실에 들어가 그대로 펼쳐놓고 진행할 수 있도록 Run-of-Show, 실습 카탈로그, 멘토링 절차, 평가·관찰 양식, 비상 대응 기준을 한곳에 모았다.

실습 블록 ID(`EX-1A`, `EX-1B`, ... )는 운영 중 회차와 실습을 추적하는 식별자이므로 변경하지 않는다.

- §3 실습 블록 카탈로그 — 모든 실습의 ID·목적·산출물·체크리스트·fallback
- §4 멘토링 표준 절차 — 60분 운영, 팀 진단, 다음 주 약속 양식

## 1. 운영 원칙 5가지 (강사 행동 헌법)

이 6주는 "수업"이 아니라 "매주 GitHub에 흔적이 쌓이는 워크숍"이다. 강사는 다음 5원칙을 모든 회차에서 지킨다. 한 가지라도 깨지면 그 회차는 실패로 본다.

### 원칙 1 — 듣는 시간보다 만드는 시간이 더 길어야 한다 (Make > Listen)

매 회차 90분 강의 중 학생이 손가락 또는 입을 움직이는 시간이 **45분 이상**이어야 한다. 강사가 일방적으로 말하는 시간은 **회차당 누적 35분 이내**로 제한. 도입 훅(5분) + 개념 강의 1블록(15분) + 개념 강의 2블록(15분)이 상한이다. 이 한도가 넘으면 즉시 다음 실습으로 이동.

**측정**: 강사 본인이 회차 후 자기 핸드아웃 표지에 "내가 말한 시간 ___분 / 학생이 만든 시간 ___분"을 기록. 비율이 1:1.3 미만이면 다음 회차에서 강제로 강의 분량을 줄인다.

### 원칙 2 — 졸음을 만드는 단조로움을 깨라 (Rhythm Breaks)

90분 안에 **최소 4번의 모드 전환**이 일어나야 한다. 모드 = (강의 / 실습 / 페어 토론 / 라이브 데모 / 투표 / 화면 공유) 중 하나. 같은 모드가 18분 이상 연속되면 졸음이 시작된다. 모드 전환 시 강사는 반드시 **"자, 이제 ___ 합시다"** 를 명시적으로 선언해 학생의 주의 리셋을 강제한다.

**참여 장치 카탈로그** (회차마다 골고루 분산):
- 라이브 코딩 / GitHub 화면 공유
- 투표 (멘티미터, slido, 또는 손들기)
- 페어 워크 (옆 사람과 2분)
- 갤러리 워크 (다른 팀의 PR 화면을 보고 한 줄 코멘트)
- "1분 강의"(학생이 강사 역할로 1분 설명)
- 핫시트 (한 팀이 자기 저장소를 화면에 올리고 5분 라이브 점검)

### 원칙 3 — 우선순위는 개념 < 실습 < 회고 (When 90 Min Overflows)

90분에 다 들어가지 않으면 잘리는 순서:
1. 가장 먼저 자르는 것: **개념 설명 부분 강의** — 슬라이드 분량 절반으로 압축, 나머지는 핸드아웃·자료 풀에 넘긴다.
2. 그 다음: **2번째 실습**(있다면) — 단축형 또는 멘토링 시간으로 이월.
3. **절대 자르지 말 것**: 도입 훅(5분), 1번째 핵심 실습, 회차 끝 5분 회고(KPT).

**판단 기준**: 강의 시작 60분 시점에 "나머지 30분에 1번 실습 + 정리 + 회고가 다 들어가는가?" 자문. 안 들어가면 즉시 개념 강의를 끊고 실습으로 이동.

### 원칙 4 — A트랙 / B트랙은 같은 교실에서 동시에 굴린다 (Track Parallelism)

A트랙(외부 OSS 기여)과 B트랙(자체 운영)은 회차마다 실습 블록이 갈라진다. 강사는 **같은 시간에 두 종류의 실습이 진행되도록** 다음 3가지 도구를 사용한다:

- **분기 슬라이드 1장**: 실습 시작 시 "지금부터 25분 — A트랙은 화면 왼쪽 / B트랙은 화면 오른쪽" 한 장 슬라이드로 분기. 모든 실습 슬라이드는 좌-A / 우-B 2단 레이아웃으로 준비한다.
- **트랙별 멘토 분담**: 멘토 2명 이상일 때 멘토 1은 A, 멘토 2는 B를 전담. 멘토 1명일 때는 강사가 A/B를 5분 단위로 라운드 로빈.
- **공통 종료 신호**: 실습 종료 5분 전 타이머와 함께 "트랙 무관, 5분 후 합류" 알림 → 합류 후 갤러리 워크 또는 핫시트로 양 트랙의 산출물을 서로 본다.

### 원칙 5 — 매 회차 끝 5분 회고는 비협상 (Retro is Sacred)

90분의 마지막 5분(85:00~90:00)은 **무조건 KPT 회고**. 강의가 밀려도 안 자른다. 회고는 (1) 학생이 한 줄로 자기 학습을 언어화하는 시점이며, (2) 강사가 다음 회차 운영을 조정할 데이터를 얻는 시점이다. 회고 양식은 §7.

이 5분이 사라지면 강의는 "들었지만 남은 게 없는" 상태로 끝난다. 강사가 회고를 자르려는 충동을 느끼면, 차라리 정리 멘트 30초로 줄이고 회고 5분을 지킨다.

## 2. 회차별 Run-of-Show

각 회차는 19:00~21:00 (총 120분 = 강의 90분 + 멘토링 60분). 분 단위 시각은 회차 시작(19:00)을 0:00으로 기준한다. **시각 / 분량 / 모드 / 활동 / 산출/체크 / 강사 멘트** 6열 구조.

도입 훅, 전이 문장, 핵심 메시지(샌드위치), LO 노출, 누적 산출물 확인, AI 미니 활동, 회고가 분 단위로 매핑되어 있다.

### 2.2 회차 2 — GitHub 협업 워크플로우와 오픈소스 운영 기본기 (5/7 목)

### 3.2 실습 상세 — 회차 1

#### EX-1A — 도입 훅: 라이브러리 라이선스 침묵 (5분)
- **목적**: "내가 이미 매일 OSS를 쓰고 있다"는 자각을 인지 충격으로 주입.
- **산출물**: 칠판에 학생들이 적은 패키지 이름 5~10개 + 강사가 측정한 침묵 길이.
- **준비물**: 칠판, 마커.
- **체크리스트**: ☐ 학생 5명 이상 패키지 이름 적기 ☐ "라이선스 설명 가능한 사람?" 질문 후 침묵 5초 이상 측정 ☐ 강의 시작 멘트 ("오늘은 첫 회차이지만 ...")로 자연스럽게 전환.
- **fallback**: 학생이 안 적으면 강사가 직접 axios, requests, react를 적고 "여러분도 이거 쓰셨죠?" 압박 질문.

#### EX-1B — GitHub Trending 5축 분석 (페어 워크, 30분)
- **목적**: "Trending에서 보는 안목" + "README/Issues/PRs/Releases/LICENSE 5축"을 손으로 익힘.
- **산출물**: 조당 1슬라이드 또는 1포스트잇 (5축 분석 결과).
- **준비물**: 학생 노트북 + 인터넷, 강사가 미리 만든 5축 분석 양식.
- **체크리스트**: ☐ 2명 1조 짝짓기 ☐ 25분 작업 + 5분 갤러리 워크 ☐ README/Issues/PRs/Releases/LICENSE 5축 모두 작성 ☐ "별 수가 많다고 좋은 프로젝트는 아니다" 한 번 강조.
- **fallback**: Trending이 비어있거나 한국어 학생들에게 어려운 프로젝트만 보일 경우 강사가 미리 큐레이션한 추천 리스트 5개 제공.

#### EX-1C — AI 후보 분석 (5분)
- **목적**: AI를 1회차부터 쓰게 하며 "AI 사용 로그" 양식을 처음 노출.
- **산출물**: AI 출력 + 사람의 검증 메모 1줄.
- **준비물**: Claude / Copilot / Cursor 중 하나 (학생 본인 계정).
- **체크리스트**: ☐ 학생이 직접 프롬프트 입력 ☐ 출력 결과를 1줄이라도 검증 (사실 확인 / 출처 확인) ☐ 핸드아웃 "AI 사용 로그" 표 빈칸 채움.
- **fallback**: AI 도구 접근 권한이 없는 학생은 옆 사람 화면을 같이 보는 페어 형태로.

#### EX-1D — 팀 결성 + 팀 헌법 1페이지 (10분)
- **목적**: 팀 해체 위험을 줄이기 위해 첫 주에 약속을 명문화.

- **fallback**: 인원 부족하면 강사가 다른 관심 도메인 팀과 합병 권유.

- **산출물**: 팀당 (1) 팀 저장소 (2) 첫 머지 PR 1개 (3) 라벨 붙은 Issue 5개.
- **준비물**: GitHub 계정, 팀 저장소 권한, 강사가 준비한 .gitignore / README 스켈레톤.
- **체크리스트**: ☐ 저장소 생성 (public 또는 internal-but-soon-public) ☐ README 1줄 ☐ 팀원 각자 브랜치 생성 ☐ PR 발행 ☐ 다른 팀원이 리뷰 코멘트 1개 이상 ☐ 머지 ☐ 브랜치 삭제 ☐ Issue 5개 + 라벨.
- **fallback**: 권한 문제 발생 시 GitHub Codespaces 또는 강사 데모 저장소에 fork-PR. 30분 안에 못 끝낸 팀은 멘토링 시간으로 이월 (개념·실습·회고 우선순위에 따라 회고는 보존).

#### EX-2D — AI로 PR 본문 다듬기 (15분)
- **목적**: "AI 사용 로그" 본격 도입. PR 본문이라는 구체 타깃에 AI를 쓰고 사람이 검증.
- **산출물**: AI 출력 + 사람 검증 흔적 + 업데이트된 PR 본문.
- **준비물**: AI 도구 (각자), AI 사용 로그 양식.
- **체크리스트**: ☐ AI에게 PR diff 요약 요청 ☐ 출력 받음 ☐ 사람이 1줄 이상 수정/보완 ☐ PR 본문 업데이트 ☐ AI 로그 표 채움.
- **fallback**: AI가 부정확한 답 주면 그 자체가 학습 — "이게 왜 틀렸는지" 1줄 메모.

#### EX-2E — 트랙 확정 1줄 선언 (5분)
- **목적**: 트랙 결정을 강의 시간 내에 강제. 멘토링에서 최종 사인.
- **산출물**: "우리 팀 ___은 [A/B]트랙. ___을(를) 한다" 1줄.
- **체크리스트**: ☐ 1줄 작성 ☐ 멘토링에서 사인.

### 3.4 실습 상세 — 회차 3

- **A트랙 체크리스트**: ☐ 외부 프로젝트의 5종 세트 분석 ☐ 분석 결과를 자기 팀 분석 저장소에 모방 ☐ 차이점 1줄 메모.

#### EX-3C — AI에게 라이선스 리스크 질문 (12분)

- **체크리스트**: ☐ AI에 자기 라이선스 + "6개월 뒤 리스크 3개" 질문 ☐ 답변 받음 ☐ 사람이 1개 이상 검증/수정 ☐ CONTRIBUTING에 녹임.

## 4. 멘토링 표준 절차 (60분)

이 절차는 매 회차 1:30~2:30 멘토링 슬롯에 그대로 적용된다. 회차별 변형은 §2 Run-of-Show의 마지막 행에 명시.

### 4.1 멘토 1인당 팀 수 권장

- 멘토 1인당 2~3팀 권장.
- 팀 8개라면 멘토 2명이 각 4팀 또는 3+3+2 분배.
- 멘토 1명일 때는 "팀당 5~7분 라운드 로빈 + 비동기 보충"으로 운영.

### 4.2 60분 운영 표준

| 시각 (멘토링 시작 = 0:00) | 분량 | 활동 | 산출/체크 |
|----------------|---|----|--------|
| 0:00~0:10 | 10분 | **팀 헬스 체크 (Health Check)** — 멘토가 모든 팀 1분씩 순회. "지난 한 주 잘 굴러갔나? 빠진 사람 있나? 분쟁 있나?" | 팀별 헬스 신호등 (녹/황/적) |

- ☐ 팀원 전원이 첫 commit 이상 했는가?
- ☐ AI 사용 로그가 채워지고 있는가?
- ☐ 다음 회차 도입 훅에 답할 준비가 되어 있는가?

### 4.5 멘토링 부족 시 비동기 대안

멘토 1명·팀 8개 같은 극한 상황에서는 60분 안에 모든 팀을 깊이 보기 어렵다. 다음 비동기 장치로 보완:

- **GitHub Discussions 카테고리 4개**: `general` / `track-A` / `track-B` / `ai-help`. 학생이 막히면 여기에 글을 올리고, 멘토가 24시간 내 답변.
- **단톡방 운영 룰**: 단톡방은 "공지 + 시급한 차단" 전용. 깊은 토론은 Discussions로.
- **회차 종료 후 30분 보충 슬롯**: 21:00~21:30 강사·멘토 1명 잔류, 원하는 학생 1:1.
- **멘토링 라운드 로빈**: 멘토 1명·8팀이면 회차마다 4팀씩 깊이 본다 — 1·3·5회차에 A팀 그룹, 2·4·6회차에 B팀 그룹. 단 헬스 체크 10분은 매 회차 모든 팀 대상.

## 5. AI 활용 매트릭스

AI 활용은 4~5회차에 몰아 넣지 않고 1회차부터 매주 분산한다. 각 회차에서는 아래 표의 AI 활동, 검증 기준, 누적 효과를 반드시 확인한다.

### 5.1 회차별 AI 활용 분산표

| 회차 | AI 활동 ID | 활동명 | 분량 | AI 도구 | 사람이 검증할 것 | 누적 효과 |
|----|---------|----|---|------|-----------|------|
| 1 | EX-1C | AI 후보 분석 (한 줄 요약 + 진입점 3개) | 5분 | Claude/Copilot/Cursor 택1 | 출력의 사실성 / 진입점 1개 이상 검증 | "AI 사용 로그" 양식 첫 노출 |
| 2 | EX-2D | AI로 PR 본문 다듬기 | 15분 | 위와 동일 | PR 본문 1줄 이상 수정 | "AI 로그" 두 번째 누적 |
| 3 | EX-3C | AI에게 라이선스 리스크 질문 | 12분 | 위와 동일 | 리스크 1개 이상 사실 확인 | "AI 로그"가 CONTRIBUTING에 녹음 |
| 4 | EX-4B + EX-4C | 본격 PR + README/changelog 자동 보강 | 30분 + 10분 | Claude Code / Cursor / Copilot Workspace | PR 모든 라인 + 라이선스 출처 | AI 사용 로그 본격화 |
| 5 | EX-5B | 멀티 에이전트로 작은 기능 끝까지 | 30분 | Claude Code subagents / Cursor agents | 역할별 프롬프트 로그 + 안전 장치 4종 | 5역할 프롬프트 로그 |
| 6 | EX-6D | 다음 기여 액션 플랜에 "AI 도구 1개" 명시 | 12분 | 학생이 30일간 쓸 도구 1개 선택 | 검증 방법 1줄 명시 | AI 활용 회고 문서 완성 |

### 5.2 누적 분포 그래프

```
1회차 ▓                    5분 (씨앗)
2회차 ▓▓▓                  15분
3회차 ▓▓▓                  12분
4회차 ▓▓▓▓▓▓▓▓             40분 (본격화)
5회차 ▓▓▓▓▓▓               30분 (심화)
6회차 ▓▓▓                  12분 (회고+미래)
```

**핵심**: 4·5회차에 정점은 맞되, 1~3회차에서 이미 누적 32분 (5+15+12)으로 학생이 AI 활용에 적응되어 있다. 4회차 첫 노출이 아니다.

### 5.3 AI 사용 로그 양식 (4회차부터 PR 템플릿 강제 필드)

```markdown
## AI 사용 로그
| 단계 | AI 도구 | 프롬프트 (요약) | AI 출력 (요약) | 사람이 검증한 것 | 결과 |
|----|------|----------|----------|----------|----|
| 분석 | Claude | ... | ... | ... | 채택/수정/기각 |
| 구현 | ... | ... | ... | ... | ... |
```

이 표가 비어 있거나 "사람이 검증한 것" 칸이 모두 비어 있으면 평가 시 AI 활용 능력 20% 만점 중 0~5점.

## 8. A/B 트랙 동시 진행 절차 (같은 교실, 두 트랙)

분기 매트릭스를 시간 단위로 운영하는 절차.

### 8.1 분기 발생 시각 (회차별)

| 회차 | 분기 시각 | 분기 실습 ID | 합류 시각 | 합류 활동 |
|----|------|---------|------|------|
| 1 | 분기 없음 (전원 공통) | — | — | — |
| 2 | 1:20~1:25 (트랙 확정 1줄 선언만) | EX-2E | 즉시 | 트랙 사인 후 일상 |
| 3 | 0:30~0:55 | EX-3B | 0:55 | 갤러리 워크 5분 (양 트랙 산출물 비교) |
| 4 | 0:30~1:00 | EX-4B | 1:00 | 양 트랙 PR 1개씩 핫시트로 5분 비교 |
| 5 | 0:30~1:00 | EX-5B | 1:00 | 양 트랙 멀티 에이전트 결과 갤러리 워크 |
| 6 | 분기 없음 (발표 통합) | — | — | — |

### 8.2 동시 진행 운영 도구

#### 도구 1 — 분기 슬라이드 1장
모든 분기 실습은 단 1장의 슬라이드로 분기. 좌-A 트랙 / 우-B 트랙 2단 레이아웃을 슬라이드 마스터에 반영해 일관성을 유지한다.

```
+-----------------------------+-----------------------------+
| A트랙 — 외부 OSS 기여       | B트랙 — 자체 운영           |
| (예: EX-3B-A)               | (예: EX-3B-B)               |
| ☐ 체크리스트 1            | ☐ 체크리스트 1            |
| ☐ ...                       | ☐ ...                       |
| 산출물: ...                  | 산출물: ...                  |
+-----------------------------+-----------------------------+
              25분 작업 + 5분 합류 (갤러리 워크)
```

#### 도구 2 — 트랙별 멘토 분담
- 멘토 2명 이상: 멘토 1 = A 전담 / 멘토 2 = B 전담. 회차 시작 시 어느 멘토가 어느 트랙인지 명시.
- 멘토 1명: 5분 단위 라운드 로빈. 강사가 보조 멘토 역할.
- 강사는 항상 양 트랙 모두 인지 — "지금 A는 어디까지, B는 어디까지" 분기 실습 중 1번 이상 큰 소리로 안내.

#### 도구 3 — 합류 신호와 갤러리 워크
실습 종료 5분 전 타이머 알림 + "트랙 무관, 5분 후 합류" 멘트. 합류 후 갤러리 워크:
- A 팀 1개 + B 팀 1개 짝지어 서로의 화면을 1분씩 보고 1줄 코멘트.
- 큰 차이점 1개 + 공통점 1개 도출.
- 이 갤러리 워크가 "A·B 트랙은 다르지만 평가 기준은 같다"는 메시지를 강화.

### 8.3 트랙 전환 절차 (A → B 또는 A+B 하이브리드)

A트랙 외부 응답 지연 시 4·5회차에 트랙 전환 가능.

**절차**:
1. 4회차 멘토링에서 멘토가 "외부 응답 SLA 점검" 후 위험 판단.
2. 위험 시 팀에게 3가지 옵션 제시:
   - (a) 두 번째 작은 PR을 같은 또는 다른 OSS에 동시 시도.
   - (b) A→B 완전 전환 (자체 mini-tool 신규 생성).
   - (c) A+B 하이브리드 (자체 mini-tool로 6회차 발표 보강 + A는 그대로 유지).
3. 팀 의사결정 → 5회차 시작 전까지 새 산출물 계획 수립.
4. 평가는 트랙 무관 동일 기준 — 6회차 발표 8개 항목 충족하면 OK.

## 9. 비상 대응 매뉴얼

### 9.1 90분 초과 시 우선순위 (§1 원칙 3 재확인)

자르는 순서:
1. 개념 강의 부분 (슬라이드 절반 압축 → 핸드아웃 이월).
2. 2번째 실습 단축 또는 멘토링 이월.
3. **자르지 말 것**: 도입 훅 (5분), 1번째 핵심 실습, 회고 5분.

판단 기준: 60분 시점에 "30분에 1번 실습 + 정리 + 회고가 다 들어가는가?" 자문. 안 들어가면 즉시 개념 강의 컷.

### 9.2 멘토링 60분 부족 시 (§4.5 재확인)

- GitHub Discussions 4개 카테고리로 비동기 보완.
- 단톡방은 시급한 차단만.
- 회차 종료 후 30분 보충 슬롯 (21:00~21:30) 잔류.
- 멘토 라운드 로빈 — 회차마다 깊이 보는 팀 절반씩.

### 9.3 라이브 데모/실습 환경 사고

- **인터넷 끊김**: 사전 녹화 클립 30초 또는 강사 캐시 화면 캡처로 대체.
- **GitHub 권한 문제**: Codespaces 또는 강사 데모 저장소에 fork-PR.
- **AI 도구 접근 불가**: 옆 사람 화면 페어 또는 강사 시연 보조.
- **멀티 에이전트 환경 셋업 실패 (5회차)**: 1명이 5역할 프롬프트 순차 시뮬레이션 fallback.

### 9.4 발표 시간 초과 (6회차)

- 5분 정확히 컷. 종료 30초 전 알림.
- 컷된 부분은 핸드아웃 또는 발표 자료 부록에 보존.
- 8개 항목 누락 시 감점 명시 — 5분 안에 8개를 모두 다루는 것이 발표 능력의 핵심.

### 9.5 보안 사고

- 비밀키 커밋 / GPL 코드 MIT 프로젝트 복붙 등 발생 시:
- 즉시 강사가 회수 절차 시연 (BFG 또는 git filter-repo).
- force-push 금지 룰 명문화.
- 사고 발생 팀의 CONTRIBUTING.md에 재발 방지 룰 추가.
