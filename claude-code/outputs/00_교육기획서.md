## 1. 강의 개요

- **강의명**: AI 시대의 오픈소스 실전: 기여자에서 메인테이너까지
- **부제**: 매주 GitHub에 흔적이 쌓이는 워크숍
- **한 줄 소개**: GitHub Trending을 구경하던 학생이 6주 뒤 자기 GitHub 프로필에 공개 Pull Request(PR) 또는 운영 중인 저장소를 남기는 실전 워크숍.
- **키워드**: 오픈소스, GitHub, AI, 멀티 에이전트, 메인테이너
- **운영 형태**: 강의 + 실습 + 팀 멘토링. 1팀 3~5인으로 운영한다.
- **수강 정원**: 6~10팀, 24~50명 권장. 멘토 1인당 2~3팀을 맡는다.

## 2. 수업 목표

### 2.1 큰 목표

학생이 오픈소스를 "사용하는 사람"에서 끝나지 않고, **6주 후 GitHub에 공개적으로 검증 가능한 흔적**을 남긴다.

### 2.2 세부 목표

1. 오픈소스 생태계와 문화 이해
2. GitHub 기반 협업 워크플로우 습득
3. 라이선스, 거버넌스, 커뮤니티 운영 이해
4. AI를 활용한 기여, 리뷰, 문서화, 테스트 자동화 경험
5. 팀 단위 오픈소스 프로젝트 운영 또는 외부 프로젝트 기여
6. 포트폴리오와 커리어로 연결 가능한 결과물 완성

### 2.3 이 강의가 다른 GitHub 강의와 다른 점

- GitHub 사용법을 넘어 **AI 시대의 협업·기여·운영 방식**까지 함께 다룬다.
- AI 활용을 특정 회차에 몰아넣지 않고 1회차부터 매주 분산 배치한다.
- 라이선스를 법률 지식으로만 다루지 않고, README / CONTRIBUTING / CODE_OF_CONDUCT / SECURITY / Templates를 묶어 **오픈소스 운영의 신뢰 구조**로 가르친다.
- 최종 목표는 발표가 아니라 **공개 흔적**이다. 30일 후 다음 기여 약속까지 운영의 일부로 본다.

### 4.3 매 회차 학생이 GitHub에 남기는 흔적

6주 흐름은 **탐색 → 협업 기반 → 운영 신뢰 → 실제 기여 → AI 에이전트 개발 → 공개 발표와 다음 기여**로 이어진다.

```text
1회차: 후보 프로젝트 3개 + 팀 결성 + 프로필 README 초안
        ↓ (선택)
2회차: 팀 저장소 + Issue 목록 + 첫 머지 PR + 트랙 확정 + 역할 분담
        ↓ (정돈)
3회차: LICENSE + CONTRIBUTING + CODE_OF_CONDUCT + Templates + README v2
        ↓ (활용)
4회차: 첫 본격 PR(외부 또는 자체) + AI 사용 로그 + 체크리스트
        ↓ (조직)
5회차: 멀티 에이전트 기반 기능 PR + 테스트/데모 + 발표 초안
        ↓ (공개화)
6회차: 정돈된 프로필 + 공개 발표 자료 + 회고 + 다음 기여 30일 액션 플랜
```

## 5. 트랙 안내

학생 수준이 다양하므로 팀별로 두 가지 트랙 중 하나를 선택한다. 트랙 확정 시점은 **2회차 멘토링**이다.

### 5.1 A트랙 — 외부 오픈소스(Open Source Software, OSS) 기여

기존 오픈소스 프로젝트를 선정하고 6주 동안 실제 기여를 시도한다. AI OSS도 포함할 수 있으며, 기여 목표는 팀 역량에 맞춰 다음 범위에서 정한다.

- **입문**: README 개선, 문서 번역, 예제 추가
- **초급**: 이슈 재현, 버그 리포트, 테스트 케이스 추가
- **중급**: 작은 버그 수정, CLI 옵션 추가, 문서 자동화
- **고급**: 기능 구현, 지속적 통합(Continuous Integration, CI) 개선, 플러그인/확장 개발

### 5.2 B트랙 — 자체 AI 오픈소스 운영형

팀이 직접 작은 오픈소스 프로젝트를 만들고 공개 저장소를 운영한다. 적합한 프로젝트 유형은 다음과 같다.

- **AI 도구**: 작은 Retrieval-Augmented Generation(RAG) 데모, Model Context Protocol(MCP) 서버 예제, 프롬프트 관리 도구
- **개발자 도구**: GitHub Issue 정리 봇, PR 리뷰 보조 스크립트
- **교육 자료**: 특정 기술 입문자를 위한 오픈소스 튜토리얼
- **데이터/문서**: 공개 데이터셋 정리, API 문서화 프로젝트
- **캠퍼스 문제 해결**: 동아리, 연구실, 수업용 자동화 도구

### 5.3 트랙 선택 가이드

- **A트랙 권장**: 팀원 절반 이상이 GitHub PR 경험을 갖고 있거나, 특정 오픈소스 프로젝트의 응답 속도와 기여 가이드를 미리 확인한 팀.
- **B트랙 권장**: 직접 만들고 싶은 도구 아이디어가 명확하거나, 팀원 다수가 첫 오픈소스 경험이고 시간 제약이 큰 팀.
- **전환 가능성**: 외부 프로젝트 응답이 늦어지거나 시간이 부족하면 A트랙에서 B트랙으로 전환하거나 A+B 하이브리드로 조정한다.

> **합류 원칙**: 두 트랙은 **핵심 메시지, 학습 목표, 평가 기준**을 통합한다. 회차마다 활동만 갈라지고, 6회차 발표는 동일한 8개 항목으로 진행해 평가 형평성을 유지한다.

## 7. 평가 기준

### 7.1 평가 항목 비율

| 항목 | 비중 | 측정 기준 |
|----|---|----|
| 오픈소스 이해도 | **20%** | 라이선스, 기여 문화, 운영 구조 이해 |
| GitHub 협업 | **20%** | Issue, PR, Review, Branch 사용 흔적 |
| 실제 기여/운영 | **25%** | 외부 PR 또는 자체 프로젝트 운영 결과 |
| AI 활용 능력 | **20%** | AI를 적절히 활용하고 사람이 검증했는가 |
| 발표/문서화 | **15%** | README, 발표, 회고, 커리어 연결성 |

### 7.2 채점 원칙

- **AI 활용은 "썼다"가 아니라 "쓰고 검증했다"** — PR 본문에 "AI 사용 로그 + 사람 검증" 양식을 강제 필드로 둔다.
- **외부 PR 머지 여부는 가산점, 필수 아님** — 평가의 본체는 PR을 보낸 사실과 검증 과정이다. 외부 메인테이너 응답 지연 위험은 평가에서 분리한다.
- **두 트랙은 동일 채점 기준** — A트랙은 외부 PR 1개 이상, B트랙은 공개 저장소 1개와 외부 반응을 만들기 위한 시도를 기준으로 본다.

상세 루브릭은 `02_운영가이드.md` §6과 `회차별/6회차_프로젝트발표커리어.md`의 발표 평가 루브릭을 참조한다.

## 9. 운영팀과 역할

- **강사(Lead) 1명**: 90분 강의를 진행하고 핵심 개념, 데모, 실습 안내, 슬라이드를 책임진다.
- **멘토 2~3명**: 팀별 진단과 코칭, 다음 주 약속 확인을 맡는다. 6~10팀 기준으로 멘토 1인당 2~3팀이 적정하다.
- **TA(조교) 1~2명**: 실습 트러블슈팅, 슬랙/디스코드 응답, GitHub 운영 보조를 맡는다.

멘토는 A트랙과 B트랙 전담으로 나누는 것을 권장한다. 멘토 1명만 배정되는 경우 멘토링 60분을 5분 라운드 로빈으로 운영한다. 상세는 `02_운영가이드.md` §4 멘토링 표준 절차를 참조한다.

## 10. 위험과 완충 장치

- **학생 실력 격차**: 1회차 진단 설문, 팀 편성 시 균등화, 매 회차 필수/심화 과제 분리로 완충한다.
- **A트랙 외부 오픈소스 응답 지연**: 응답이 빠른 오픈소스 추천 풀, 두 번째 PR 동시 시도, 외부 머지 여부 가산점 처리로 대응한다.
- **팀 해체 또는 인원 이탈**: 1회차 팀 헌법 1페이지, 멘토링 첫 10분 헬스 체크, 솔로팀 허용으로 관리한다.
- **AI 의존 과다 또는 검증 부재**: PR 템플릿에 AI 사용 로그와 사람 검증 필드를 넣고 5회차에서 안전 장치를 강조한다.
- **라이선스/보안 사고**: 3회차 라이선스 호환성 점검, secret scanning, 사고 회수 절차로 대응한다.
- **6회차 발표 시간 초과**: 5회차 멘토링에서 5분 발표 리허설을 측정하고, 6회차에는 타이머와 사전 자료 제출을 적용한다.

상세는 `01_전체커리큘럼.md` §5 위험·완충 장치를 참조한다.
