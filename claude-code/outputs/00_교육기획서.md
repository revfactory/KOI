## 2. 수업 목표

### 2.1 큰 목표 (한 줄)

학생이 오픈소스를 "사용하는 사람"에서 끝나지 않고, **6주 후 GitHub에 공개적으로 검증 가능한 흔적**을 남긴다.

### 2.2 6개 세부 목표 (source.txt 16~25행)

1. 오픈소스 생태계와 문화 이해
2. GitHub 기반 협업 워크플로우 습득
3. 라이선스 · 거버넌스 · 커뮤니티 운영 이해
4. AI를 활용한 기여 · 리뷰 · 문서화 · 테스트 자동화 경험
5. 팀 단위 오픈소스 프로젝트 운영 또는 외부 프로젝트 기여
6. 포트폴리오와 커리어로 연결 가능한 결과물 완성

### 2.3 이 강의가 다른 GitHub 강의와 다른 점

- "GitHub 사용법"이 아니라 **AI 시대의 협업·기여·운영 방식**까지 함께 다룬다 (source.txt 3행).
- AI 활용을 **4·5회차에만 몰지 않고** 1회차부터 매주 분산 배치한다 (source.txt 보강 요청 2번).
- 라이선스를 **법률 지식**으로 끝내지 않고, README/CONTRIBUTING/CODE_OF_CONDUCT/SECURITY/Templates를 묶어 **"오픈소스 운영의 신뢰 구조"**로 가르친다 (source.txt 보강 요청 1번).
- 최종 목표는 **"발표"가 아니라 "공개 흔적"** — 30일 후 다음 기여 약속까지 운영의 일부다 (source.txt 보강 요청 3번).

### 4.2 회차당 2시간 운영 구조

| 구성 | 시간 | 내용 |
|----|----|----|
| 강의 + 실습 | 90분 | 핵심 개념(20~25분) → **짧은 실습 블록 20~30분** → 데모/토론(15~25분) → 두 번째 실습/AI 미니(15~25분) → 미니 회고 5분 |

### 4.3 매 회차 학생이 GitHub에 남기는 흔적 (누적)

```
1회차: 후보 프로젝트 3개 + 팀 결성 + 프로필 README 초안
        ↓ (선택)
2회차: 팀 저장소 + Issue 목록 + 첫 머지 PR + 트랙 확정 + 역할 분담
        ↓ (정돈)
3회차: LICENSE + CONTRIBUTING + CODE_OF_CONDUCT + Templates + README v2
        ↓ (활용)
4회차: 첫 "본격" PR (외부 또는 자체) + AI 사용 로그 + 체크리스트
        ↓ (조직)
5회차: 멀티 에이전트 기반 기능 PR + 테스트/데모 + 발표 초안
        ↓ (공개화)
6회차: 정돈된 프로필 + 공개 발표 자료 + 회고 + 다음 기여 30일 액션 플랜
```

## 5. 트랙 안내

학생 수준이 다양하므로 팀별로 두 가지 트랙 중 하나를 선택한다. 트랙 확정 시점은 **2회차 멘토링**이다.

### 5.1 A트랙 — 외부 오픈소스(AI OSS 포함) 기여

기존 오픈소스 프로젝트를 선정하고, 6주 동안 실제 기여를 시도한다. 기여 목표는 중·고급.

| 난이도 | 기여 유형 |
|----|----|
| 입문 | README 개선, 문서 번역, 예제 추가 |
| 초급 | 이슈 재현, 버그 리포트, 테스트 케이스 추가 |
| 중급 | 작은 버그 수정, CLI 옵션 추가, 문서 자동화 |
| 고급 | 기능 구현, CI 개선, 플러그인/확장 개발 |

### 5.2 B트랙 — 자체 AI 오픈소스 운영형

팀이 직접 작은 오픈소스 프로젝트를 만들고, 공개 저장소를 운영한다.

| 분야 | 예시 |
|----|----|
| AI 도구 | 작은 RAG 데모, MCP 서버 예제, 프롬프트 관리 도구 |
| 개발자 도구 | GitHub Issue 정리 봇, PR 리뷰 보조 스크립트 |
| 교육 자료 | 특정 기술 입문자를 위한 오픈소스 튜토리얼 |
| 데이터/문서 | 공개 데이터셋 정리, API 문서화 프로젝트 |
| 캠퍼스 문제 해결 | 동아리/연구실/수업용 자동화 도구 |

### 5.3 트랙 선택 가이드

| 신호 | 권장 트랙 |
|----|--------|
| 팀원 절반 이상이 GitHub PR 경험 있음 | A트랙 |
| 팀이 명확한 "직접 만들고 싶은 도구" 아이디어를 가짐 | B트랙 |
| 팀원 다수가 첫 OSS 경험 + 시간 제약 큼 | B트랙 (난이도 조정 용이) |
| 팀이 특정 OSS의 응답 속도·기여 가이드를 미리 확인함 | A트랙 |
| 외부 응답이 늦어지거나 시간 부족 | A→B 전환 또는 A+B 하이브리드 |

> **합류 원칙**: 두 트랙은 **핵심 메시지 · 학습 목표 · 평가 기준은 통합**된다. 회차마다 활동만 갈라지고, 6회차 발표는 동일한 8개 항목으로 진행해 평가 형평성을 유지한다.

## 7. 평가 기준

### 7.1 평가 항목 비율 (source.txt 절대 기준)

| 항목 | 비중 | 측정 기준 |
|----|---|----|
| 오픈소스 이해도 | **20%** | 라이선스, 기여 문화, 운영 구조 이해 (1·3회차 활동) |
| GitHub 협업 | **20%** | Issue, PR, Review, Branch 사용 흔적 (2회차 핵심 + 매 회차 누적) |
| 실제 기여/운영 | **25%** | 외부 PR 또는 자체 프로젝트 운영 결과 (4·5회차 핵심) |
| AI 활용 능력 | **20%** | AI를 적절히 활용하고 **사람이 검증**했는가 (전 회차 누적) |
| 발표/문서화 | **15%** | README, 발표, 회고, 커리어 연결성 (3·5·6회차 누적) |

### 7.2 채점 원칙

- **AI 활용은 "썼다"가 아니라 "쓰고 검증했다"** — PR 본문에 "AI 사용 로그 + 사람 검증" 양식이 강제 필드.
- **외부 PR 머지 여부는 가산점, 필수 아님** — "PR을 보낸 사실"이 25% 평가의 본체. 외부 메인테이너 응답 지연 위험을 평가에서 분리.
- **두 트랙은 동일 채점 기준** — A트랙은 외부 PR 1개 이상, B트랙은 공개 저장소 1개 + 외부에서 fork·issue가 1건이라도 일어나도록 노력.

상세 루브릭은 `02_운영가이드.md` §6과 `회차별/6회차_프로젝트발표커리어.md`의 발표 평가 루브릭 참조.
