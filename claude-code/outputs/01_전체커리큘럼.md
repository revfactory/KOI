## 1. 과정 한 줄 요약

> **GitHub Trending을 "구경하던 학생"이 6주 뒤에 자기 GitHub 프로필에 공개된 PR 또는 운영 중인 저장소를 가진 "기여자/메인테이너 후보"로 진화하는, AI 도구를 매주 손에 쥔 채 진행하는 실전 워크숍.**

(부제: 이 과정은 "강의를 듣는 수업"이 아니라 "매주 GitHub에 흔적이 쌓이는 워크숍"이다.)

## 2. 학습 여정 서사 (Narrative Arc)

이 6주는 "왜(Why) → 어떻게(How) → 무엇을(What) → 함께(Together) → 자동화(Scale) → 그 다음(Next)"의 6막 구조로 흐른다. 각 회차는 직전 회차의 산출물을 입력으로 받아 다음 회차의 입력을 출력한다. 매 회차 도입부에는 **"지난 주에 우리가 X를 만들었다, 그래서 오늘 Y를 한다"**는 인과가 드러나야 한다.

### 1막 — 1회차 (5/6 수): "왜 지금 다시 오픈소스인가"

학생은 GitHub을 "라이브러리 다운로드 사이트"로 알고 들어온다. 1회차는 그 인식을 부순다. 2025 Octoverse, GitHub Trending, AI 에이전트 도구가 만든 새로운 협업 풍경을 보여주면서, **"오픈소스는 코드 공개가 아니라 AI 시대의 학습·협업·커리어·제품 확산 방식"**이라는 명제를 학생이 자기 입으로 말하게 만든다. 이 회차의 끝에서 학생은 GitHub 계정을 점검하고, 팀을 구성하고, 6주 동안 따라갈 후보 프로젝트 3개를 손에 쥔다. **첫 흔적**: GitHub 프로필 README 초안과 후보 프로젝트 3개를 정리한 팀 노션/이슈.

### 2막 — 2회차 (5/7 목): "오픈소스의 언어를 익힌다"

1회차에서 "왜 중요한지"를 알았으니, 이제 "어떻게 참여하는지"를 배운다. 핵심 메시지는 **"오픈소스의 기본 언어는 코드가 아니라 Issue, Branch, Commit, Pull Request, Review"**이다. 학생은 팀 저장소를 만들고, 첫 Issue를 발행하고, 첫 PR을 자기들끼리 주고받는다. 동시에 트랙(A/B)을 확정한다. **이 회차가 끝나면 모든 팀의 GitHub에는 살아있는 저장소 + 라벨 붙은 Issue 목록 + 최소 1개의 머지된 PR이 남는다.** 도입부에서는 "1회차에서 우리가 후보 3개를 골랐다 → 오늘 그중 하나에 깃발을 꽂는다"라는 인과를 명시한다.

### 3막 — 3회차 (5/14 목): "신뢰의 골격을 세운다"

2회차에서 저장소를 "만들었다면", 3회차는 그 저장소를 "신뢰할 수 있게 만든다." 본 회차는 라이선스(MIT/Apache-2.0/GPL/AGPL/MPL) + README + CONTRIBUTING + CODE_OF_CONDUCT + SECURITY + Issue/PR Template을 **"오픈소스 운영의 신뢰 구조"**라는 단일 우산 아래 묶어 가르친다. AI 시대의 라이선스 이슈(모델 weights, 학습 데이터, OSI의 Open Source AI Definition)도 함께 다룬다. **첫 흔적**: 모든 팀 저장소에 LICENSE/CONTRIBUTING.md/Issue Template/PR Template이 추가되고, README가 표준 골격으로 재정렬된다.

### 4막 — 4회차 (5/21 목): "AI를 기여 보조 시스템으로 쓴다"

3회차에서 운영 신뢰 구조를 갖췄다면, 4회차는 그 구조 위에서 AI를 본격 활용한다. 핵심 메시지는 **"AI는 코드를 대신 써주는 도구가 아니라, 오픈소스 기여의 진입장벽을 낮춰주는 기여 보조 시스템"**이다. 낯선 저장소 분석, 이슈 재현, PR 본문 작성, 문서화, 라벨링, changelog 자동화를 Copilot/Claude Code/Cursor로 직접 해본다. RAG·MCP·A2A의 개념도 여기서 처음 도입된다(상세는 5회차). **AI 활용은 1회차부터 매주 누적되어야 하며, 4회차는 그것을 명시화·체계화하는 회차**이지 첫 노출이 아니다. **첫 흔적**: 외부 프로젝트 기여 PR 초안 또는 자체 프로젝트 기능 PR + AI 사용 로그(어떤 작업에 AI를 썼고 사람이 무엇을 검증했는지).

### 5막 — 5회차 (5/28 목): "AI 멀티 에이전트로 개발 흐름을 설계한다"

4회차에서 AI를 "도구로 쓰는" 법을 익혔다면, 5회차는 AI를 "협업자로 조직하는" 법을 다룬다. 핵심 메시지는 **"AI를 잘 쓰는 개발자는 챗봇에게 질문하는 사람이 아니라, 역할이 나뉜 AI 작업 흐름을 설계하는 사람"**이다. Planner / Coder / Reviewer / Tester / Documenter로 역할을 분리하고, MCP는 도구 연결, A2A는 에이전트 간 협업이라는 구도로 학생들이 작은 기능 하나를 멀티 에이전트로 끝까지 밀어본다. hallucination · 보안 · 라이선스 · 의존성 검증의 안전 장치도 같이 가르친다. **첫 흔적**: 멀티 에이전트로 만든 기능 PR 1개 + 테스트 또는 데모 + AI 멀티 에이전트 사용 기록 + 최종 발표 초안.

### 6막 — 6회차 (6/4 목): "공개적으로 검증 가능한 실력의 흔적"

마지막 회차는 발표가 아니라 **"공개 흔적의 정리와 다음 기여로 이어가기"**다. 최종 목표는 발표 자체가 아니라 GitHub에 남는 공개 흔적이다. 6주 동안 누적된 Issue/PR/Commit을 GitHub 프로필에 정리하고, README와 pinned repo를 다듬고, 5분 발표 + 5분 Q&A로 동료 팀 앞에서 자기 여정을 말한다. 그리고 마지막 30분은 "다음 기여 1건"을 정해서 강의 종료 후에도 흔적이 이어지도록 만든다. **마지막 흔적**: 정돈된 GitHub 프로필 + 발표 자료(공개) + AI 활용 회고 문서 + 개인 액션 플랜.

### 6주 누적 산출물 흐름 (한눈에)

```
1회차: 후보 프로젝트 3개 + 팀 결성 + 프로필 README 초안
        ↓ (선택)
2회차: 팀 저장소 + Issue 목록 + 첫 머지 PR + 트랙 확정 + 역할 분담
        ↓ (정돈)
3회차: LICENSE + CONTRIBUTING + CODE_OF_CONDUCT + Templates + README v2
        ↓ (활용)
4회차: 첫 "본격" PR (외부 또는 자체) + AI 사용 로그 + 체크리스트
        ↓ (조직)
5회차: 멀티 에이전트 기반 기능 PR + 테스트/데모 + 발표 초안
        ↓ (공개화)
6회차: 정돈된 프로필 + 공개 발표 자료 + 회고 + 다음 기여 액션 플랜
```

## 4. A/B 트랙 분기·합류 가이드

학생 수준은 다양하므로 A트랙(외부 OSS 기여)과 B트랙(자체 공개 저장소 운영)을 함께 운영한다. 두 트랙은 회차마다 활동이 갈라지지만, **핵심 메시지·LO·평가 기준은 통합**된다. 매 회차는 "공통 본문 → A트랙 박스 → B트랙 박스"의 3블록 구조를 따른다.

### 4.1 회차별 분기 매트릭스

| 회차 | A트랙 (외부 OSS 기여) | B트랙 (자체 공개 저장소 운영) | 합류 지점 |
|----|------------------|--------------------|---------|
| 1회차 | 후보 외부 프로젝트 3개 발굴(GitHub Trending + 관심 도메인) | 자체 프로젝트 아이디어 3개(AI 도구/개발자 도구/교육 자료/데이터/캠퍼스 문제) | 둘 다 "후보 3개"로 형식 통일 → 비교 가능 |
| 2회차 | 후보 1개 확정 + 그 프로젝트의 good-first-issue 5개 분석 + 팀 저장소(분석/실험용) 생성 | 자체 프로젝트 1개 확정 + 팀 저장소(메인) 생성 + 첫 Issue 5개 발행 | 둘 다 "팀 저장소 + Issue 목록 + 첫 PR" 보유 |
| 3회차 | 외부 프로젝트의 LICENSE/CONTRIBUTING 분석 + 팀 분석 저장소에도 동일 구조 적용 | 팀 저장소에 LICENSE/CONTRIBUTING/COC/SECURITY/Templates 직접 추가 | 둘 다 "신뢰 골격 갖춘 저장소" 결과 |
| 4회차 | 외부 프로젝트에 PR 초안 작성(small fix, docs, test) + AI 사용 로그 | 자체 프로젝트 기능 PR(팀 내) + AI 사용 로그 | 둘 다 "본격 PR 1개 + AI 로그" |
| 5회차 | 외부 프로젝트의 더 큰 PR(또는 두 번째 PR) 시도 with 멀티 에이전트 / 또는 팀 분석 저장소에서 멀티 에이전트 데모 | 자체 프로젝트 기능을 멀티 에이전트로 구현한 PR | 둘 다 "멀티 에이전트 사용 기록" |
| 6회차 | 외부 PR의 머지 여부와 관계없이 6주 여정 발표 + 다음 기여 액션 | 자체 저장소를 공개 전환 + 6주 여정 발표 + 다음 기여 액션 | 둘 다 "공개 흔적 + 발표 + 다음 액션" |

### 4.2 트랙 선택 가이드 (2회차 멘토링에서 결정)

- **A트랙 권장**: 팀원 절반 이상이 GitHub PR 경험이 있거나, 특정 OSS의 응답 속도·기여 가이드를 미리 확인한 팀.
- **B트랙 권장**: 직접 만들고 싶은 도구 아이디어가 명확하거나, 팀원 다수가 첫 OSS 경험이고 시간 제약이 큰 팀.
- **전환 권장**: 시간이 부족하거나 외부 응답이 늦어지면 A→B 전환 또는 A+B 하이브리드(자체 mini-tool 보조)를 허용한다.

### 4.3 합류 원칙

- **공통 산출물 형식**: 6회차 발표는 두 트랙이 같은 항목(우리 문제 / 프로젝트 / Issue·PR·Commit / AI 활용 / 검증 / 라이선스 / 로드맵 / 배운 점) 8개로 발표한다. 이로써 평가 형평성이 유지된다.
- **공통 평가 기준**: 20/20/25/20/15는 트랙과 무관. A트랙은 "외부 PR 1개 이상", B트랙은 "공개 저장소 1개 + 외부 누군가가 fork·issue를 남기게 하는 노력"으로 "실제 기여/운영" 25%를 측정한다.
- **공통 회고 포맷**: AI 활용 회고 문서 형식을 통일(어떤 작업 / 어떤 도구 / 어떤 프롬프트 / 사람이 검증한 부분 / 다시 하면 어떻게 할지).

## 5. 위험·완충 장치

운영 중 발생 가능한 6대 위험과 완충 장치. 강사는 회차별 운영 노트에 해당 회차에서 가장 가능성 높은 위험과 완충책을 명시한다.

### 5.1 위험 — 학생이 따라오지 못함 (실력 격차)

- **신호**: 2회차에서 GitHub Flow 실습이 30분 안에 안 끝나는 팀이 30% 이상.
- **완충**: (1) 1회차 종료 시점 사전 진단 설문(GitHub 경험, Git CLI 경험, AI 도구 경험)으로 팀을 섞을 때 실력 분포를 강제 균등화. (2) 매 회차 강의 90분 안에 "필수 트랙"과 "심화 트랙"을 핸드아웃에 분리 표기. (3) 강의 종료 후 30분 보충 멘토링 슬롯 운영.

### 5.2 위험 — A트랙 외부 OSS 응답 지연

- **신호**: 4회차에 PR 초안을 올렸는데 5회차까지 메인테이너 응답이 없음.
- **완충**: (1) 2회차 트랙 확정 시 "응답 SLA가 빠른 OSS"를 추천 풀로 제공. (2) 4회차 시점에 응답 없으면 "두 번째 작은 PR"을 같은 또는 다른 OSS에 동시 시도. (3) 외부 머지 여부는 평가에서 가산점이지 필수 아님 — "PR을 보낸 사실"이 25% 평가의 본체. (4) 응답 지연이 명백하면 5~6회차에 B트랙으로 보조 산출물 추가(자체 mini-tool로 6회차 발표 보강).

### 5.3 위험 — 팀 해체 / 인원 이탈

- **신호**: 3회차 이후 팀원 1~2명 이탈, 또는 팀 내 분쟁.
- **완충**: (1) 1회차 팀 결성 시 "팀 헌법 1페이지"(역할, 회의 주기, 의사결정 방식, 이탈 시 처리)를 작성하게 함. (2) 인원 이탈 시 다른 팀과 합병 또는 솔로-팀 허용(평가는 1인 기준 산출물로 조정). (3) 멘토가 매 회차 멘토링 60분 중 첫 10분에 "팀 헬스 체크"를 한다.

### 5.4 위험 — AI 의존 과다 / 검증 부재

- **신호**: 4·5회차 PR 본문이 AI 출력 그대로, 사람의 검증 흔적 없음.
- **완충**: (1) 4회차부터 모든 PR에 "AI 사용 로그 + 사람이 검증한 부분"을 PR 템플릿에 강제 필드로 추가. (2) 5회차 안전 장치 강조(라이선스 출처 의심 코드 머지 금지). (3) 평가 기준 "AI 활용 능력 20%"의 채점은 "사용했는가"가 아니라 "사용하고 검증했는가"로 명시.

### 5.5 위험 — 라이선스 / 보안 사고

- **신호**: 학생이 GPL 코드를 MIT 프로젝트에 복붙, 또는 비밀키를 커밋.
- **완충**: (1) 3회차에 라이선스 호환성 매트릭스를 명시 교육. (2) 모든 팀 저장소에 secret scanning 활성화 + .gitignore 표준 템플릿 제공. (3) 사고 발생 시 즉시 회수 절차(force-push 금지, BFG 또는 git filter-repo 안내)를 운영 매뉴얼에 명시.

### 5.6 위험 — 6회차 발표 시간 초과 / 형평성 문제

- **신호**: 첫 팀이 8분 발표 → 뒤로 갈수록 시간 압박.
- **완충**: (1) 5회차 멘토링에 "발표 초안 5분 시간 측정"을 필수화. (2) 6회차 발표는 타이머 엄격 적용 + 5분 + 5분 Q&A 룰. (3) 발표 자료는 사전 제출 + 공개 저장소 업로드(평가 가능성 보장).

## 6. 강사용 회차 설계 기준

각 회차 강의자료와 핸드아웃은 다음 7개 규칙을 따른다.

1. **첫 줄은 직전 회차 전이 문장**으로 시작. (1회차는 예외)
2. **마지막 줄은 다음 회차 전이 문장**으로 끝. (6회차는 "30일 후 다음 기여 약속"으로 갈음)
3. **핵심 메시지는 회차 도입 슬라이드와 마무리 슬라이드 양쪽에 동일 문장으로 등장**해야 한다(샌드위치 룰).
4. **LO 3개는 회차 시작 5분 안에 학생에게 명시**. 핸드아웃 첫 페이지 상단에 동일 LO 3개를 박스로 배치.
5. **누적 산출물은 회차 종료 시점에 학생이 GitHub에 무엇을 올렸는지로 표현**. "이해했다"가 아니라 "올렸다/만들었다/추가했다".
6. **AI 활용 미니 활동을 매 회차 1개 이상 포함**. AI를 4·5회차에만 몰지 않는다.
7. **A/B 트랙 분기 박스**를 모든 실습/멘토링 섹션에 두고, "공통 본문 → A 박스 → B 박스" 3블록을 따른다.
