## 1. 회차 간 일관성

### 1.1 6막 서사 매핑

| 회차 | 마스터 플랜 6막 | 회차 핵심 메시지 | 검증 |
|---|---|---|---|
| 1 | Why | 오픈소스는 코드 공개가 아니라 AI 시대의 학습, 협업, 커리어, 제품 확산 방식이다. | ✓ |
| 2 | How | 오픈소스의 기본 언어는 코드가 아니라 Issue, Branch, Commit, Pull Request, Review다. | ✓ |
| 3 | What / Trust | 오픈소스는 공개가 아니라 사용, 수정, 배포, 기여를 허용하는 신뢰 구조다. | ✓ |
| 4 | Together | AI는 코드를 대신 써주는 도구가 아니라 오픈소스 기여의 진입장벽을 낮추는 기여 보조 시스템이다. | ✓ |
| 5 | Scale | AI를 잘 쓰는 개발자는 챗봇에게 질문하는 사람이 아니라 역할이 나뉜 AI 작업 흐름을 설계하는 사람이다. | ✓ |
| 6 | Next | 오픈소스 경험은 수업 결과물이 아니라 커리어에서 공개적으로 검증 가능한 실력의 흔적이다. | ✓ |

6개 회차는 Why -> How -> Trust -> AI 도구 -> 멀티 에이전트 -> 공개 흔적의 순서로 진행된다. 각 회차의 핵심 메시지는 마스터 플랜의 6막 서사와 충돌하지 않는다.

### 1.2 누적 산출물 사슬

산출물 사슬은 끊기지 않는다. 1회차의 후보 OSS 3개, 팀 헌법, 프로필 README, AI 로그 1행은 2회차의 협업 언어 실습으로 이어진다. 2회차의 팀 저장소, Issue 5개, 첫 머지 PR, 트랙 사인은 3회차의 신뢰 구조 실습 입력이 된다. 3회차의 LICENSE, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, Templates, README v2는 4회차의 AI 기여 실습 기반이 된다. 4회차의 본격 PR, AI 사용 로그, RAG/MCP/A2A 한 줄 정의 노트는 5회차의 멀티 에이전트 실습으로 이어진다. 5회차의 멀티 에이전트 PR, 발표 초안 5장, 5분 측정 결과는 6회차의 발표와 공개 흔적 정리에 사용된다.

### 1.3 핵심 메시지 샌드위치

| 회차 | 도입·마무리 핵심 메시지 동일성 |
|---|---|
| 1 | ✓ 도입과 마무리에서 "오픈소스는 코드 공개가 아니라 AI 시대의 학습·협업·커리어·제품 확산 방식"을 반복 |
| 2 | ✓ 도입과 마무리에서 Issue, Branch, Commit, Pull Request, Review를 오픈소스의 기본 언어로 반복 |
| 3 | ✓ 도입과 마무리에서 라이선스와 운영 문서를 신뢰 구조로 반복 |
| 4 | ✓ 도입과 마무리에서 AI를 기여 보조 시스템으로 반복 |

핵심 메시지는 전 회차에서 도입과 마무리에 같은 의미로 재등장한다. v2.0에서 회차별 강의자료 라인이 바뀌었으므로 라인 번호 대신 섹션 단위 검증 결과만 남겼다.

### 1.4 전이 문장과 LO 노출

직전 회차 전이 문장은 1회차 예외를 제외하고 모두 유지된다. 2회차는 "왜 오픈소스인가"에서 "어떻게 참여하는가"로, 3회차는 살아있는 저장소에서 외부 신뢰로, 4회차는 신뢰 구조 위의 AI 도구화로, 5회차는 한 명의 AI에서 다섯 역할의 흐름으로, 6회차는 5주 산출물에서 공개 흔적과 다음 기여로 넘어간다.

다음 회차 전이 문장도 1~5회차 말미에 배치되어 회차 간 목적이 자연스럽게 이어진다. 6회차는 "30일 안에 다음 기여 1건"을 졸업 조건처럼 제시해 수업 이후 행동으로 닫는다.

LO 3개는 모든 회차에서 초반부에 노출된다. 1·6회차는 도입 직후, 2·3·4·5회차는 훅과 전이 뒤에 제시되어 수업 목표를 놓치지 않게 구성됐다.

### 1.5 외래어와 약어 흐름

Pull Request / 풀 리퀘스트 / PR, Issue / 이슈, Branch / 브랜치, Commit / 커밋, Maintainer / 메인테이너, Repository / 저장소, Fork / 포크, Review / 리뷰, LICENSE / 라이선스, README, RAG, MCP, A2A의 표기는 회차 간 흔들림이 없다. 첫 등장에서는 영문과 한글 또는 약어를 병기하고, 이후에는 맥락에 맞게 한글 또는 약어 중심으로 사용한다.

## 2. 인용 검증 결과

### 2.1 결번 [8] 미사용

source.txt 결번 [8]은 6개 회차 강의자료에서 0회 사용됐다. 자료 풀에서도 [8]이 결번임을 분리해 두었고, 회차별 강의자료는 결번을 제외한 출처만 사용한다. 결번 인용으로 인한 P0 차단 요소는 없다.

### 2.2 대표 출처 접근성

대표 출처 5개를 확인한 결과 Octoverse 2025, Open Source Definition, RAG 원논문, MCP Introduction, A2A Home 모두 응답 200을 반환했고 발행처도 자료 풀 표기와 일치했다. WebFetch 기준 5/5 접근 가능으로, 본 검수 범위에서는 깨진 핵심 링크가 발견되지 않았다.

### 2.3 자료 풀 등재 여부

6개 회차의 외부 인용은 자료 풀의 마스터 표 또는 회차별 자료 풀에 등재된 항목만 사용한다. 1회차는 Octoverse, GitHub flow, Profile README, AI 도구와 후보 저장소 자료를 사용했고, 2회차는 Pull Request, Issue, GitHub flow, Conventional Commits 자료를 사용했다. 3회차는 OSD, Open Source AI Definition, Choose a License, COC, SECURITY, FOSSA, ScanCode 계열 자료를 사용했다. 4·5회차는 Claude Code, RAG, MCP, A2A, 멀티 에이전트 및 보안 자료를 사용했고, 6회차는 GitHub 프로필, README, 한국 OSS 후보, 발표 도구 자료를 사용했다. 임의 출처 추가는 확인되지 않았다.

## 3. 시간 운영과 실습 동기화

모든 회차의 90분 운영은 정확히 일치한다. 운영표의 Run-of-Show와 회차별 시간 흐름도 같은 순서로 구성되어 강사가 수업 중 참조하기 어렵지 않다.

### 3.2 EX 실습 ID 동기화

운영표 카탈로그 기준 EX-1A~EX-6D의 22개 실습은 회차별 활동 흐름과 모두 대응된다. 1회차 4개, 2회차 5개, 3회차 3개, 4회차 3개, 5회차 3개, 6회차 4개가 각 회차 목적에 맞게 배치됐다. 회차 간 선행·후행 실습을 부르는 항목도 누적 산출물 사슬을 설명하기 위한 정상적인 참조다.

메모: v2.0 학생 핸드아웃 정제 과정에서 회차 강의 교안의 EX ID 표기는 제거됐으므로, 현재는 운영표 카탈로그와 실제 활동 흐름의 대응 관계만 확인하면 된다.

### 3.3 실행 가능성과 fallback

2회차의 GitHub CLI와 Git 명령 흐름은 인증, 저장소 생성, 브랜치 생성, PR 작성, Issue 생성, 충돌 처리까지 실행 가능한 명령으로 구성됐다. 3회차의 `git filter-repo`와 BFG는 보안 사고 회수 도구로 자료 풀에 등재되어 있고, 부록에서 절차를 안내한다.

AI 프롬프트는 1회차 후보 분석, 2회차 PR 본문 정리, 3회차 라이선스 리스크 질문, 4회차 멀티파일 수정과 README 보강, 5회차 멀티 에이전트 역할 분리까지 단계적으로 올라간다. 5회차의 sub-agents 기반 실습은 환경 변동 가능성이 있었으나, v1.1에서 fallback과 사전 점검 항목이 반영됐다.

핵심 실습 22개에는 네트워크, 권한, AI 계정, 시간 부족, 팀 규모, 발표 시간 초과에 대비한 fallback이 마련되어 있다.

## 4. 교수법적 완성도

### 4.1 보강 요청 반영

라이선스는 법률 지식으로 분리하지 않고 LICENSE, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, Templates, README v2를 묶은 신뢰 구조로 다룬다. 3회차 실습은 이 5종 세트를 실제 저장소에 추가하고, 회차 종료 시 GitHub에 공개 흔적으로 남기는 방식이다.

AI 활용은 4·5회차에 몰리지 않는다. 1회차 AI 후보 분석, 2회차 PR 본문 다듬기, 3회차 라이선스 리스크 질문을 거쳐 4회차 AI 기여 도구화, 5회차 멀티 에이전트, 6회차 30일 액션 플랜으로 이어진다.

최종 목표는 발표 자체가 아니라 공개 흔적이다. 6회차는 GitHub 프로필 정리, 발표, AI 활용 회고, 30일 액션 플랜, 재모임 약속을 통해 수업 이후에도 이어질 활동을 남기도록 설계됐다.

### 4.2 KPT와 도입 훅

KPT 회고는 6개 회차 모두 1:25~1:30 구간에 배치된다. 6회차는 5분 미니 KPT 뒤에 30분 통합 회고가 확장되어, 전체 과정의 공개 흔적과 다음 기여 계획을 정리한다.

도입 훅도 전 회차에 있다. 1회차는 라이브러리 라이선스 침묵, 2회차는 인기 OSS Issues 단어 빈도, 3회차는 Llama·DeepSeek·Mistral 함정 투표, 4회차는 낯선 저장소를 AI 없이/AI와 함께 읽기, 5회차는 회사 1인분 활동, 6회차는 30초 채용 시뮬레이션으로 시작한다.

## 5. A/B 트랙 균형

1회차는 전원 공통으로 오픈소스 관점을 열고, 2회차에서 A트랙(외부 OSS 기여)과 B트랙(자체 저장소 운영)을 선언한다. 3·4·5회차는 두 트랙의 명령어, 체크리스트, 산출물 조건을 나란히 다루며, 6회차는 발표와 공개 흔적 정리 형식으로 다시 합류한다.

평가는 트랙과 무관하게 같은 기준을 사용한다. 6회차 발표 항목, AI 활용 회고 형식, 30일 액션 플랜의 When/Where/What/With/How/Promise 필드는 양 트랙이 동일하다.

## 6. 처리 완료된 이슈

v1.0 검수에서 발견된 P1 5건과 P2 8건은 v1.1에서 일괄 반영됐다. v2.0에서는 학생 핸드아웃 정제로 강사용 위치·라인 번호 기반 항목이 더 이상 유효하지 않게 됐다. 따라서 이 리포트에는 미해결 액션 아이템을 남기지 않는다.

## 7. 최종 요약

6주 교육 콘텐츠는 "왜 오픈소스인가"에서 시작해 "어떻게 참여하는가", "무엇을 신뢰할 수 있게 만드는가", "AI로 어떻게 기여 장벽을 낮추는가", "역할 분리형 AI 작업 흐름을 어떻게 설계하는가", "공개 흔적으로 어떻게 이어가는가"로 닫힌다. 시간 운영은 전 회차 90분에 맞고, 인용은 자료 풀 안에서 통제되며, 실습은 누적 산출물 사슬을 만든다.

최종 상태는 v1.1·v2.0 적용 후 운영 가능 상태다.
