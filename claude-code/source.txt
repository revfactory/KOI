> **6주 후 모든 팀이 하나의 공개 저장소를 운영하거나, 외부 오픈소스 프로젝트에 실제 기여 경험을 남긴다.**

GitHub의 2025 Octoverse에서도 AI, 에이전트, 타입 언어가 개발 생태계의 큰 변화를 만들고 있고, GitHub에는 2025년에 1초마다 한 명 이상 신규 개발자가 합류했으며 PR, 커밋, 저장소 활동도 크게 증가했습니다. 즉, 지금 오픈소스를 가르친다면 단순 GitHub 사용법이 아니라 **AI 시대의 협업·기여·운영 방식**까지 함께 다루는 것이 자연스럽습니다. ([The GitHub Blog][1])

## 1회차 — AI 시대의 오픈소스: 왜 지금 다시 중요한가

**핵심 메시지**

오픈소스는 더 이상 “코드를 공개하는 행위”만이 아니라, AI 시대의 학습·협업·커리어·제품 확산 방식입니다.

**강의 내용**

1회차에서는 수업의 전체 방향성을 소개하고, 학생들이 6주 동안 어떤 결과물을 만들어야 하는지 명확히 알려주는 것이 중요합니다.

다룰 내용은 다음과 같습니다.

| 파트            | 내용                                                   |
| ------------- | ---------------------------------------------------- |
| 오리엔테이션        | 6주 과정 목표, 팀 프로젝트 방식, 최종 발표 기준                        |
| 오픈소스의 변화      | 과거의 오픈소스 vs AI 시대의 오픈소스                              |
| GitHub 트렌드 읽기 | GitHub Trending, Octoverse, 인기 프로젝트 구조 분석            |
| AI와 오픈소스      | Copilot, Claude Code, Cursor, 에이전트 도구가 기여 방식을 바꾸는 방식 |
| 프로젝트 리뷰       | 최근 인기 오픈소스 프로젝트의 README, 이슈, PR, 릴리즈 방식 분석           |

GitHub Trending은 “오늘 커뮤니티가 무엇에 흥미를 느끼는지”를 보여주는 페이지라서 1회차 실습에 적합합니다. 단, 별 수가 많다고 반드시 좋은 프로젝트는 아니므로 README 품질, 이슈 관리, 최근 커밋, 기여 가이드, 라이선스를 함께 보게 하는 것이 좋습니다. ([GitHub][3])

## 2회차 — GitHub 협업 워크플로우와 오픈소스 운영 기본기

**핵심 메시지**

오픈소스의 기본 언어는 코드가 아니라 **Issue, Branch, Commit, Pull Request, Review**입니다.

**강의 내용**

| 파트            | 내용                                                       |
| ------------- | -------------------------------------------------------- |
| Git/GitHub 기본 | Repository, Fork, Clone, Branch, Commit, Push            |
| GitHub Flow   | 브랜치 생성 → 변경 → PR → 리뷰 → 병합 → 브랜치 삭제                      |
| Issue 운영      | 버그 리포트, 기능 제안, 태스크 관리, 라벨링                               |
| PR 작성법        | 좋은 PR 제목, 요약, 스크린샷, 관련 이슈 연결                             |
| 리뷰 문화         | 코드 리뷰, 문서 리뷰, 커뮤니케이션 매너                                  |
| 운영 파일         | README, CONTRIBUTING, CODE_OF_CONDUCT, LICENSE, SECURITY |

GitHub Flow에서는 관련 없는 변경마다 별도 브랜치를 만들고, PR에는 변경 요약과 해결한 문제를 적으며, 관련 이슈를 연결하는 것이 권장됩니다. ([GitHub Docs][4]) GitHub Issues는 아이디어, 피드백, 작업, 버그를 추적하는 데 쓰이며 프로젝트 관리와도 연결됩니다. ([GitHub Docs][5])

**멘토링 산출물**

각 팀은 다음을 제출합니다.

| 산출물                                                    |
| ------------------------------------------------------ |
| 팀 GitHub 저장소                                           |
| 프로젝트/기여 방향 1개 확정                                       |
| 첫 번째 Issue 목록                                          |
| 역할 분담: Maintainer, Contributor, Docs, AI/Automation 담당 |

## 3회차 — 오픈소스 라이선스, 거버넌스, 커뮤니티 운영

**핵심 메시지**

오픈소스는 “공개”가 아니라 **사용·수정·배포·기여를 허용하는 규칙과 신뢰 구조**입니다.

**강의 내용**

| 파트         | 내용                                      |
| ---------- | --------------------------------------- |
| 오픈소스의 정의   | 자유롭게 사용, 수정, 공유할 수 있다는 의미               |
| 주요 라이선스    | MIT, Apache-2.0, GPL, AGPL, LGPL, MPL   |
| 라이선스 선택 기준 | 개인 프로젝트, 기업 사용, SaaS, 연구 코드             |
| AI와 라이선스   | 모델, 데이터셋, 학습 코드, weights의 공개 범위         |
| 커뮤니티 운영    | CONTRIBUTING, CODE_OF_CONDUCT, SECURITY |
| 메인테이너 역할   | 이슈 분류, PR 리뷰, 릴리즈, 로드맵 관리               |

OSI는 오픈소스 라이선스를 소프트웨어를 자유롭게 사용·수정·공유할 수 있게 하는 라이선스로 설명하며, OSI 승인을 위해서는 Open Source Definition을 준수해야 합니다. ([Open Source Initiative][6]) AI 영역에서도 OSI의 Open Source AI Definition은 사용, 연구, 수정, 공유의 자유와 함께 데이터 정보, 학습·실행 코드, 파라미터 접근을 중요한 요소로 봅니다. ([Open Source Initiative][7])

## 4회차 — AI를 활용한 오픈소스 기여와 운영 자동화

**핵심 메시지**

AI는 코드를 대신 써주는 도구가 아니라, 오픈소스 기여의 진입장벽을 낮춰주는 **기여 보조 시스템**입니다.

**강의 내용**

| 파트          | 내용                            |
| ----------- | ----------------------------- |
| AI 기반 코드 이해 | 낯선 저장소 구조 파악, 의존성 분석, 함수 설명   |
| AI 기반 이슈 분석 | 버그 재현, 원인 추정, 수정 범위 산정        |
| AI 기반 PR 작성 | 변경 요약, 테스트 설명, 리뷰 대응          |
| AI 기반 문서화   | README, 튜토리얼, 예제, FAQ 생성      |
| AI 기반 운영    | 이슈 라벨링, 릴리즈 노트, changelog 자동화 |
| AI 프로젝트 기술  | RAG, MCP, A2A, 에이전트 워크플로우 개요  |

RAG는 모델의 내부 파라미터 지식과 외부 검색 기반 지식을 결합해 답변하는 접근으로, 지식 집약적 태스크에서 널리 쓰이는 방식입니다. ([arXiv][9]) MCP는 LLM 애플리케이션이 외부 데이터 소스와 도구에 표준 방식으로 연결되도록 하는 공개 프로토콜이고, 컨텍스트 공유·도구 노출·조합 가능한 워크플로우 구성을 목표로 합니다. ([Model Context Protocol][10]) A2A는 서로 다른 프레임워크나 벤더의 AI 에이전트들이 통신하고 협업할 수 있게 하는 오픈 표준으로 설명됩니다. ([a2a-protocol.org][11])

## 5회차 — AI 멀티 에이전트로 오픈소스 프로젝트 개발하기

**핵심 메시지**

AI를 잘 쓰는 개발자는 하나의 챗봇에게 질문하는 사람이 아니라, 역할이 나뉜 AI 작업 흐름을 설계하는 사람입니다.

**강의 내용**

| 파트         | 내용                                           |
| ---------- | -------------------------------------------- |
| 멀티 에이전트 개념 | Planner, Coder, Reviewer, Tester, Documenter |
| 오픈소스 개발 흐름 | 이슈 → 설계 → 구현 → 테스트 → 문서 → PR                 |
| 에이전트 역할 분리 | 코드 생성 에이전트와 리뷰 에이전트 분리                       |
| 안전한 AI 개발  | hallucination, 보안, 라이선스, 의존성 검증              |
| 자동화 예시     | PR 리뷰 봇, 문서 생성 봇, 테스트 생성 봇                   |
| 실전 데모      | 작은 기능을 멀티 에이전트 방식으로 개발                       |

A2A 공식 문서는 A2A를 에이전트 간 통신 표준으로, MCP를 에이전트가 도구·API·리소스에 연결되는 보완 표준으로 설명합니다. 따라서 5회차에서는 “MCP는 도구 연결, A2A는 에이전트 간 협업”이라는 식으로 구분해 설명하면 학생들이 이해하기 쉽습니다. ([a2a-protocol.org][11])

# 평가 기준 예시

점수 평가가 필요하다면 다음 비율이 좋습니다.

| 항목        |  비중 | 기준                           |
| --------- | --: | ---------------------------- |
| 오픈소스 이해도  | 20% | 라이선스, 기여 문화, 운영 구조 이해        |
| GitHub 협업 | 20% | Issue, PR, Review, Branch 사용 |
| 실제 기여/운영  | 25% | 외부 PR 또는 자체 프로젝트 운영 결과       |
| AI 활용 능력  | 20% | AI를 적절히 활용하고 검증했는가           |
| 발표/문서화    | 15% | README, 발표, 회고, 커리어 연결성      |

---

# 커리큘럼에서 특히 보강하면 좋은 부분

가장 중요한 보강점은 **3회차 라이선스 특강을 법률 지식으로만 끝내지 않는 것**입니다. 라이선스, README, CONTRIBUTING, CODE_OF_CONDUCT, SECURITY, Issue Template까지 묶어서 “오픈소스 운영의 신뢰 구조”로 가르치는 편이 좋습니다.

두 번째는 **AI 활용을 4~5회차에만 몰아넣지 않는 것**입니다. 1회차부터 AI로 프로젝트 분석하기, 2회차에서 AI로 PR 요약하기, 3회차에서 라이선스 리스크 질문하기처럼 매주 조금씩 쓰게 해야 합니다.

세 번째는 **최종 목표를 ‘발표’가 아니라 ‘공개 흔적’으로 잡는 것**입니다. 수업이 끝났을 때 학생 GitHub에 다음 중 하나가 남아야 합니다.

| 최소 결과물                       |
| ---------------------------- |
| 외부 오픈소스 PR 1개 이상             |
| 자체 오픈소스 저장소 1개               |
| README/라이선스/기여 가이드가 갖춰진 프로젝트 |
| Issue/PR/Review 기록           |
| AI 활용 회고 문서                  |
| 포트폴리오에 넣을 수 있는 발표 자료         |

이 강의는 대학생·대학원생이 AI 시대의 오픈소스 생태계를 이해하고, GitHub 협업·라이선스·기여 문화·AI 개발 도구를 실습하며, 6주 안에 실제 오픈소스 기여 또는 공개 프로젝트 운영 경험을 완성하는 실전형 과정입니다.
