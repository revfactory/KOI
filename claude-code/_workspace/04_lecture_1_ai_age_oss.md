# 1회차 — AI 시대의 오픈소스: 왜 지금 다시 중요한가

> **핵심 메시지** 오픈소스는 더 이상 "코드를 공개하는 행위"만이 아니라, AI 시대의 학습·협업·커리어·제품 확산 방식이다.
>
> **선수 지식**
> - GitHub 계정 보유 (없으면 강의 시작 전 30분 사전 학습 시간에 발급)
> - 한 가지 프로그래밍 언어를 하나 이상 만져본 경험
> - AI 챗봇(ChatGPT, Claude, Gemini 중 하나)을 한 번이라도 써본 경험
>
> **이 회차가 끝나면 학생은**
> - 2025년 오픈소스 생태계의 변화 3가지(AI/에이전트, 신규 개발자 유입, 타입 언어 확산)를 자기 언어로 설명할 수 있다. (LO 1)
> - GitHub Trending에서 인기 프로젝트를 골라 README · Issue · PR · Release · License 5개 축으로 분석할 수 있다. (LO 2)
> - 자신의 관심 분야에서 6주 동안 따라갈 후보 OSS(Open Source Software, 오픈소스 소프트웨어) 프로젝트 3개를 후보군으로 정리할 수 있다. (LO 3)

> **회차 일시** 2026-05-06(수) 19:00~21:00 (강의 90분 + 멘토링 60분)
> **트랙 분기** 본 회차는 트랙 분기 없음(전원 공통). 단 후보 프로젝트 3개를 정리할 때 두 트랙(A: 외부 OSS 기여 / B: 자체 공개 저장소 운영)의 양쪽 후보를 모두 두는 것을 권장.
> **누적 산출물 (오늘 GitHub에 남길 것)**
> 1. 개인 GitHub 프로필 README 초안 (1단락 + 학습 흔적 영역)
> 2. 후보 OSS 프로젝트 3개 정리표 (저장소 / 활동성 / 라이선스 / 기여 난이도 후보)
> 3. 팀 결성 + 팀 헌법 1페이지 초안 (팀명 / 역할 / 회의 주기 / 의사결정 / 이탈 처리)
> 4. AI(인공지능)로 후보 1개를 분석한 노트 (AI 사용 로그 양식 첫 노출)

> **여러분이 가장 마지막으로 `npm install` 또는 `pip install`로 깔았던 라이브러리 이름.** 한 사람당 하나, 30초 드릴게요. 시작.

### 실습 EX-1B: GitHub Trending 5축 분석

**소요시간** 30분 (작업 25분 + 갤러리 워크 5분)
**난이도** ★★ (입문~초급)
**산출물** 페어당 1슬라이드 또는 1포스트잇 (5축 분석 결과)
**도구** GitHub (https://github.com/trending), 학생 노트북, 핸드아웃의 5축 분석 양식
**팀 단위** 2명 1조 페어 (3명일 경우 한 명은 옵저버 역할 후 갤러리 워크에서 보완)

**목표**
GitHub Trending 또는 강사 추천 풀에서 인기 OSS 프로젝트 1개를 골라, README · Issue · PR · Release · License 5개 축으로 분석하고 한 슬라이드(또는 한 장)에 정리한다. "별 수가 많다고 좋은 프로젝트는 아니다"는 통찰을 손으로 발견한다.

**진행**

1. **페어 짝짓기 (1분)**
   - 강사가 좌우 줄 단위로 페어를 정한다. 같은 학과/학년끼리 안 묶이도록 의도적으로 섞기.
   - 페어 한쪽이 노트북, 다른 한쪽이 핸드아웃 작성 담당. 5분 후 역할 교대.

2. **Trending 페이지 열기 (1분)**
   ```
   https://github.com/trending
   ```
   - 기본은 "All languages, Today" 탭. 다른 탭(Languages: TypeScript / Python / Rust, Time: This week)도 둘러봅니다.

3. **분석 대상 1개 선택 (3분)**
   - 다음 우선순위로 1개만 선택:
     1. 페어 둘 다 분야에 관심이 있는 것 (웹/AI/모바일/시스템 등).
     2. README가 영어 또는 한국어로 읽힌다.
     3. 별 수 100개 이상.
   - 선택이 어려우면 강사 추천 풀(아래 fallback)에서 1개 골라도 됩니다.

4. **5축 분석 (15분)**
   - 다음 표를 핸드아웃에 채웁니다. 막히면 옆에 강사·멘토 부르세요.

5. **한 슬라이드 / 포스트잇 정리 (5분)**
   - 5축 핵심 발견을 한 장에 압축. 발표하지 않더라도 다음 단계(EX-1C AI 미니)에서 입력으로 사용.

3. AI가 "이건 좋은 진입점"이라고 한 이슈가 진짜 24~48시간 만에 끝낼 수 있는 크기인가?
4. AI 답에 출처 URL이 함께 나오는가? (출처 없으면 hallucination 가능성 ↑)

**hallucination 위험 지점 (강사가 학생에게 보여줄 것)**
- AI가 "good-first-issue 라벨이 있다"고 했는데 실제 라벨 이름이 다를 수 있음 (예: `help wanted`, `easy`, `starter`).
- AI가 추천한 이슈 번호가 닫힌 지 오래됐거나 아예 없을 수 있음.
- AI가 라이선스를 잘못 답할 수 있음 (특히 큰 프로젝트의 sub-package가 다른 라이선스인 경우).

> **강사 메모 — 라이브 데모 운영 핵심**
> - 시범으로 1번만. 학생들이 자기 노트북에서 따라 할 시간(2~3분)을 별도 확보.
> - 학생들이 자기 화면에서 비슷한 답을 받으면 손드는 식으로 페이스 체크.
> - 답이 다르게 나오는 게 정상 — AI 응답이 결정적이지 않음을 실시간으로 보여주는 게 메시지.
> - 만약 AI가 hallucination을 즉석에서 보여주면 그게 가장 좋은 1회차 학습 자료. "사용했는가가 아니라 검증했는가"의 살아있는 증거.

### 강사 멘트 (5분 시작 시)

> "자, 5분 동안 AI에게 시킬 게 있습니다. 방금 분석한 프로젝트 README를 AI에 붙이고, 한 줄 요약 + 기여 진입점 3개를 받아봅시다.
>
> 단, 받은 답을 그대로 믿지 마세요. **AI가 알려준 진입점 1개를 직접 GitHub에서 확인**하세요. 라벨이 진짜 있는지, 이슈가 진짜 진입할 만한지.
>
> 우리는 오늘부터 AI를 매주 한 번 이상 손에 쥡니다. 4·5회차에 몰아 쓰지 않습니다. 그리고 매번 'AI 출력 + 사람이 검증한 것'을 한 표에 적어둡니다. 이게 'AI 사용 로그'고, 4회차 PR 템플릿에 강제 필드로 들어갑니다.
>
> 5분 시작."

### 체크리스트

- [ ] AI가 준 진입점 3개 중 1개를 GitHub에서 확인했다.
- [ ] "AI 사용 로그" 표 1행을 채웠다 (사람이 검증한 것 칸 비어있지 않음).
- [ ] 결과를 채택/수정/기각 중 하나로 분류했다.

### 막혔을 때 (트러블슈팅)

| 증상 | 원인 | 대응 |
|-----|-----|-----|

> - "AI 답이 틀렸다" 사례를 발견한 학생을 일부러 강사가 호명해 1줄 발표시켜라. "환각이 나왔다"는 자각이 4·5회차의 검증 문화를 만든다.

> - 본 실습은 source.txt 보강 요청 2번의 핵심 진입점이다. 절대 생략하지 마라. 시간이 부족하면 개념 2의 강의를 줄여라.
> - "사용했는가가 아니라 검증했는가" 한 마디를 5분 안에 한 번 더 박는다. 이게 4회차에서 평가 룰로 쓰인다.

| 팀 헌법을 5분 안에 못 채움 | 팀원 의견 충돌 | "오늘은 잠정으로 채우고 다음 주 멘토링 때 확정" 멘트 — 미완 헌법도 OK |

### 확장 (시간 남으면 또는 멘토링 시간에)

- 팀 헌법을 GitHub Issues에 1개 이슈로 발행 (다음 주 팀 저장소 만들면 거기 첫 이슈로).
- 팀명 로고 1초 손그림 — 6회차 발표 자료에 활용.

> **강사 메모 — EX-1D 운영 핵심**
> - source.txt가 1회차에 명시한 "팀 빌딩"은 이 EX-1D다. 절대 생략 금지.
> - PR 경험자 분포 균등화는 강사의 가장 중요한 1회차 책임이다 (마스터 플랜 §5.1 위험 — 실력 격차 완충).
> - 10분 안에 모든 팀 헌법이 다 채워질 수 없다. 4항목 중 3개라도 잠정으로 채우고 다음 주 멘토링에서 보완 — 60% 완성도가 1회차 합격선.
> - "이탈 / 갈등 처리" 항목은 첫 주에 학생에게 어색할 수 있지만, 3회차 이후 팀 분쟁 위험을 미리 막는다. 한 줄이라도 적게 만들어라.
> - 팀 헌법은 다음 주 팀 저장소 `docs/team-constitution.md`로 옮긴다 — 2회차에서 누적.

> **멘토 메모 — 1회차 멘토링 핵심**
> - 1회차 멘토링은 점검이 아니라 **약속의 첫 사인**. 학생이 다음 주에 다시 와야 할 이유를 손에 쥐게 하는 자리.
> - 위험 신호(팀 이탈, AI 도구 0인 학생, 후보 3개 중 활동성 0)는 즉시 강사에게 메모로 공유 → 다음 주 시작 전 조정.
> - 60분이 빠듯하다. 팀당 5~7분 라운드로 빈 팀 없이 한 번씩.
> - 멘토 사인은 "잘했다"가 아니라 "약속을 받았다"의 의미.

### A.2 슬라이드 마스터 권장

- **헤더**: 좌상단에 "1회차 / 0:XX~0:XX / 단계명".

- **색상**: 강조 1색 + 본문 1색 + 배경 1색 (3색 이상 사용 금지).
- **폰트**: 본문 한글 24pt 이상, 영문 22pt 이상 (강의실 뒷자리 가독성).

### A.3 코드 블록 / 표 처리

- 코드 블록은 슬라이드 1장당 **15줄 이내**. 초과 시 부록 슬라이드로 분리.

### A.4 강사 메모 분리
