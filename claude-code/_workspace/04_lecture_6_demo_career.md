- **LO 2.** 5분 발표 + 5분 Q&A로 6주 여정을 8개 항목으로 말하기
- **LO 3.** 30일 안에 실행할 다음 기여 1건 액션 플랜 + 강사 사인

**강사 멘트**: "이 3개는 모두 GitHub URL로 검증 가능한 흔적이다. '이해했다'가 아니라 '올렸다/보냈다/사인 받았다'."

### 3.3 운영 노트

- 본 3분 안에 LO 3개를 명시하지 않으면 학생들이 "이게 졸업인가? 그냥 발표만 하는 건가?" 혼란. 반드시 LO 슬라이드 1장 노출.

## 5. 0:08~0:18 — EX-6B GitHub 프로필 정돈 1차 (10분)

### 5.1 why-now

> 직전 5분에 "내 프로필이 채용 담당자에게 어떻게 보이는가"를 옆 사람을 통해 들었다. 그 충격이 식기 전에, 10분 안에 1차 정돈을 끝낸다. 완벽이 아니라 "대분류" — README 1단락 / pinned 6개 / 사회 링크. 미세 조정은 사후 과제.

### 5.2 강사 스크립트

> **(0:08)** 지금부터 10분, 각자 노트북 켠다. 지금 옆 사람 코멘트가 머릿속에 있을 때 빠르게 1차 정돈한다. 7항목 체크리스트가 핸드아웃에 있다. 위에서부터 순서대로 친다.
>
> **(0:09)** 1번 — 프로필 README 1단락. 한 줄 자기소개 + 한 줄 관심 분야 + 한 줄 동작 호출. 영어든 한국어든 OK. 둘 다면 더 OK.
>
> **(0:11)** 2번 — pinned repo 6개. 우리 팀 저장소 1개 + 본 강의 누적 산출물 저장소 1개 + 본인이 자랑하고 싶은 4개. 6개를 채우는 게 핵심이다. 빈 자리는 약점이다.
>
> **(0:14)** 3~7번은 사회 링크, 이메일, 한 줄 동작 호출 같은 것들. 빠르게 친다. 핸드아웃 보고 따라가자.
>
> **(0:17)** 마지막 1분, 옆 사람과 1분 갤러리 워크. "어떻게 바뀌었는지" 30초 보고 30초 코멘트.

### 5.3 핸드아웃 — 7항목 정돈 체크리스트

```
□ 1. 프로필 README 1단락
   - 한 줄 자기소개 (이름·소속·관심 분야)
   - 한 줄 현재 무엇을 하는지 (지금 진행 중인 프로젝트 1개)
   - 한 줄 동작 호출 ("open to internship", "looking for ...")

□ 2. pinned repo 6개 선택
   - 본 강의 팀 저장소 1개 (필수)
   - 본인 자랑 4개
   - 학습용/실험용 1개 (선택)

□ 3. 사회 링크 (Edit profile → Social accounts)
   - LinkedIn / Twitter(X) / 블로그 / 이메일 중 1개 이상

□ 4. 프로필 사진
   - 얼굴 또는 일관된 아이콘 (기본 회색 실루엣 금지)

□ 5. 위치(Location)
   - 한국·서울·대학명 등 (선택, 비워둘 수도 있음)

□ 6. 기여 그래프 노출 확인
   - Settings → Public profile → Activity 모두 공개

□ 7. 한 줄 자기소개 (Bio, 160자 제한)
   - "AI 시대의 OSS 기여자 / 학생 개발자 / [관심 분야]"
```

### 5.4 프로필 README 1단락 표준 골격 (학생이 그대로 복붙)

```markdown
### Hi, I'm [이름] 👋

대학교 [학년]·[전공]. 관심 분야는 [AI / 백엔드 / 데이터 / 모바일 / OSS 운영].

지금은 **[팀명]** 에서 [한 줄 — 어떤 OSS에 어떤 기여를 / 어떤 자체 프로젝트를]
를 하고 있다. 6주 동안 만든 흔적은 pinned repo 1번에서 볼 수 있다.

- 🔭 **지금 만들고 있는 것**: [한 줄]
- 🌱 **지금 배우는 것**: [한 줄]
- 💬 **이야기 나누고 싶은 주제**: [한 줄]
- 📫 **연락**: [이메일 또는 LinkedIn]
- ⚡ **6주 OSS 강의 흔적**: [팀 저장소 URL] / [개인 PR 모음]
```

> **why-now**: README가 1줄짜리 인사로 끝나면 채용 담당자는 30초도 못 머무른다. 5줄짜리 ‘무엇을 / 왜 / 어디서 볼 수 있는지’ 골격이 30초 시선을 60초로 늘린다.

### 5.5 pinned repo 6개 선택 가이드

| 자리 | 무엇을 핀 할지 | 보여주려는 메시지 |
|---|----|----|
| 1 | 본 강의 팀 저장소 | "최근 6주 진짜 협업한 경험" |
| 2 | 본인 외부 PR 또는 자체 운영 저장소 | "공개 흔적 — fork/star 가능" |
| 3 | 가장 코드 비중 큰 본인 프로젝트 | "코드 능력 증명" |
| 4 | 특정 도메인(AI/Web/Data) 프로젝트 | "관심 분야 시그널" |
| 5 | 학습용 — 잘 정리된 README가 있는 곳 | "학습 능력·문서화 능력" |
| 6 | 실험용 / 사이드 — 본인의 호기심 | "성장 폭" |

### 5.6 운영 노트

- 학생이 7항목 모두 못 끝내면 "1·2번만이라도" 우선. 3~7번은 사후 과제.

## 7. 1:08~1:13 — 본 회차 유일한 강의 블록 (5분)

### 7.1 why-now

### 7.2 슬라이드 (3장)

#### 슬라이드 6-4 (1:08~1:10) — 좋은 기여자의 태도 4가지

**제목**: 좋은 기여자는 누구인가

| 태도 | 한 줄 정의 | 학생 행동 |
|----|----|----|
| 1. 작은 기여 | 1줄 typo·번역·docs PR로 시작 | up-for-grabs / First Timers Only / good-first-issue |
| 2. 꾸준함 | 1주 1 PR보다 1년 50 PR | contribution graph가 증명 |
| 3. 리뷰 대응 | 리뷰어의 코멘트에 24시간 안에 응답 | 안 받아들여도 답변·근거·재PR |
| 4. 문서화 | 코드만 짜는 사람 ≠ 좋은 기여자 | README·CHANGELOG·comment까지 |

**강사 멘트**: "5주 동안 우리는 1번~3번을 다 했다. 4번이 약하다. 30일 액션 플랜에 4번을 넣어라."

#### 슬라이드 6-5 (1:10~1:11) — 오픈소스 커리어 5경로

**제목**: GitHub 흔적이 만드는 다음 자리

```
1. 인턴십 — 채용 담당자 30초가 본인 GitHub. 기여 그래프가 살아있는가.
2. 연구 — 연구실 코드를 GitHub에 공개. 동료 평가가 인용으로 이어진다.
3. 취업 — 1년 200 PR이 학력보다 강한 신호일 수 있다.
4. 창업 — 본인이 만든 OSS의 user 100명이 첫 customer 100명이다.
5. 커뮤니티 — 한국 OSS 커뮤니티 (KOSSLAB · 한국 카카오·네이버·라인 OSS) → 멘토 → 메인테이너.
```

**강사 멘트**: "5경로 중 3개 이상은 6주 강의 끝나도 GitHub에 흔적 안 쌓이면 못 들어간다."

#### 슬라이드 6-6 (1:11~1:13) — 메인테이너로 성장하는 길

**제목**: 기여자 → 메인테이너 6단계

```
1. 첫 PR 머지              ← 1~5회차에서 했음
2. 같은 저장소 5번째 PR     ← 30일 액션 플랜
3. Issue triage 도와주기    ← 90일 액션
4. 다른 기여자 PR 리뷰      ← 6개월 안
5. 작은 sub-module 책임     ← 1년 안
6. core maintainer 후보     ← 2년 이상
```

**강사 멘트**: "본 강의에서 1번만 했다. 2번이 30일 액션 플랜이다. 6번까지 가는 사람이 본 강의의 진짜 졸업생이다."

### 7.3 강사 스크립트 (5분 통째)

> **(1:08)** 자, 본 회차에 단 한 번 강의다. 5분이다. 슬라이드 3장.
>
> **(1:09)** **좋은 기여자의 태도 4가지** — 작은 기여 / 꾸준함 / 리뷰 대응 / 문서화. 1~3번은 5주 동안 다 했다. 4번이 약하다. 30일 액션 플랜에 문서화를 넣어라.
>
> **(1:10)** **오픈소스 커리어 5경로** — 인턴 / 연구 / 취업 / 창업 / 커뮤니티. 30초 채용 시뮬레이션이 이미 1번을 시연했다.
>
> **(1:11)** **메인테이너로 성장하는 6단계** — 우리는 1번 했다. 30일 액션 플랜은 2번이다. 6번까지 가는 사람이 진짜 졸업생.
>
> **(1:12)** 한 줄로 — *오픈소스 커리어는 GitHub 활동 그래프가 자기 자신을 증명하는 게임이다.* 다음 12분 EX-6D가 그 게임을 30일짜리로 잘게 쪼갠다.

## 13. A/B 트랙 분기 — 6회차는 통합 (분기 없음)

> 운영표 §8.1 — *회차 6: 분기 없음 (발표 통합).* 마스터 플랜 §4.3 — *6회차 발표는 두 트랙이 같은 8개 항목으로 발표한다. 이로써 평가 형평성이 유지된다.*

### 13.1 본 회차 분기 없음의 이유

| 이유 | 설명 |
|----|----|
| 평가 형평성 | A/B 트랙이 발표 형식·평가 항목이 다르면 채점이 어그러진다. 8개 항목이 트랙 무관 통일. |
| 청중 학습 | A·B 트랙 결과를 동일 무대에서 보면 서로 배운다. 분기 운영은 학습 손실. |
| 30일 액션 플랜 | A/B 트랙 무관 같은 양식 — 저장소 / 기여 종류 / AI 도구 / 검증. |

### 13.2 단, 발표 슬라이드에서 트랙 구분 1줄

발표 슬라이드 1번 상단에 **"트랙: A (외부 OSS 기여)"** 또는 **"트랙: B (자체 공개 저장소 운영)"** 1줄 명시. 청중·강사가 평가 시 트랙 차이를 인지하도록.

### 13.3 트랙별 발표 8개 항목 차이 (참고)
