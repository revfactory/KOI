# 5회차 — AI 멀티 에이전트로 오픈소스 프로젝트 개발하기

> **직전 회차로부터의 전이 (도입 첫 30초에 강사가 그대로 말함)**
>
> "지난주에 우리는 AI를 한 명의 도구처럼 썼다. 너희가 만든 PR 초안에서 직접 발견했지 — 한 명의 AI에게 분석·재현·수정·문서까지 다 시키니까, 어느 시점부터 답이 흐려졌다. **오늘은 그 한 명을 다섯 명으로 쪼갠다.** 한 명의 챗봇이 아니라, 역할이 나뉜 AI 작업 흐름을 설계하는 사람이 된다."

---

> **핵심 메시지** (도입과 마무리에 동일하게 등장 — 샌드위치 룰)
> AI를 잘 쓰는 개발자는 챗봇에게 질문하는 사람이 아니라, **역할이 나뉜 AI 작업 흐름을 설계하는 사람**이다.
>
> **선수 지식**
> 4회차까지의 산출물 — 본격 PR 1개(외부 또는 자체) + AI 사용 로그(작업·프롬프트·검증) + RAG/MCP/A2A 한 줄 정의 노트. 4회차에서 너희는 "AI 한 명에게 5가지 역할을 다 시켰을 때 답이 흐려진다"는 인지를 가지고 들어온다.
>
> **이 회차가 끝나면 (오늘의 산출물 약속)**
> 1. 멀티 에이전트로 만든 기능 PR 1개
> 2. 테스트 또는 데모
> 3. 문서 업데이트 (CONTRIBUTING.md에 "AI가 제안한 코드의 라이선스 출처가 의심스러우면 머지하지 않는다" 룰 명문화 포함)
> 4. AI 멀티 에이전트 사용 기록 — 5역할별 프롬프트와 결과
> 5. **최종 발표 초안** — 6회차에 다듬을 5분 분량 슬라이드 5장 + 5분 측정 1회 완료

## 사전 학습 (수업 전 30분)

### 학생용 사전 학습 체크리스트

수업 전에 아래 4가지를 마치고 들어온다. 시간이 부족하면 (3)·(4)는 강의 시작 전 10분 안에 따라잡을 수 있도록 강사가 슬라이드 1장으로 요약 노출.

1. **(필수, 10분) 4회차 산출물 다시 보기** — 너희가 만든 본격 PR 본문과 AI 사용 로그를 다시 읽고, "한 명의 AI에게 너무 많은 역할을 시켜서 답이 흐려졌던 지점" 1곳을 찾아 메모. (이 메모가 0:00~0:03 직전 전이 발화의 동력이 된다.)
2. **(필수, 10분) MCP / A2A 한 줄 정의 복습** — 4회차에서 적은 한 줄 노트를 다시 펼친다.
   - **MCP** — LLM 애플리케이션이 외부 데이터·도구에 표준 방식으로 연결되는 오픈 프로토콜. 1차 출처: [modelcontextprotocol.io](https://modelcontextprotocol.io/).
   - **A2A** — 서로 다른 프레임워크/벤더의 AI 에이전트가 통신·협업하는 오픈 표준. 1차 출처: [a2a-protocol.org](https://a2a-protocol.org/latest/) (Linux Foundation 이관, 2025).
3. **(권장, 5분) Claude Code sub-agents 또는 Cursor agents 또는 CrewAI 중 1개 환경 점검**
   - Claude Code 사용자 — `claude --version` 으로 CLI 동작 확인. 공식: https://docs.claude.com/en/docs/claude-code/sub-agents
   - Cursor 사용자 — Cursor 1.0 이상 / Agents 메뉴 노출 확인.
   - 둘 다 안 되면 — CrewAI 또는 1명이 5역할 순차 시뮬레이션(fallback) 환경. CrewAI: https://docs.crewai.com/
4. **(권장, 5분) MCP awesome 목록 1번 훑기** — [punkpeye/awesome-mcp-servers](https://github.com/punkpeye/awesome-mcp-servers) 에서 너희 프로젝트에 쓸 만한 MCP 서버 1개 후보 메모. (filesystem / github / postgres / fetch 중에서 출발 권장.)

### 사전 학습 산출물

## 0:13~0:30 — 개념 1블록: 5역할 + MCP/A2A 심화 (17분)

### why-now

### 0:13~0:20 (7분) — 5역할 책임표

#### 역할 책임표 (슬라이드 1장)

| 역할 | 인풋 | 아웃풋 | 책임 (한 문장) | 사람이 검증할 것 |
|----|----|----|------|-------------|
| **Planner** | 이슈 또는 기능 요구 | 작업 카드 N개 (To-Do) | 큰 일을 작은 일로 쪼개고 우선순위를 매긴다 | 작업 카드가 빠짐없이 이슈를 덮는가 |
| **Coder** | 작업 카드 1개 | 변경된 파일 + 커밋 | 카드 1개를 코드로 구현한다 | diff가 카드 범위를 넘지 않는가 |
| **Reviewer** | PR diff | 리뷰 코멘트 + 승인/요청변경 | 라이선스·보안·스타일·로직을 점검한다 | 외부 라이선스 의심 코드 차단 |
| **Tester** | 변경된 코드 | 테스트 케이스 + 실행 결과 | 정상 시나리오 + 경계 + 회귀 테스트 | 실제 실행되는가 (가짜 통과 금지) |
| **Documenter** | 머지된 변경 | README / CHANGELOG / API 문서 갱신 | 외부 사용자가 변경을 이해할 수 있게 만든다 | 문서가 코드와 맞는가 |

#### 강사 멘트

> "이 표를 외울 필요는 없어. 다만 한 가지만 기억해 — **각 역할은 인풋과 아웃풋이 다르다.** 같은 LLM에게 똑같은 컨텍스트를 던지면서 '코더 모자 썼다 → 리뷰어 모자 썼다' 하는 게 아니라, 인풋 자체를 분리해서 던진다. 그래야 답이 흐려지지 않는다."

#### 1. 멀티 에이전트 환경 셋업 실패 30% 초과

- **2차**: 6회차 도입 5분에 환경 점검 미니 슬롯 추가.

#### 2. AI 사용 로그 양식이 비어 있다

## 부록 C: 평가 매핑 — 학생이 본 회차 활동을 안 하면 어디서 점수를 잃는가

마스터 플랜 §3.6에 따라 5회차는 **실제 기여 25% 마무리 측정 / AI 활용 20% 심화 측정 / 발표 15% 시작점**의 3개 평가 항목이 동시에 측정되는 회차다.

| 활동 안 하면 | 잃는 평가 항목 | 잃는 점수 |
|---------|----------|------|
| 멀티 에이전트 PR을 안 만들면 | 실제 기여/운영 25% | 약 5~10점 |
| 5역할 프롬프트 로그를 안 쓰면 | AI 활용 능력 20% | 약 5~10점 |
| CONTRIBUTING.md AI 정책 안 추가하면 | 발표·문서화 15% / AI 활용 20% (안전장치) | 약 3~5점 |
| 발표 초안 5분 측정 안 하면 | 발표·문서화 15% | 약 3점 + 6회차 시간 초과 감점 |
| 4축 안전 장치 점검 흔적 0이면 | AI 활용 능력 20% (검증 부재) | 약 5~7점 |

**핵심 룰 재노출**: AI 활용 능력 20%의 채점은 "사용했는가"가 아니라 **"사용하고 검증했는가"**.

```
  - Tester: 프로젝트에 첫 테스트가 없다면 이번에 도입
  - Documenter: README / CHANGELOG 갱신

산출물 표기:
  - PR URL (자체)
  - 5역할 프롬프트 로그
  - 머지된 commit hash
  - CHANGELOG 엔트리
```

### 합류 (1:00~1:30)

분기는 EX-5B 30분만. 1:00 이후는 두 트랙 모두 같은 합류 활동.
- 안전 장치 4종 (공통)
- 발표 초안 5분 측정 (공통, 5장 구조 동일)
- KPT (공통)

## 부록 G: MCP 서버 카탈로그 (학생이 EX-5B에서 골라 쓸 수 있는 7종)

자료 풀 §3.5A 5A-1 (modelcontextprotocol/servers) 의 reference 서버 중 학생 30분 실습에 적합한 7개를 추린다. 각 서버는 1줄 설치 + 1줄 등록으로 즉시 사용 가능.

| # | 서버 | 용도 | 설치 명령 | 5역할 매핑 |
|---|----|----|-----|--------|
| MCP-1 | filesystem | 로컬 파일 읽기·쓰기 | `npx -y @modelcontextprotocol/server-filesystem ~/projects` | Coder / Documenter |
| MCP-2 | github | 이슈·PR 읽기·쓰기 | `npx -y @modelcontextprotocol/server-github` | Planner / Reviewer |
| MCP-3 | fetch | HTTP API 호출 | `npx -y @modelcontextprotocol/server-fetch` | Coder (외부 API 호출) |
| MCP-4 | git | 로컬 git 조작 | `npx -y @modelcontextprotocol/server-git --repository ~/projects/myrepo` | Coder / Reviewer |
| MCP-5 | sqlite | 로컬 DB 조회 | `npx -y @modelcontextprotocol/server-sqlite ~/db.sqlite` | Coder (DB 접근) |
| MCP-6 | postgres | Postgres DB 조회 | 환경변수로 연결 문자열 | Coder (실제 운영 DB) |
| MCP-7 | memory | 에이전트 간 컨텍스트 공유 | `npx -y @modelcontextprotocol/server-memory` | Planner ↔ Coder |

### Claude Code 등록 예시

```bash
# 1) filesystem
claude mcp add fs npx -y @modelcontextprotocol/server-filesystem ~/projects

# 2) github (token 필요)
export GITHUB_TOKEN=ghp_...
claude mcp add github npx -y @modelcontextprotocol/server-github

# 3) fetch
claude mcp add fetch npx -y @modelcontextprotocol/server-fetch

# 등록 확인
claude mcp list
```

### Cursor 등록 예시 (`~/.cursor/mcp.json`)

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me/projects"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    }
  }
}
```

### 학생이 EX-5B에 5분 안에 1개 등록하는 미니 가이드

```
1. 옵션 선택 — Claude Code / Cursor 둘 중 하나
2. MCP-1 (filesystem) 부터 시작 — 가장 안전·간단
3. claude mcp add 또는 cursor mcp.json 편집
4. 새 대화 시작 → "Planner sub-agent 호출"
5. Planner가 filesystem 서버로 README를 읽었는지 화면 확인
   (사이드바에 "fs" 도구 호출 로그가 보임)
```

### 보안 주의 (안전 장치 2축 발동)

- `GITHUB_TOKEN` 같은 비밀키는 절대 코드에 박지 말고 환경변수만 사용.
- filesystem 서버 경로는 너무 넓게 잡지 말 것 (`~/` 전체보다 `~/projects` 처럼 좁게).
- `.claude/settings.local.json` 또는 `.cursor/mcp.json` 은 `.gitignore` 에 포함.
- MCP 서버를 쓰는 순간 `Cmd Inj`, `SSRF`, `Prompt Injection` 위험이 등장 — OWASP LLM Top 10 LLM01 (Prompt Injection) 참고.

## 부록 H: 5역할 프롬프트 라이브러리

학생이 EX-5B에서 그대로 복사해 쓸 수 있는 5역할 프롬프트 템플릿 모음. 각 프롬프트는 한국어 + 영어 2버전.

### Planner 프롬프트

```
[한국어]
너는 오픈소스 프로젝트의 Planner다. 다음 이슈 본문과 README를
인풋으로 받아, 5줄 이내의 작업 카드 N개로 쪼갠다.

이슈: {{issue_body}}
README: {{readme_excerpt}}

각 카드는 다음 형식:
- ID (T-1, T-2, ...)
- 한 줄 제목
- 변경할 파일 경로 후보
- 완료 정의(DoD) 1줄
- 의존성 (다른 카드와의 선후 관계)

코드 작성 금지. 카드 형식만 출력.

[English]
You are a Planner for an open source project. Given the following
issue body and README excerpt as input, break down the work into N
task cards, each fitting in 5 lines.

Issue: {{issue_body}}
README: {{readme_excerpt}}

Each card format:
- ID (T-1, T-2, ...)
- One-line title
- Candidate file paths
- Definition of Done (one line)
- Dependencies (which other cards must come first)

Do not write code. Output only the card format.
```

### Coder 프롬프트

```
[한국어]
너는 Coder다. 다음 작업 카드 1개를 인풋으로 받아 코드로 구현한다.

카드: {{card}}
관련 파일: {{relevant_files}}

규칙:
- 카드 ID 외 영역은 건드리지 않는다
- 새 의존성 추가 시 라이선스를 함께 메모
- 끝나면 변경 파일 목록과 한 줄 요약 출력
- diff 형식으로 출력 (+/- prefix)

[English]
You are a Coder. Given the following single task card, implement it
in code.

Card: {{card}}
Relevant files: {{relevant_files}}

Rules:
- Do not touch areas outside the card's scope
- For any new dependency, note its license
- After done, output the list of changed files and a one-line summary
- Output in diff format (+/- prefix)
```

### Reviewer 프롬프트

```
[한국어]
너는 Reviewer다. 다음 PR diff를 인풋으로 받아 4축으로 점검한다.

PR diff: {{diff}}
프로젝트 라이선스: {{project_license}}
프로젝트 컨벤션 (있다면): {{convention_excerpt}}

4축:
1. 라이선스 — 외부 코드를 가져왔는가? 출처와 라이선스 호환?
2. 보안 — 비밀키/토큰 노출? 입력 검증 누락?
3. 로직 — 카드 범위와 일치? 회귀 가능성?
4. 스타일 — 프로젝트 컨벤션 준수?

판정:
☐ 통과
☐ 변경 요청 + 코멘트 N개 (각 코멘트는 줄 번호 명시)

[English]
You are a Reviewer. Given the following PR diff, inspect it on 4 axes.

PR diff: {{diff}}
Project license: {{project_license}}
Project convention (if any): {{convention_excerpt}}

4 axes:
1. License — Any external code? Are sources and licenses compatible?
2. Security — Any leaked secrets/tokens? Missing input validation?
3. Logic — Does it match the card's scope? Any regression risk?
4. Style — Does it follow the project's conventions?

Verdict:
☐ Approve
☐ Request changes + N comments (each with line number)
```

### Tester 프롬프트

```
[한국어]
너는 Tester다. 다음 변경된 코드에 대해 테스트를 작성하고 실제로
실행해 결과를 보고한다.

변경 코드: {{changed_code}}

- 경계 조건 1개 이상 (빈 입력 / 매우 큰 입력 / null 등)
- 회귀 방지 1개 이상 (기존 기능이 깨지지 않음)

실행 후 출력:
- 통과 ___ / 실패 ___ / 총 ___
- 실패가 있으면 어떤 케이스인지 1줄 설명

[English]
You are a Tester. Given the following changed code, write tests and
actually execute them, then report the results.

Changed code: {{changed_code}}
Project test framework: {{test_framework}}

Items to write (no fake passes):
- At least 1 happy path
- At least 1 boundary condition (empty / huge / null, etc.)
- At least 1 regression test (existing behavior preserved)

After execution:
- Passed ___ / Failed ___ / Total ___
- If any failure, describe the case in one line
```

### Documenter 프롬프트

```
[한국어]

머지 요약: {{merge_summary}}

규칙:
```

## 부록 I: 학생 자가 진단 — "내 멀티 에이전트는 어디서 막혔나?"

EX-5B 30분이 끝난 직후, 학생이 자기 작업을 5분 안에 점검할 수 있는 자가 진단 체크리스트.

### 자가 진단 7문항

```
[학생명] _________ [팀명] _________

1. 5역할을 모두 분리해 호출했는가?
   ☐ 5개 모두 / ☐ 3~4개 / ☐ 1~2개 / ☐ 한 명에게 몰아 시켰다 (4회차 회귀)

2. 각 역할의 인풋이 분리되어 있었는가?
   (Planner는 이슈만 / Coder는 카드만 / Reviewer는 diff만)
   ☐ 모두 분리 / ☐ 일부 섞임 / ☐ 같은 컨텍스트를 다 던졌다

3. 5역할 프롬프트 로그 양식 5칸을 모두 채웠는가?
   ☐ 5개 모두 / ☐ 3~4개 / ☐ 1~2개 / ☐ 0개

4. Reviewer 4축 (라이선스·보안·로직·스타일) 점검을 모두 수행했는가?
   ☐ 4축 모두 / ☐ 2~3축 / ☐ 1축 / ☐ 통과만 찍고 안 봤다

5. Tester는 가짜 통과가 아닌 실제 실행 결과를 냈는가?
   ☐ 실제 실행 / ☐ AI가 "통과했다고 봅니다" 식 / ☐ 실행 안 했다

6. CONTRIBUTING.md AI 정책 섹션을 추가했는가?
   ☐ PR 머지됨 / ☐ PR 작성 중 / ☐ 시작 못 했다

7. 6회차 발표 슬라이드 5장 중 멀티 에이전트 사례가 박힌 슬라이드는?
   ☐ 슬라이드 3 (AI 활용) / ☐ 슬라이드 4 (검증) / ☐ 둘 다 / ☐ 둘 다 없음
```

### 점수 산정

- 7문항 모두 첫 번째 칸 (☐ 5개 모두 등) → **A** — 6회차 발표에 그대로 가져가도 좋다.
- 5~6문항 첫 번째 칸 → **B** — 멘토링 60분에서 한두 가지만 보강하면 OK.
- 3~4문항 첫 번째 칸 → **C** — 멘토링이 사실상 보충 수업. 최소 2개를 멘토링 안에 끝낸다.
- 0~2문항 첫 번째 칸 → **D** — 옵션 C(1명 5역할 시뮬레이션)로라도 처음부터 다시. 다음 주 발표 형평성을 위해 강사 개입 필수.

### A/B/C/D 별 멘토링 처방

| 등급 | 멘토링 60분 처방 |
|----|----------|
| A | 발표 초안 2차 측정 + 다음 기여 후보 검색 |
| B | 빠진 1~2칸 채우기 + 발표 초안 2차 측정 |
| C | 빠진 5역할 다시 호출 1회 + 발표 초안 2차 측정 |
| D | 강사가 라이브로 옆에서 1회 같이 돌려준다 + 발표 초안은 종이 5장으로 |
