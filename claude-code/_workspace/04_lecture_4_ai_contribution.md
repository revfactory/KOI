# 4회차 — AI를 활용한 오픈소스 기여와 운영 자동화

> **지난주에 우리는 저장소를 신뢰할 수 있게 만들었다. LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT, SECURITY, Issue/PR Template — 외부 사람이 들어와도 "여기 안전하다"고 느낄 수 있는 신뢰 골격을 갖췄다. 오늘부터는 그 위에서 AI를 본격적으로 손에 쥔다. 단, 잊지 말자 — 1·2·3회차에서도 우리는 이미 매주 AI를 써왔다. 1회차에 후보 프로젝트 README를 AI에게 요약시켰고, 2회차에 PR 본문을 AI로 다듬었고, 3회차에 라이선스 리스크를 AI에게 물었다. 오늘은 그것을 **체계화하고 본격화**하는 날이다. AI는 코드를 대신 써주는 마법 도구가 아니라, 오픈소스 기여의 진입장벽을 낮춰주는 **기여 보조 시스템**이다.**

## 0. 회차 메타 정보

> **핵심 메시지 (도입·마무리 양쪽 노출, 샌드위치 룰)**
>
> **AI는 코드를 대신 써주는 도구가 아니라, 오픈소스 기여의 진입장벽을 낮춰주는 기여 보조 시스템이다.**

> **선수 지식 (이번 회차 들어올 때 챙겨와야 할 것)**
>
> - 1·2·3회차 누적 산출물 (팀 저장소 + LICENSE + CONTRIBUTING + Issue Template + PR Template + README v2)
> - GitHub 계정 + 본인 노트북에 git CLI + VS Code 또는 Cursor 또는 IDE
> - AI 도구 1개 로그인 상태 — Claude (claude.ai 웹) 또는 GitHub Copilot Free, 둘 다면 더 좋음
> - 트랙 확정 (A: 외부 OSS 기여 / B: 자체 공개 저장소 운영)
> - 팀 저장소에 우리 트랙용 작업 후보 Issue 1~3개 (멘토링 약속 카드에 적은 것)

> **이 회차가 끝나면 (학생용 한 문단)**
>
> 너는 오늘 처음 보는 외부 저장소 또는 너희 자체 저장소에서 **본격 PR 1개**를 draft로 올렸을 거다. 그 PR 본문에는 AI가 도와준 부분과 사람이 검증한 부분이 명확히 구분된 **AI 사용 로그**가 붙어 있다. 너는 이슈 → 재현 → 수정 범위 산정 → PR 본문 작성 → 리뷰 대응의 5단계 흐름의 각 지점에서 "AI에게 무엇을 시키고, 사람이 무엇을 검증해야 하는지" 구분할 수 있다. RAG, MCP, A2A — 다음 주에 깊이 들어갈 세 약어의 한 줄 정의를 노트에 적었다. 이 모든 흔적은 너희 팀 GitHub에 공개적으로 남는다.

### 1차 (5분, AI 없이)

> 강사: "지금부터 5분. AI 도구 모두 끄고 — Copilot, Claude, Cursor 모두 비활성. 브라우저 + GitHub 원본만. 5분 안에 핵심 모듈 3개 + 좋은 첫 이슈 1개를 찾아오면 손을 든다."

```bash
# 학생이 실제로 할 동선 (참고)
1. github.com/astral-sh/ruff → README 스크롤
2. 디렉토리 트리 — crates/, ruff/, scripts/ 등 어디서 막히는지 관찰
3. Issues 탭 → 라벨 필터 → "good first issue" 클릭
4. 라벨 안에 어떤 게 첫 이슈로 좋은지 판단
```

> **예상 결과**: 학급의 30~50%만 손을 든다. 손 든 학생도 "확신 없음"이라고 말함. 강사는 그 모습을 칠판에 적는다 — `1차: 손 든 학생 N명 / 전체 M명`.

### 2차 (5분, AI와 함께)

> 강사: "AI 도구를 켜라. 같은 5분, 같은 과제. 단, **AI 출력을 그대로 답으로 내지 말 것** — 너희는 사람이고, AI는 보조다."

```text
# 학생용 권장 프롬프트 (강사가 슬라이드에 띄움)

[Claude / Copilot Chat]
이 저장소를 처음 본다고 가정하고 답해줘:
1) 핵심 모듈 3개 (디렉토리 단위로)
2) 각 모듈이 무슨 일을 하는지 한 줄
3) 기여 진입점으로 좋은 첫 이슈 1개 (URL 포함)
참고: github.com/astral-sh/ruff

조건: 추측 금지. README와 CONTRIBUTING.md를 근거로 답해줘.
근거가 부족하면 "확실치 않음"이라고 말해줘.
```

> **예상 응답 패턴 (강사용 시범)**:
>
> ```
> 핵심 모듈 3개:
> 1. crates/ruff_linter/ — 실제 린트 룰 구현 (수백 개 룰 각각 파일)
> 2. crates/ruff_python_parser/ — Python 파서, 룰의 입력 AST 생성
> 3. crates/ruff_workspace/ — 설정 파일 (ruff.toml, pyproject.toml) 처리
>
> 첫 이슈 1개:
> https://github.com/astral-sh/ruff/labels/good%20first%20issue
> 위 라벨에서 "rule: ..." 로 시작하는 이슈 — 룰 1개 추가는 정해진 패턴이 있어
> 첫 PR로 적합. (단, 실제 URL은 내가 접근할 수 없으니 사람이 라벨 페이지 직접 확인 권장)
>
> 확실치 않은 부분: 룰 추가의 최신 가이드라인 — CONTRIBUTING.md 직접 확인 필요.
> ```

## 6. 0:16~0:30 — 개념 1블록: 5단계 흐름과 AI/사람 분리 (14분)

> **why-now**: 도입 훅에서 학생은 "AI 활용 = 검증 능력"을 받았다. 이제 그 검증을 **어느 단계에 어떻게** 적용하는지 명시화한다. 이 표가 오늘 모든 실습의 베이스라인이다.

### 슬라이드 S3 — 5단계 흐름 다이어그램

```
[H2] 외부 OSS 기여 또는 자체 기능 PR의 5단계

   이슈 (Issue)
       │
       ▼
   재현 (Reproduce)
       │
       ▼
   수정 범위 산정 (Scope)
       │
       ▼
   PR 본문 작성 (PR Body)
       │
       ▼
   리뷰 대응 (Review)
```

### 슬라이드 S3-2 — 단계별 AI / 사람 분리 표 (오늘의 핵심 표)

| 단계 | AI가 도와줄 수 있는 것 | 사람이 반드시 할 것 | 위험 신호 |
|----|-------------------|-----------------|--------|
| 1. 이슈 (Issue) | 이슈 본문 요약 / 비슷한 이슈 검색 / 라벨 추천 | 진짜로 메인테이너가 원하는 작업인가 / 이미 누가 하고 있나 (`Assignees`, 최근 코멘트) | AI가 "쉬운 이슈"라고 했지만 실제로는 RFC 진행 중 |
| 2. 재현 (Reproduce) | 최소 재현 코드 생성 / 환경 셋업 명령어 | 본인 PC에서 **실제로** 재현되는가 / 환경 의존이 있는가 | AI가 만든 재현 코드가 동작 안 함 → 이슈 자체를 잘못 이해 |
| 3. 수정 범위 (Scope) | 영향 받는 파일 목록 / 함수 추적 | "한 줄 수정"이 진짜 한 줄인가 / 테스트가 있는가 | AI가 "간단한 수정"이라고 한 게 실제로는 11파일 변경 |
| 4. PR 본문 (PR Body) | 변경 요약 / 관련 이슈 링크 / 체크리스트 | 변경 요약의 모든 줄이 사실인가 / 빠진 변경이 없는가 | AI가 "테스트 추가"라고 적었는데 실제 테스트 0개 |
| 5. 리뷰 대응 (Review) | 리뷰 코멘트 의도 해석 / 답변 초안 / 패치 안 | 리뷰어의 진짜 우려를 본인이 이해했나 / 모든 코멘트에 응답했나 | AI가 만든 답변이 영혼 없는 형식적 답변 → 머지 거절 |

### 강사 멘트 (단계별 1분씩, 총 5분)

> **단계 1 — 이슈**:
>
> "이슈를 고를 때 AI에게 'good-first-issue 라벨에서 좋은 거 추천해줘'라고 시킨다. AI는 답을 준다. 그런데 — 너희가 그 이슈에 가서 **Assignees**가 비어있는지, 최근 한 달 내 코멘트가 있는지, 메인테이너가 'Closing — out of scope'라고 한 적 없는지 사람이 봐야 한다. 이걸 안 보면 — 다른 사람이 이미 손대는 이슈에 PR 올려서 '중복'으로 닫힌다."

> **단계 2 — 재현**:
>
> "AI는 재현 코드를 잘 만든다. 단, 환경 의존이 있는 버그는 AI가 모른다. Python 3.11에서만 나는 버그를 AI가 'Python 3.10 코드'로 답하면 — 너는 재현 안 된다. 이때 사람이 실제로 본인 PC에서 한 번 돌려본 결과를 PR 본문에 적어야 한다."

> **단계 3 — 수정 범위**:
>
> "이게 가장 위험하다. AI가 '한 줄만 고치면 돼'라고 했는데, 너가 따라가 보면 11파일이 변경되더라. 왜? 그 한 줄이 호출되는 곳이 다 영향 받기 때문. AI에게 `git grep`이나 `rg` 결과를 같이 주면 정확도가 올라간다."

> **단계 4 — PR 본문**:
>
> "AI가 만든 PR 본문은 깔끔하다. 너무 깔끔해서 — 변경에 없는 내용까지 넣을 때가 있다. 예: 'tests added' 라고 적었는데 실제 PR diff에 테스트 0개. 메인테이너가 30초 안에 알아챈다. **PR 본문의 모든 줄을 diff와 대조**하는 게 사람의 일."

> **단계 5 — 리뷰 대응**:
>
> "리뷰어가 'why this approach?' 라고 물으면 AI는 'thanks for the review!' 같은 영혼 없는 답을 만든다. 메인테이너는 그런 답을 1초 만에 알아본다. **리뷰의 진짜 의도를 본인이 이해했는지** 확인하고, AI는 표현을 다듬는 데만 쓴다."

### 학생 활동 (3분, 페어)

### 강사 멘트 (각 약어 3분, 총 9분)

#### 9.1 RAG (3분)

> "RAG는 2020년 Lewis 외 9인 논문에서 시작했다. 한 문장으로 — '모델이 답하기 전에 외부에서 자료를 검색해 그 자료를 함께 보고 답한다.'
>
> 왜 만들었나? — LLM은 학습 시점 이후의 정보를 모른다. 너희가 GPT-4에게 '오늘 코스피 종가'를 물으면 모른다. 그런데 RAG로 — 검색 엔진이 가져온 오늘 뉴스를 LLM에게 같이 주면 — LLM이 그걸 보고 답한다.
>
> 오픈소스 기여에서는? — 너희가 낯선 저장소를 분석할 때, AI에게 '이 저장소의 README'를 같이 주는 게 사실은 RAG의 가장 단순한 형태다. **사람이 직접 검색 결과를 가져다 붙여넣는** RAG. 5회차에는 이 검색을 자동화하는 도구를 본다.
>
> 1차 출처는 자료 풀 [9] — Lewis et al., arXiv:2005.11401 (2020). 너희 노트에 한 줄: **'RAG = 외부 검색 + LLM 답변'**."

#### 9.2 MCP — Model Context Protocol (3분)

> "MCP는 2024년 11월 Anthropic이 발표하고 2025년 11월에 최신 스펙이 나온 오픈 프로토콜이다. 한 문장으로 — '**LLM 앱이 외부 데이터와 도구에 표준 방식으로 연결**되도록 하는 통신 규약.'
>
> 왜 만들었나? — 지금까지는 모든 AI 도구가 GitHub, Slack, Postgres에 연결할 때 각자 다른 방식으로 짰다. MCP는 'USB-C 같은 표준'을 만들겠다는 거다. MCP 서버 1개를 만들면 — Claude Code, Cursor, Continue 어디서든 같은 도구를 쓸 수 있다.
>
> 오픈소스 기여에서는? — 너희가 'GitHub 이슈 가져와서 분석해'라고 AI에게 시킬 때, 그 AI가 GitHub MCP 서버를 통해 이슈를 가져온다. 자료 풀 [10] — modelcontextprotocol.io. 한 줄: **'MCP = LLM이 외부 도구에 표준 방식으로 연결'**.
>
> 5회차 실습에서 너희는 modelcontextprotocol/servers 레퍼런스 서버 (filesystem, github, postgres 등)를 직접 띄운다."

#### 9.3 A2A — Agent2Agent (3분)

> "A2A는 2025년 4월 Google이 발표하고, 2025년 6월 Linux Foundation으로 이관된 오픈 표준이다. 한 문장으로 — '**서로 다른 프레임워크/벤더의 AI 에이전트가 통신하고 협업하는 표준**.'
>
> MCP와 차이? — **MCP는 도구 연결 (LLM ↔ 도구), A2A는 에이전트 간 협업 (에이전트 ↔ 에이전트)**. 이 한 줄이 시험에 나온다고 생각해라.
>
> 오픈소스 기여에서는? — 너희가 5회차에 만들 멀티 에이전트 — Planner, Coder, Reviewer, Tester, Documenter — 가 서로 다른 LLM 위에서 돌 때, 그들끼리 일을 주고받는 표준이 A2A다. 너희가 직접 짜진 않는다. 단, **표준이 있다는 사실**을 안다.
>
> 자료 풀 [11] — a2a-protocol.org. 한 줄: **'A2A = 에이전트 간 협업 표준'**."

조건: 추측엔 "추정" 표시.
```

#### Claude Code (CLI) — 저장소 전체 컨텍스트

```bash
claude
> 이 저장소를 처음 본다고 가정. 핵심 모듈 3개와 good-first-issue 라벨에서 첫 PR로 좋은 이슈 1개 추천. 추측엔 "추정" 표기.
```

#### Cursor — 멀티파일 컨텍스트

```text
@codebase 이슈 #<NN> 의 acceptance criteria 를 만족하려면
어떤 파일을 수정해야 하나? 각 파일의 변경 라인 추정.
```

### 15.9.2 단계 2 — 재현

#### Claude

```text
이슈 #<NN>을 재현하는 최소 코드를 작성해줘.
환경: <Python 3.11 / Node 20 / Rust stable / 본인 환경>
실행 방법도 함께. 본인이 직접 실행해 검증할 거다.

조건: 재현이 환경 의존이면 어떤 환경 변수가 영향 주는지 명시.
```

#### Aider (CLI, git-aware) — 자동 재현·패치 흐름

```bash
aider --message "이슈 #<NN> 재현 코드 + 가능하면 fix 시도. 단, 사람이 검증할 거니 일단 draft 패치만."
```

### 15.9.3 단계 3 — 수정 범위 산정

#### 어떤 도구든 — 직접 grep/rg 결과를 같이 주기

```bash
# 사람이 먼저
rg "변경할_함수명" --type rust > /tmp/scope.txt
git grep "변경할_함수명" >> /tmp/scope.txt
```

```text
# 그 다음 AI에게
다음 grep 결과를 보고 변경할 범위를 산정해줘.

grep 결과:
"""
<scope.txt 내용>
"""

이 중 실제로 수정해야 할 곳과 그대로 둘 곳을 분리해줘.
조건: 모르겠으면 "확실치 않음".
```

### 15.9.4 단계 4 — PR 본문 작성

#### Claude / Copilot — diff 기반

```text
다음 git diff 를 보고 PR 본문을 작성해줘.

diff:
"""
<git diff main..HEAD 출력>
"""

형식:
## What
## Why  (이슈 #NN 링크)
## How (핵심 파일/함수)
## Verification (테스트 명령어 + 결과)
## AI 사용 로그 (표)

조건: diff에 없는 변경은 본문에도 적지 말 것.
PR 본문은 30초 안에 읽힐 수 있게 짧게.
```

#### Claude Code — 자동 PR 생성

```bash
# Claude Code 안에서
> 방금 만든 변경으로 PR 본문을 작성하고 gh pr create --draft 까지 실행해줘.
> 단, AI 사용 로그 표는 비워두고 사람이 채울 칸으로 둬.
```

### 15.9.5 단계 5 — 리뷰 대응

```text
# Claude / Copilot 프롬프트

리뷰어 코멘트 (원문):
"""
<리뷰어가 단 코멘트 그대로>
"""

내가 짠 코드 (관련 파일):
"""
<코드 붙여넣기>
"""

질문:
1. 리뷰어가 진짜로 우려하는 게 뭔가?
2. 그에 답할 한 줄 답변 초안 (짧게)
3. 코드를 바꿔야 한다면 어떤 패치?

조건: "thanks for the review!" 같은 영혼 없는 답 만들지 말 것.
리뷰어 의도를 다시 묻는 게 더 나으면 그렇게 답할 것.
```

| **실제 기여/운영** | **25%** | **핵심** | **EX-4B 본격 PR 1개 (drafted)** — A: 외부 / B: 자체 |
| **AI 활용 능력** | **20%** | **핵심** | **AI 사용 로그 표의 "사람이 검증한 것" 칸 두께** |
| 발표/문서화 | 15% | 보조 | EX-4C README/changelog 보강 PR + PR 본문 품질 |

### 19.2 학생이 본 회차 활동을 안 하면 잃는 점수

> **핸드아웃에 명시 (마스터 플랜 §3.1 주석)**:

| 빠뜨리면 | 잃는 점수 (대략) |
|------|-----------|
| EX-4B PR draft 1개 (drafted) | 실제 기여/운영 25% 중 5~10점 |
| AI 사용 로그 표 (5단계) | AI 활용 능력 20% 중 5~10점 |
| AI 사용 로그의 "사람이 검증한 것" 칸 모두 비움 | AI 활용 능력 20% 만점 중 0~5점 (위험 5.4 강제 룰) |
| RAG/MCP/A2A 한 줄 정의 노트 | OS 이해도 20% 중 1~2점 (보조) |
| EX-4C README/changelog 보강 | 발표/문서화 15% 중 1~3점 (보조) |

### 19.3 채점 룰 핵심 (재강조)

> **AI 활용 능력 20%의 채점은 "썼는가"가 아니라 "쓰고 검증했는가"다.**
>
> AI 출력 그대로 PR에 붙여넣고 사람의 검증 흔적이 0인 경우 — 본 회차 AI 활용 영역 0점.
